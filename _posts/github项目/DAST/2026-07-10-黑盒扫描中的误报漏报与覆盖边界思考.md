## 黑盒扫描的误报、漏报与覆盖边界

做黑盒扫描平台落地的时候，最常被问到的其实不是"能不能扫"，而是三个更现实的问题：为什么漏洞在但没扫出来、为什么扫出来了但不是真漏洞、到底哪些场景能扫哪些扫不到。

这三个问题拆开看：漏报、误报、覆盖边界。搞清楚问题出在哪一层，才知道该改 POC、改链路、改资产发现，还是承认这个场景黑盒本来就不适合。

***

### 漏报

漏洞在，但平台没产出结果。原因大致有五种。

**响应判定过窄。** 很多 POC 默认要求 status 200 + 固定关键字 + 某个响应头，但真实站点里成功利用经常返回 302/403/500，或者统一异常页改写了返回体，或者副作用已经发生但页面只返回空白。结果就是——利用成功了，matcher 没命中。

**POC 覆盖不全。** 同一个漏洞可能对应多条利用路径：不同版本路径不一样、不同部署方式入口不一样、不同二开版本返回特征不一样。模板只写了其中一个变体，其余的就漏了。这不是目标没漏洞，是模板没覆盖到那个分支。

**资产发现不完整。** 黑盒只能扫到已经进入扫描面的资产。域名没解析出来、端口没扫出来、后台挂在非常规端口、漏洞入口在隐藏路径下——这些情况下，漏洞入口根本没进入扫描链路，后面再强也没用。

**分布式链路丢数据。** 在多 Worker 队列平台上，"没扫出来"和"扫到了但丢了"是两回事。阶段执行了但没入队下一阶段、Worker 子进程提前退出、JSON 解析失败、写库失败、上报报错、去重逻辑误删新结果——这些都是平台链路漏报，不是检测逻辑漏报。

**场景超出 YAML 表达能力。** 需要登录态的后台漏洞、多步交互漏洞、带 token/session/nonce 的接口、复杂 TCP 二进制协议——这些不是没有 POC，是很难用通用 YAML 模板稳定表达，最终容易表现为漏报。

***

### 误报

扫出来了，但不是真漏洞。原因大致有六种。

**路径可访问 ≠ 漏洞存在。** 模板只判断"路径返回 200 + body 包含某关键字"，但站点可能对不存在路径也返回 200（404 被 API 网关包装成了业务页面），或者返回的是 fallback 页面。把"路径存在"当成"漏洞存在"是最常见的误报来源。

**前端框架页面。** Vue/React 这类 SPA 是误报高发区。任意路径都返回 index.html，前端壳结构高度统一，app 根节点和 JS bundle 到处都有。模板如果只靠状态码和静态关键字，很容易把前端壳页面误判成漏洞入口。

**DNSLog / OOB 误报。** DNSLog 的思路是：扫描器生成唯一子域名放进 payload，目标执行了漏洞逻辑就会去解析这个域名，DNSLog 平台记录到这次解析后判定漏洞存在。它验证的不是页面返回值，而是目标是否对外发起了 DNS/HTTP 请求。

但看到 DNS 查询 ≠ 漏洞真的被利用成功。目前能想到的误报来源：

1. 自己或自己的工具意外触发了解析（比如扫描器预解析了 payload 里的域名，或者调试脚本、代理工具帮着访问了）
2. WAF、安全设备、沙箱做预检——它们的目的是"先帮你访问看看"，结果 DNSLog 看到了请求，但发起者不是目标应用本身
3. 互联网爬虫或外部安全探针碰到了 payload
4. 目标内部的日志服务、威胁检测、审计平台自动解析了域名

还有没有其他的？目前想到的可能还有：DNS 转发链路中的递归服务器和缓存行为带来的噪音、企业 DNS 出口代理产生的额外查询——有时看到的是中间 DNS 行为而不是最终应用行为。

所以 DNSLog 误报分析本质上是一次触发来源归因问题。想区分是否误报，要追问几个关键问题：

- DNSLog 平台是怎么搭建的
- 用了几个域名
- 扫描器和 DNSLog 是怎么联动的（token 怎么申请、怎么轮询结果）
- 触发时经过哪些 DNS 服务器
- 是否同时有 HTTP 回连
- 能否看到来源 IP、查询时间和查询类型

这里常说的"框架"，指的是 DNSLog/OOB 平台的整体实现架构——怎么搭建、怎么和扫描器联动、触发链路是什么——不是 Vue/Spring 这种业务开发框架。

**指纹误判的连锁效应。** 先把产品识别错了（标题/favicon/字符串碰巧相似），再套错了漏洞模板，最后把普通响应误判成漏洞。先指纹错，再漏洞错。

**中间层干扰。** WAF 统一改写返回、CDN 缓存旧内容、API 网关返回通用结构、反向代理包装错误页——模板匹配到的可能是中间设施行为，不是业务系统漏洞。

**模板写法过宽。** matcher 只写了 status==200 + body 包含 success，没做上下文限定、多条件交叉、负向判断、二次确认。这类模板天然高误报。

***

### 覆盖边界

哪些场景黑盒能稳定扫，哪些扫不了。

**能覆盖的：**

- 已知 HTTP 漏洞：路径清晰、请求固定、成功特征明确、公开 POC 成熟（任意文件读取、未授权访问、已知接口 RCE、信息泄露）
- 中间件和组件暴露类：产品标准化程度高、暴露面稳定、自动化收益高（Redis 未授权、WebLogic 历史 RCE、Jenkins/Kibana/ES 暴露、Actuator 暴露、管理台弱口令）
- 部分 OOB 漏洞：XXE 外带、SSRF 外带、JNDI/Log4Shell，前提是 DNSLog 链路设计足够清晰能区分真触发和噪音

**不容易覆盖的：**

- 后台漏洞：需要登录、需要权限、依赖业务上下文、依赖 token/session/nonce，后台入口本身不在公开资产面上。没有认证态和业务态时很难稳定自动化覆盖。
- 复杂 TCP/二进制协议漏洞：协议握手复杂、payload 和响应是二进制、有长度/状态/编码字段、多轮交互。基于 TCP 的漏洞 YAML 不是很好写，这方面的 POC 不一定能完全覆盖。更适合用专用脚本、PoC 程序或插件化检测器。
- 多步业务逻辑漏洞：先注册再登录再上传再触发、先拿 token 再调接口、多次请求传递状态、依赖真实用户行为的 CSRF/XSS/越权。状态机和上下文依赖强，自动化成本高稳定性差。

***

### POC 建设优先级

如果要把 nuclei 现有模板之外的能力补起来，思路不是"尽可能多加"，而是先按公司场景做分层。

先明确一个筛选三角：**公开有 POC + 有危害 + 有资产**。三条都满足的优先补。

然后按这个顺序来：

1. **公司真实资产中高频且高危的漏洞**：过往事件中出现频率高、公司资产中真实存在。投入产出比最高。
2. **改造误报高/漏报高的现有模板**：matcher 太宽、只覆盖单一变体、不匹配公司部署方式、缺二次确认。保留漏洞编号和核心思路，重写成贴合公司资产形态的版本。
3. **公司内部高频系统的专项 POC 库**：常见 OA、网关/VPN/堡垒机、中间件、业务框架、云组件暴露点。专项 POC 价值通常高于泛化模板库。
4. **YAML 不适合的转成插件化检测**：复杂 TCP 协议、登录态后台、多步交互、强 OOB 归因——不建议硬塞通用模板，该补 Go/Python 插件、专项脚本、协议级探测器或浏览器自动化验证器。

换一个角度想：如果把 nuclei 的 POC 全去掉，应该加哪些 POC 匹配公司场景？这个问题的答案，才是 POC 建设的真正方向。

***

### 结论笔记

**漏报排查方向：**
- 先查响应判定——matcher 是不是太死板了，只认 200
- 再查 POC 变体——同一个漏洞是不是有多个利用路径没覆盖
- 再查资产发现——漏洞入口是不是根本没进入扫描面
- 再查平台链路——是不是扫到了但结果在某个环节丢了
- 最后看场景——是不是 YAML 本身就很难表达这种漏洞

**误报排查方向：**
- 先查模板写法——matcher 是不是太宽，缺负向条件和二次确认
- 再查目标环境——是不是 SPA/前端壳页面返回了假 200
- 再查 OOB 归因——DNSLog 记录到底是谁触发的，能不能看到来源 IP
- 再查中间层——是不是 WAF/CDN/网关改写了真实响应
- 再查指纹——是不是先识别错了产品再套错了模板

**覆盖判断：**
- 黑盒最稳的场景：已知 HTTP 漏洞 + 中间件暴露 + 部分 OOB
- 黑盒最不稳的场景：后台漏洞 + 复杂 TCP + 多步业务逻辑
- 覆盖率提升的关键不在模板数量，而在：资产发现能力 + POC 变体覆盖 + 平台链路稳定性

**POC 建设核心思路：**
- 筛选三角：公开有 POC + 有危害 + 有资产，三条都满足的优先
- 优先级排序：公司高频高危 > 改造高误报模板 > 专项 POC 库 > 插件化检测
- 对 YAML 不适合的场景不要硬塞，该转 Go/Python 插件或专项脚本
- 最终方向：把 nuclei 的 POC 全去掉，只加匹配公司场景的那些
