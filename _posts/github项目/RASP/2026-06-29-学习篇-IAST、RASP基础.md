## IAST与RASP基础

为了学习灰盒测试，今天来了解下IAST与RASP的原理部分。它们共享相似的底层实现机制——都依赖Java Agent技术进行字节码插桩，但IAST旨在发现漏洞，RASP则专注于阻断攻击。

### 一、IAST与RASP概念

**IAST（交互式应用安全测试）** 是一种漏洞检测工具，融合了静态分析（SAST）和动态分析（DAST）的优势。它通过Agent监控应用运行时的所有请求、代码和数据流，据此扫描相关代码中的漏洞。这意味着IAST的代码覆盖率取决于应用的被使用程度，但能精准定位漏洞在代码中的位置。

**RASP（运行时应用自我保护）** 则是一种防护工具，部署在生产环境。当应用运行时，RASP Agent能够监控输入输出和数据流，检测并**主动阻断**恶意攻击企图。RASP并不能替代漏洞检测工具，但在OpenMRS中能够有效防止注入类漏洞的利用。它与WAF（Web应用防火墙）的关键区别在于：RASP拥有应用源码上下文，能理解行为是否恶意，而非仅做模式匹配。

| 维度 | IAST（交互式应用安全测试） | RASP（运行时应用自我保护） |
| :--- | :--- | :--- |
| **核心目标** | **漏洞检测**（发现风险） | **攻击阻断**（实时防护） |
| **部署场景** | 测试、QA环境 | 生产环境 |
| **核心能力** | 结合静态与动态分析，追踪污点数据流 | 监控行为，拦截恶意调用 |
| **目标** | 效率2.14 VpH，发现8个OWASP Top 10风险 | 主要有效防止注入类攻击 |

### 二、技术底座：Java Agent、Instrumentation API

无论是IAST还是RASP，在Java生态中的实现都离不开Java Agent和`java.lang.instrument`包。这项技术从Java SE 5引入，其底层核心是**JVMTI（JVM Tool Interface）**——JVM提供的一组原生接口，允许外部工具与JVM交互，监控和修改JVM运行状态。Instrumentation API正是对JVMTI的封装，为开发者提供更友好的Java接口。

基于此，Java Agent有两种加载方式，对应不同场景：

1.  **静态加载（premain）**：通过JVM启动参数`-javaagent`在应用启动前加载。Agent的`premain`方法会被调用，可在类首次被加载时修改字节码。适用于需要在应用启动阶段就注入逻辑的场景，如性能监控工具。
2.  **动态加载（agentmain）**：Java SE 6引入，通过`com.sun.tools.attach.VirtualMachine` API在JVM运行后动态加载Agent Jar包。Agent的`agentmain`方法被调用，支持对已加载的类进行重定义（`redefineClasses`）或重转换（`retransformClasses`）。这为热部署、线上诊断（如Arthas）提供了可能。

### 三、字节码插桩的工作流程

IAST和RASP实现监控的核心，是在目标方法中插入检测逻辑。这一过程通过实现`ClassFileTransformer`接口完成。其标准工作流程如下：

1.  **Agent加载**：JVM启动时加载Agent Jar包，或运行时通过Attach API动态加载。
2.  **初始化**：JVM将`Instrumentation`实例传递给Agent的`premain`或`agentmain`方法。
3.  **注册转换器**：Agent调用`Instrumentation.addTransformer()`方法注册自定义的`ClassFileTransformer`。
4.  **类加载/重定义拦截**：当JVM加载一个新类，或调用`retransformClasses`重转换已加载类时，JVM会回调所有注册的`ClassFileTransformer`的`transform`方法。
5.  **字节码修改**：在`transform`方法中，开发者可以分析并修改传入的字节码（`classfileBuffer`），返回修改后的字节数组。
6.  **类定义**：JVM使用修改后的字节码定义类。修改后的类将包含Agent注入的检测逻辑，从而实现IAST/RASP的监控功能。

### 四、实践：监控命令执行的Agent Demo

基于上述原理，实现一个监控`Runtime.exec`命令执行的Agent Demo，核心在于利用`ClassFileTransformer`在`java.lang.Runtime`类加载时，修改其`exec`方法的字节码。

**关键步骤**：

1.  **创建Agent入口**：定义`premain`方法，注册`ClassFileTransformer`。
2.  **实现Transformer**：在`transform`方法中，筛选目标类（`java/lang/Runtime`）和方法（`exec`）。
3.  **字节码增强**：使用ASM、ByteBuddy等库修改字节码，在方法入口和出口注入日志打印逻辑。

**代码示例（使用ByteBuddy简化开发）**：

```java
// Agent入口：匹配Runtime类的exec方法，委托给拦截器
public class CommandMonitorAgent {
    public static void premain(String agentArgs, Instrumentation inst) {
        new AgentBuilder.Default()
            .type(ElementMatchers.named("java.lang.Runtime"))
            .transform((builder, type, cl, module) ->
                builder.method(ElementMatchers.named("exec"))
                       .intercept(MethodDelegation.to(CommandInterceptor.class))
            ).installOn(inst);
    }
}

// 拦截器：在原始方法执行前后打印日志
public class CommandInterceptor {
    @RuntimeType
    public static Object intercept(@AllArguments Object[] args, 
                                   @SuperCall Callable<?> zuper) throws Exception {
        System.out.println("[Agent] 检测到命令执行: " + Arrays.toString(args));
        Object result = zuper.call(); // 调用原始exec方法
        System.out.println("[Agent] 命令执行完成");
        return result;
    }
}
```

当目标应用执行`Runtime.getRuntime().exec("ls -al")`时，控制台将输出检测日志，证明插桩成功。
