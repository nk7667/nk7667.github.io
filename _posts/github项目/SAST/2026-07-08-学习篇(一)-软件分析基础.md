## 第一章：A1 — 活跃变量分析，理解数据流的起点

### 这是 A1 的内容

A1 是整套实验的入门。你需要实现两个东西：

1. **`LiveVariableAnalysis`** — 活跃变量分析
2. **`IterativeSolver`** — 基于迭代的数据流求解器

### 核心概念：什么是"活跃变量"？

一个变量 $v$ 在程序点 $p$ 是**活跃的**，当且仅当存在一条从 $p$ 到程序出口的路径，在这条路径上 $v$ 在下次被定义（写）之前会被使用（读）。

换个说法：**活跃变量就是"将来还要用的变量"**。

### 为什么这是后向分析？

活跃变量分析是**后向（backward）**的，因为要判断一个变量将来是否会被使用，必须从程序出口往入口方向推导。数据流方程是：

$$IN[B] = USE[B] \cup (OUT[B] - DEF[B])$$

- **USE[B]**：基本块 B 中在定义前就被使用的变量集合
- **DEF[B]**：基本块 B 中定义的变量集合
- **OUT[B]**：从 B 的后继节点 meet 过来的结果

#### 方向声明

#### 后向初始化

```java
// Solver.java 第 87-98 行 —— A1 你的代码
protected void initializeBackward(CFG<Node> cfg, DataflowResult<Node, Fact> result) {
    // 出口节点的 OUT 设成 boundary fact（空集）
    result.setOutFact(cfg.getExit(), analysis.newBoundaryFact(cfg));
    // 其他所有节点的 IN 和 OUT 都初始化为空集
    for (Node node : cfg) {
        if (node != cfg.getExit()) {
            result.setOutFact(node, analysis.newInitialFact());
        }
        result.setInFact(node, analysis.newInitialFact());
    }

```

关键点：后向分析的初始化是**从出口节点开始的**。出口的 OUT 是空集（程序结束时没有变量还需要用了），然后求解器**从出口往入口方向推**。

#### transferNode — 后向公式的实现

```java
// LiveVariableAnalysis.java 第 63-83 行 —— A1 你的代码
@Override
public boolean transferNode(Stmt stmt, SetFact<Var> in, SetFact<Var> out) {
    // 公式：IN = USE ∪ (OUT - DEF)
    
    // 第一步：从 OUT 出发
    SetFact<Var> temp = out.copy();
    
    // 第二步：去掉当前语句定义的变量（被"杀死"了）
    stmt.getDef().ifPresent(def -> {
        if (def instanceof Var) {
            temp.remove((Var) def);
        }
    });
    
    // 第三步：加上当前语句使用的变量（"激活"它们）
    for (var use : stmt.getUses()) {
        if (use instanceof Var) {
            temp.add((Var) use);
        }
    }
    
    // 第四步：写入 IN
    boolean changed = !in.equals(temp);
    if (changed) {
        in.set(temp);
    }
    return changed;
}
```

#### IterativeSolver — 后向迭代求解

```java
// IterativeSolver.java 第 40-62 行 —— A1 你的代码
protected void doSolveBackward(CFG<Node> cfg, DataflowResult<Node, Fact> result) {
    boolean changed;
    do {
        changed = false;
        for (Node node : cfg) {
            if (node == cfg.getExit()) {
                continue;   // 出口节点跳过，它的 OUT 是固定的
            }
            // OUT[B] = 所有后继 IN 的并集
            Fact out = result.getOutFact(node);
            for (Node succ : cfg.getSuccsOf(node)) {
                analysis.meetInto(result.getInFact(succ), out);
            }
            // 调用 transferNode 计算 IN[B]
            Fact in = result.getInFact(node);
            if (analysis.transferNode(node, in, out)) {
                changed = true;
            }
        }
    } while (changed);
}
```

后向分析遍历节点的顺序无所谓（它用迭代到不动点），但**数据流的方向是反的**：

```
后向：先算出后继的 OUT 和 IN → 然后 merge 后继的 IN 得到当前节点的 OUT → 再算当前节点的 IN
前向：先算出前驱的 OUT 和 IN → 然后 merge 前驱的 OUT 得到当前节点的 IN → 再算当前节点的 OUT
```

### 完整推算过程 — 以 Assign.java 为例

源程序：

```java
// Assign.java —— A1 测试用例
int assign(int a, int b, int c) {
    int d = a + b;    // [0]
    b = d;            // [1]
    c = a;            // [2]
    return b;         // [3]
}
```

CFG 结构：入口 → [0] → [1] → [2] → [3] → 出口

A1 期望的输出：

```
[0] d = a + b;    [a, d]     ← OUT 集中有 a 和 d，表示这条语句之后 a 和 d 还需要用
[1] b = d;        [a, b]     ← OUT 集中有 a 和 b
[2] c = a;        [b]        ← OUT 集中有 b（c 不需要，a 也不需要了）
[3] return b;      []         ← OUT 空集（程序结束）
```

**方括号里的内容是每句话执行之后（OUT），哪些变量将来还会被用到。**

#### 迭代之前：初始化

```
节点      IN        OUT
[0]       {}        {}
[1]       {}        {}
[2]       {}        {}
[3]       {}        {}
出口      -         {}    ← 只有出口的 OUT 是空集（程序结束，啥都不需要了）
```

#### 第一轮迭代

**处理 [3] `return b`**：

```
步骤1: OUT[B] = 后继 IN 的并集
   [3] 的后继是出口，出口没有 IN（出口只有 OUT）
   所以 OUT[3] = {}（空集）

步骤2: transferNode([3], IN[3], OUT[3])
   公式 IN = USE ∪ (OUT - DEF)
   
   temp = OUT.copy() = {}
   getDef(): "return b" 没有 def → 不删任何东西
   getUses(): "return b" 用了 b → temp.add(b)
   
   temp = {b}
   IN[3] 原来是 {}，变成 {b} → changed = true
```

**处理 [2] `c = a`**：

```
步骤1: OUT[2] = merge(IN[3])
   IN[3] = {b} → OUT[2] = {b}

步骤2: transferNode([2], IN[2], OUT[2])
   temp = {b}
   getDef(): "c = a" 定义了 c → temp.remove(c) → c 不在里面，不变
   getUses(): "c = a" 使用了 a → temp.add(a)
   
   temp = {a, b}
   IN[2] = {a, b}  ← changed = true
```

**处理 [1] `b = d`**：

```
步骤1: OUT[1] = merge(IN[2])
   IN[2] = {a, b} → OUT[1] = {a, b}

步骤2: transferNode([1], IN[1], OUT[1])
   temp = {a, b}
   getDef(): "b = d" 定义了 b → temp.remove(b) → temp = {a}
   getUses(): "b = d" 使用了 d → temp.add(d)
   
   temp = {a, d}
   IN[1] = {a, d}  ← changed = true
```

**处理 [0] `d = a + b`**：

```
步骤1: OUT[0] = merge(IN[1])
   IN[1] = {a, d} → OUT[0] = {a, d}

步骤2: transferNode([0], IN[0], OUT[0])
   temp = {a, d}
   getDef(): "d = a + b" 定义了 d → temp.remove(d) → temp = {a}
   getUses(): "d = a + b" 使用了 a, b → temp.add(a), temp.add(b)
   
   temp = {a, b}
   IN[0] = {a, b}  ← changed = true
```

**第一轮结束，有变化 → 继续第二轮。**

#### 第二轮迭代

```
[3]: OUT[3] = merge(IN[出口]) = {}，不变
[2]: OUT[2] = merge(IN[3]) = {b}，不变

[1]: OUT[1] = merge(IN[2])
     IN[2] 现在是 {a, b} → OUT[1] = {a, b}
     transferNode: temp = {a, b}, 删 b, 加 d → {a, d}
     IN[1] 已经是 {a, d}，没变 → changed=false

[0]: OUT[0] = merge(IN[1]) = {a, d}
     transferNode: {a, d} 删 d, 加 a, b → {a, b}
     IN[0] 已经是 {a, b}，没变 → changed=false
```

**全部没变 → 结束！**

#### 最终结果

```
语句            OUT（将来还会被用的变量）
[0] d = a+b     {a, d}    ← 之后 a 要用来算 c, d 要用来算 b
[1] b = d       {a, b}    ← 之后 a 要用来算 c, b 要用来返回
[2] c = a       {b}       ← 之后 b 要用来返回   ← 注意！c 不在里面！
[3] return b     {}        ← 程序结束
```

**`[2] c = a` 这条语句执行完后，c 不在活跃集合里！** 因为 c 定义之后再也没有被使用过。这就是 "c 没人用" 的结论来源。

### SAST 视角：这解决了什么问题？

在漏洞扫描中，活跃变量分析是**死代码/死赋值检测**的输入。如果一个变量在定义后立即被"杀死"（不再活跃），那么这个赋值就是无意义的——它可能意味着：

- 开发者写了多余的代码
- 某个返回值被忽略了，可能是资源泄漏的信号
- 安全相关的检查结果被丢弃了

## 第二章：A2 — 常量传播 + Worklist 求解器

### 这是 A2 的内容

虽然 A2 在实验仓库中没有独立目录，但它的核心实现分布在 A3 和 A4 的 `ConstantPropagation.java` 中。A2 需要你实现：

1. **`ConstantPropagation`** — 常量传播
2. **`WorkListSolver`** — 基于 Worklist 的求解器

### 方向对比：前向 vs 后向

A1 是**后向**分析，从出口往入口推。A2 是**前向**分析，从入口往出口推。核心区别一行代码：

```java
// LiveVariableAnalysis.java（A1）第 44 行
public boolean isForward() { return false; }   // 后向 ← A1

// ConstantPropagation.java（A2）第 53 行
public boolean isForward() { return true; }    // 前向 ← A2
```

前向分析意味着**从程序入口往出口方向推数据**，初始化也是从入口开始：

```java
// Solver.java 第 79-90 行 —— 前向初始化
protected void initializeForward(CFG<Node> cfg, DataflowResult<Node, Fact> result) {
    // 入口节点的 IN 设成 boundary fact
    result.setInFact(cfg.getEntry(), analysis.newBoundaryFact(cfg));
    // 其他节点初始化为空
    for (Node node : cfg) {
        if (!cfg.getEntry().equals(node)) {
            result.setInFact(node, analysis.newInitialFact());
        }
        result.setOutFact(node, analysis.newInitialFact());
    }
    // 入口节点先跑一次 transferNode
    analysis.transferNode(cfg.getEntry(),
            result.getInFact(cfg.getEntry()),
            result.getOutFact(cfg.getEntry()));
}
```

### 核心数据结构：三值格（Value）

常量传播使用一个**三值格（lattice）**来表示每个变量的可能值：

```
        NAC（肯定不是常量）
       /    \
  ... -2  -1  0  1  2 ...  （具体常量值）
       \    /
       UNDEF（还没赋值，可能是任何值）
```

- **UNDEF**：变量还没被赋值，值是未知的
- **具体常量**：确定的值，比如 `5`
- **NAC**：不可能是常量（比如从参数传入、从方法调用返回）

#### Meet 运算

```java
// ConstantPropagation.java 第 84-90 行 —— A2 你的代码
public Value meetValue(Value v1, Value v2) {
    if (v1.isUndef()) return v2;         // 一条路没值，取另一条
    if (v2.isUndef()) return v1;         // 同上
    if (v1.isNAC() || v2.isNAC()) return Value.getNAC();  // 任一条不确定 = 不确定
    if (v1.equals(v2)) return v1;        // 两条路值相同 = 确定
    return Value.getNAC();               // 两条路值不同 = 不确定
}
```

例子：

```
meetValue(UNDEF, 6)  = 6        → "一条路还没赋值，就以另一条为准"
meetValue(6, 6)      = 6        → "两条路都是 6，确定"
meetValue(6, 7)      = NAC      → "分歧了，不是常量"
meetValue(NAC, 6)    = NAC      → "不确定传染"
```

#### CPFact：变量→值的映射

```java
// CPFact.java —— A2/A3/A4 共用
public class CPFact extends MapFact<Var, Value> {
    @Override
    public Value get(Var key) {
        return map.getOrDefault(key, Value.getUndef());  // 没记录 = UNDEF
    }
    
    @Override
    public boolean update(Var key, Value value) {
        if (value.isUndef()) {
            return remove(key) != null;     // 设为 UNDEF = 删掉
        } else {
            return super.update(key, value);
        }
    }
}
```

关键设计：**用"不存在"来代表 UNDEF**。如果 fact 中没有某个变量，它的值就是 UNDEF。设为 UNDEF 等于删除。

### transferNode — 常量怎么"算"出来的

```java
// ConstantPropagation.java 第 92-104 行 —— A2 你的代码（带修复版）
@Override
public boolean transferNode(Stmt stmt, CPFact in, CPFact out) {
    boolean changed = out.copyFrom(in);      // 第一步：IN 复制到 OUT
    
    if (stmt instanceof DefinitionStmt<?, ?> defStmt) {
        var lhs = defStmt.getLValue();       // 等号左边 = 谁
        var rhs = defStmt.getRValue();       // 等号右边 = 什么表达式
        
        if (lhs instanceof Var var && canHoldInt(var)) {
            Value result = evaluate(rhs, in);     // 计算右边的值
            if (result == null) {
                result = Value.getNAC();          // 算出 null = 不是常量
            }
            changed |= out.update(var, result);    // 更新 OUT
        }
    }
    return changed;    // ← 关键修复：非定义语句也返回 copyFrom 的变化状态
}
```

每一步的逻辑：
1. `out.copyFrom(in)`：先假设"什么都不变"
2. 如果是定义语句：用 `evaluate` 计算右边表达式的值
3. 把结果写进 OUT

### evaluate — 表达式怎么求值

```java
// ConstantPropagation.java 第 112-163 行 —— A2 你的代码
public static Value evaluate(Exp exp, CPFact in) {
    // 情况1：exp 是变量 → 查 IN fact 中这个变量的值
    if (exp instanceof Var var) {
        if (canHoldInt(var)) {
            return in.get(var);    // 从 IN 中读
        } else {
            return Value.getNAC();
        }
    }
    // 情况2：exp 是字面量 → 直接返回常量
    else if (exp instanceof IntLiteral literal) {
        return Value.makeConstant(literal.getValue());  // 6 → Value(6)
    }
    // 情况3：exp 是二元运算 → 递归求两个操作数，再算
    else if (exp instanceof BinaryExp binary) {
        Value v1 = evaluate(binary.getOperand1(), in);
        Value v2 = evaluate(binary.getOperand2(), in);
        
        if (v1.isNAC() || v2.isNAC()) return Value.getNAC();   // 任一不确定
        if (v1.isUndef() || v2.isUndef()) return Value.getUndef(); // 任一未定义
        
        // 都是常量 → 直接运算
        int c1 = v1.getConstant();
        int c2 = v2.getConstant();
        
        if (binary instanceof ArithmeticExp arith) {
            switch (arith.getOperator()) {
                case ADD: return Value.makeConstant(c1 + c2);
                case SUB: return Value.makeConstant(c1 - c2);
                case MUL: return Value.makeConstant(c1 * c2);
                case DIV:
                    if (c2 == 0) return Value.getUndef();  // 除零 → 未定义
                    return Value.makeConstant(c1 / c2);
                case REM:
                    if (c2 == 0) return Value.getUndef();
                    return Value.makeConstant(c1 % c2);
            }
        }
        // ShiftExp, BitwiseExp, ConditionExp 等类似处理
    }
    return null;   // 其他表达式（如方法调用）返回 null → 会被设为 NAC
}
```

用 `a = 6; b = a + 3` 跑一遍：

```
语句 [0] a = 6:
  IN = {}
  out.copyFrom(IN) → OUT = {}
  evaluate(IntLiteral(6)) = Value.makeConstant(6)
  out.update(a, 6) → OUT = {a=6}
  changed = true

语句 [1] b = a + 3:
  IN = {a=6}   ← 从前驱 merge 来的
  out.copyFrom(IN) → OUT = {a=6}
  evaluate(a + 3, IN):
    evaluate(a) = IN.get(a) = Value(6)
    evaluate(IntLiteral(3)) = Value(3)
    ADD: 6 + 3 = Value(9)
  out.update(b, Value(9)) → OUT = {a=6, b=9}
  changed = true
```

### 一个隐蔽的 Bug（影响 A4）

原始的 `transferNode`（未经修复版）有一个问题：

```java
// 原始版（有 bug）
public boolean transferNode(Stmt stmt, CPFact in, CPFact out) {
    out.copyFrom(in);                          // 确实改了 OUT
    if (stmt instanceof DefinitionStmt) {
        // ...
        return out.update(var, result);        // 只有这里返回 true
    }
    return false;                              // ❌ 非定义语句返回 false
}
```

像 `nop`、`return y` 这样的语句，`out.copyFrom(in)` 确实改变了 OUT，但方法返回了 `false`。WorkListSolver 收到 `false` 就认为"没变化"，不把后继加入 worklist，传播就断了。

**在过程内分析（A2）中这个问题可能不致命**，但在过程间分析（A4）中这是致命的——比如 addOne 里的 `return y`，如果返回 false，exit 节点的后继（ReturnEdge 的目标）就不会被触发，数据链就断了。

修复：

```java
// 修复版
boolean changed = out.copyFrom(in);   // 记录这里的变化
if (stmt instanceof DefinitionStmt) {
    changed |= out.update(var, result);
}
return changed;    // 非定义语句也可能有变化
```

### A2 的关键突破：从迭代到 Worklist

A1 的迭代求解器每次都处理所有节点，效率低。A2 的 Worklist 求解器做了一个关键优化：**只把 OUT 发生变化的节点的后继加入 worklist**。

#### WorkListSolver — 前向求解

```java
// WorkListSolver.java（A2/A3 共用）第 38-60 行
protected void doSolveForward(CFG<Node> cfg, DataflowResult<Node, Fact> result) {
    // 初始化：所有非入口节点加入 worklist
    LinkedList<Node> worklist = new LinkedList<>();
    for (Node node : cfg) {
        if (node != cfg.getEntry()) {
            worklist.add(node);
        }
    }
    while (!worklist.isEmpty()) {
        Node node = worklist.poll();
        // merge 前驱的 OUT
        Fact in = result.getInFact(node);
        for (Node pred : cfg.getPredsOf(node)) {
            analysis.meetInto(result.getOutFact(pred), in);
        }
        // 调用 transferNode
        Fact out = result.getOutFact(node);
        if (analysis.transferNode(node, in, out)) {
            // ★ 只把后继加入 worklist，不全量迭代！
            for (Node succ : cfg.getSuccsOf(node)) {
                if (!worklist.contains(succ)) {
                    worklist.add(succ);
                }
            }
        }
    }
}
```

#### WorkListSolver — 后向求解

```java
// WorkListSolver.java 第 63-85 行
protected void doSolveBackward(CFG<Node> cfg, DataflowResult<Node, Fact> result) {
    LinkedList<Node> worklist = new LinkedList<>();
    for (Node node : cfg) {
        if (node != cfg.getExit()) {
            worklist.add(node);
        }
    }
    while (!worklist.isEmpty()) {
        Node node = worklist.poll();
        Fact out = result.getOutFact(node);
        for (Node succ : cfg.getSuccsOf(node)) {
            analysis.meetInto(result.getInFact(succ), out);
        }
        Fact in = result.getInFact(node);
        if (analysis.transferNode(node, in, out)) {
            for (Node pred : cfg.getPredsOf(node)) {
                if (!worklist.contains(pred)) {
                    worklist.add(pred);
                }
            }
        }
    }
}
```

#### 效率对比

```
A1 IterativeSolver（活变量分析）:   每轮处理全部节点，N 轮 → O(N²)
A2 WorkListSolver（常量传播）:      只处理 OUT 变化的节点的后继 → O(N) 平均
```

### A2 vs A4 对比：过程内 vs 过程间

用 Example.java：

```java
a = 6;                // [0]
b = addOne(a);        // [1]
c = b - 3;            // [4]
b = ten();            // [6]
c = a * b;            // [7]
```

**A2（过程内）结果**：

```
[0] a = 6                         → {a=6}
[1] temp$1 = invokestatic addOne(a) → {a=6}              ← temp$1 = NAC
[2] b = temp$1                      → {a=6}              ← b = NAC
[3] %intconst0 = 3                  → {a=6, %intconst0=3}
[4] c = b - %intconst0              → {a=6, %intconst0=3} ← c = NAC
[5] temp$3 = invokestatic ten()     → {a=6, %intconst0=3} ← 又不知道
[6] b = temp$3                      → {a=6, %intconst0=3} ← b 还是 NAC
[7] c = a * b                       → {a=6, %intconst0=3} ← c = NAC
```

**A2 能确定的只有 `a=6` 和 `%intconst0=3`**，遇到方法调用就全丢了。

原因在 `evaluate` 函数的最后一行：`return null` — InvokeExp 不是 Var/IntLiteral/BinaryExp，返回 null 后转为 NAC，丢失精度。

**A4（过程间）结果**：

```
[2] b = temp$1                      → {a=6, b=7, temp$1=7}     ← b=7！
[4] c = b - %intconst0              → {a=6, b=7, c=4, ...}     ← c=4！
[7] c = a * b                       → {a=6, b=10, c=60, ...}   ← c=60！
```

A4 通过 CallEdge 把 `a=6` 传给 addOne 的 x，addOne 内部算出 `y=7`，再通过 ReturnEdge 把 `y=7` 变成 `temp$1=7` 传回来。**数据"穿越"了方法边界**。

### A1 vs A2 总结

| | A1 活变量分析 | A2 常量传播 |
|--|-------------|-----------|
| 方向 | 后向（出口→入口） | **前向**（入口→出口） |
| 公式 | IN = USE ∪ (OUT − DEF) | **OUT = IN（更新定义变量的值）** |
| Fact 类型 | SetFact（变量集合） | **CPFact（变量→Value 映射）** |
| 求解器 | Iterative（全量迭代） | **WorkList（增量传播）** |
| 输出 | 每个语句后的活跃变量集 | **每个语句前的常量值映射** |
| 方法调用处理 | 不涉及（只算变量使用） | 保守返回 NAC（不知道被调方法做了什么） |

## 第三章：A3 — 死代码检测，把 A1+A2 的成果用起来

### 这是 A3 的内容

A3 需要你实现 **`DeadCodeDetection`**。这本身是一个**消费者分析**——它不自己计算数据流，而是消费 A1 的活跃变量结果和 A2 的常量传播结果。

### A3 的核心思路：两种死代码

**类型一：不可达代码** — 利用常量传播的结果，判断条件分支是否可以求值为常量。

**类型二：死赋值** — 利用活跃变量分析的结果，判断赋值是否无意义。

### 核心代码实现

#### 阶段一：找不可达代码（用 A2 常量传播结果）

```java
// DeadCodeDetection.java 第 72-130 行 —— A3 你的代码
Set<Stmt> reachable = new HashSet<>();
Queue<Stmt> queue = new LinkedList<>();
queue.add(cfg.getEntry());

while (!queue.isEmpty()) {
    Stmt node = queue.poll();
    for (Edge<Stmt> edge : cfg.getOutEdgesOf(node)) {
        Stmt target = edge.getTarget();
        if (reachable.contains(target)) continue;

        boolean isReachable = true;

        // If with constant condition: skip unreachable branch
        if (node instanceof If ifStmt) {
            CPFact inFact = constants.getInFact(node);          // ← 来自 A2！
            Value condVal = ConstantPropagation.evaluate(
                    ifStmt.getCondition(), inFact);              // ← 来自 A2！
            if (condVal.isConstant()) {
                if (condVal.getConstant() == 0) {
                    // 条件恒为 false → IF_TRUE 边不可达
                    if (edge.getKind() == Edge.Kind.IF_TRUE) {
                        isReachable = false;
                    }
                } else {
                    // 条件恒为 true → IF_FALSE 边不可达
                    if (edge.getKind() == Edge.Kind.IF_FALSE) {
                        isReachable = false;
                    }
                }
            }
        }

        // Switch with constant key: skip unreachable cases
        if (node instanceof SwitchStmt switchStmt) {
            CPFact inFact = constants.getInFact(node);          // ← 来自 A2！
            Value keyVal = inFact.get(switchStmt.getVar());
            if (keyVal.isConstant()) {
                int constKey = keyVal.getConstant();
                if (edge.isSwitchCase()) {
                    if (edge.getCaseValue() != constKey) {
                        isReachable = false;
                    }
                } else if (edge.getKind() == Edge.Kind.SWITCH_DEFAULT) {
                    if (switchStmt.getCaseValues().contains(constKey)) {
                        isReachable = false;
                    }
                }
            }
        }

        if (isReachable) {
            reachable.add(target);
            queue.add(target);
        }
    }
}

// 遍历不到的节点 = 不可达代码（排除 CFG 的 entry 和 exit 虚拟节点）
for (Stmt node : cfg) {
    if (!reachable.contains(node)
            && !cfg.isEntry(node) && !cfg.isExit(node)) {
        deadCode.add(node);
    }
}
```

#### 阶段二：找死赋值（用 A1 活跃变量结果）

```java
// DeadCodeDetection.java 第 139-153 行 —— A3 你的代码
for (Stmt node : reachable) {      // 只在可达代码里找
    if (node instanceof AssignStmt<?, ?> assign) {
        if (assign.getLValue() instanceof Var var) {
            SetFact<Var> outLiveVars = liveVars.getOutFact(node);  // ← 来自 A1！
            if (!outLiveVars.contains(var)                         // ← 左边变量不活跃
                    && hasNoSideEffect(assign.getRValue())) {      // ← 右边没有副作用
                deadCode.add(node);   // ← 死赋值！
            }
        }
    }
}
```

也就是说对每条赋值语句，去查 A1 的分析结果（`liveVars.getOutFact(node)`），看看等号左边的变量是不是在"将来还会用到"的集合里。如果不在（`!outLiveVars.contains(var)`），而且右边的表达式没有副作用（不是除法、不是 new、不是字段访问），那这个赋值就是死代码。

### 完整推算 — 以 DeadAssignment.java 为例

```java
// DeadAssignment.java —— A3 测试用例
void deadAssign() {
    int x = 1;         // IR: x = 1
    int y = x + 2;     // IR: y = x + %intconst0   ← 死赋值
    int z = x + 3;     // IR: z = x + %intconst1
    use(z);            // IR: invokestatic use(z)
    int a = x;         // IR: a = x                ← 死赋值
}
```

A1 先跑，A3 直接读结果：

```
语句 y = x + %intconst0:
  查 liveVars.getOutFact(此语句)
  如果 y 不在 OUT 中 → y 定义了但没人用 → 死代码！✓

语句 z = x + %intconst1:
  查 liveVars.getOutFact(此语句)
  z 在 OUT 中（因为 use(z) 要用 z）→ 不是死代码

语句 a = x:
  查 liveVars.getOutFact(此语句)
  a 不在 OUT 中 → 死代码！✓
```

A3 最终输出：

```
[2@L5] y = x + %intconst0;    ← 死赋值！（y 没人用）
[6@L8] a = x;                 ← 死赋值！（a 没人用）
```

### 一个容易犯的错误：排除 CFG 的 entry/exit 虚拟节点

CFG 中有两个"虚拟"节点：entry 和 exit。它们不是真实语句，但在不可达性分析中会被标记为 unreachable。必须显式排除：

```java
if (!reachable.contains(node)
        && !cfg.isEntry(node) && !cfg.isExit(node)) {
    deadCode.add(node);
}
```

没有这个判断，`testLoops` 测试会失败——exit 虚拟节点会被当成死代码输出。

### SAST 视角：这解决了什么问题？

死代码检测在安全审计中有直接价值：

1. **不可达分支**：`if (auth_level > 5) { grantAdminAccess(); }` 如果 auth_level 永远是 1，管理员代码就是死代码
2. **死赋值**：被忽略的安全检查结果可能是遗漏的安全缺陷
3. **Switch 不可达 case**：遗漏的错误处理分支

## 第四章：A4 — 过程间常量传播，让数据"穿越"方法边界

### 这是 A4 的内容

A4 是整个学习路径的最高点，有三个任务：

1. **`CHABuilder`** — 类层次分析（CHA）调用图构建
2. **`InterConstantPropagation`** — 过程间常量传播
3. **`InterSolver`** — 过程间 Worklist 求解器

### 4.1 CHA 调用图构建

#### 四种调用指令与 Dispatch 策略

JVM 用四种字节码指令区分调用类型：

| 调用类型 | 指令 | Dispatch 策略 |
|----------|------|--------------|
| static | `invokestatic` | 直接返回唯一目标方法 |
| special | `invokespecial` | 直接返回声明类的方法 |
| virtual | `invokevirtual` | 声明类 + 所有子类 |
| interface | `invokeinterface` | 所有实现该接口的类 |

#### Dispatch 算法实现

```
Dispatch(c, m):
    如果 c 中没有 m 的声明体 (非抽象方法)：
        如果 c 是接口：
            return null
        否则：
            return Dispatch(c.superClass, m)
    否则：
        return c.m
```

#### Resolve 算法：递归收集所有子类

对于 virtual 调用，需要递归收集声明类及其所有子类的 dispatch 结果。使用 BFS 遍历子类层次：

```java
// BFS 遍历子类层次
Queue<JClass> queue = new LinkedList<>();
queue.addAll(hierarchy.getDirectSubclassesOf(jclass));
queue.addAll(hierarchy.getDirectImplementorsOf(jclass));
while (!queue.isEmpty()) {
    JClass current = queue.poll();
    result.add(current);
    queue.addAll(hierarchy.getDirectSubclassesOf(current));
}
```

#### BuildCallGraph：BFS 发现可达方法

从入口方法开始，以 BFS 方式发现可达方法和调用边：

```java
public CallGraph<Invoke, JMethod> buildCallGraph(JMethod entryMethod) {
    DefaultCallGraph callGraph = new DefaultCallGraph();
    callGraph.addReachableMethod(entryMethod);
    Queue<JMethod> worklist = new LinkedList<>();
    worklist.add(entryMethod);
    
    while (!worklist.isEmpty()) {
        JMethod method = worklist.poll();
        for (var callSite : callGraph.callSitesIn(method)) {
            for (JMethod callee : resolve(callSite)) {
                callGraph.addEdge(new Edge<>(callKind, callSite, callee));
                if (callGraph.addReachableMethod(callee)) {
                    worklist.add(callee);
                }
            }
        }
    }
    return callGraph;
}
```

CHA 给出调用关系后，ICFG 就用四种"边"把三个方法的 CFG 缝合起来。

### 4.2 故事起点：A2 在哪里失败了？

回顾 Example.java 中的这四行代码：

```java
a = 6;              // L5
b = addOne(a);      // L6  ← 这里调用了 addOne
c = b - 3;          // L7
```

A2（过程内常量传播）的处理结果：

```
[0] a = 6;          → IN={}  OUT={a=6}              ✓ a=6 算出来了
[1] b = addOne(a);  → IN={a=6}  OUT={a=6, b=NAC}    ✗ b 不知道！因为 addOne 是黑盒
[2] c = b - 3;      → IN={a=6, b=NAC}  OUT={a=6, b=NAC, c=NAC}  ✗ c 也不知道
```

**A2 把 `addOne(a)` 当成黑盒**——它不知道 addOne 内部做了什么，所以 b 变成 NAC（不是常量），然后 c 也被传染。整个程序的精度就丢失了。

A4 的目标：**打开这个黑盒，追踪数据如何穿过方法边界**。

### 4.3 第一步：CHA 画地图（谁调用了谁）

A4 首先要回答："`addOne(a)` 到底调用了哪个具体方法？"

  → STATIC 调用 → 目标唯一 → Example.addOne

CHA 输出：
  可达方法：{main, addOne}  （ten 类似，这里省略）
  调用边：main[L6] --STATIC--> addOne

有了这张地图，InterSolver 就知道 `b = addOne(a)` 通向 `addOne` 方法。

### 4.5 第三步：四种边各自做什么——逐条代码讲解

当你把四种边的作用分开理解时，一切就清楚了。每条边的 `transferEdge` 函数只做一件简单的事。

#### 边① NormalEdge：什么都不做

```java
// InterConstantPropagation.java 第 99-102 行
protected CPFact transferNormalEdge(NormalEdge<Stmt> edge, CPFact out) {
    return out.copy();   // 原样照抄
}
```

**作用**：方法内部节点之间、方法 internal 的 entry→第一条语句之间，数据原封不动传递。

**例子**：`[0] a=6` 的 OUT=`{a=6}`，通过 NormalEdge 传给 `[1]`，`[1]` 收到 `{a=6}`。什么都没改。

#### 边③ CallEdge：把参数喂给被调方法

```java
// InterConstantPropagation.java 第 115-127 行
protected CPFact transferCallEdge(CallEdge<Stmt> edge, CPFact callSiteOut) {
    CPFact result = new CPFact();    // 全新的 fact，只装形参
    for (int i = 0; i < invokeExp.getArgCount(); i++) {
        Var arg = invokeExp.getArg(i);        // 实参 a
        Var param = callee.getIR().getParam(i); // 形参 x
        result.update(param, callSiteOut.get(arg));  // {x = a的值}
    }
    return result;  // 只返回形参的映射，其他变量一概不带
}
```

**作用**：从调用点的 OUT 中提取实参的值，赋给被调方法的形参。**其他变量（如调用者自己的局部变量）不传**——被调方法不需要知道调用者的 `a`，它只需要知道自己的 `x`。

**例子**：调用点 `[1]` 的 OUT=`{a=6}` → CallEdge 提取 `a=6`，赋给形参 `x` → 返回 `{x=6}` → addOne 的 entry 收到 `{x=6}`。

#### 边② CallToReturnEdge：切断未确定的变量

```java
// InterConstantPropagation.java 第 104-113 行
protected CPFact transferCallToReturnEdge(CallToReturnEdge<Stmt> edge, CPFact out) {
    CPFact result = out.copy();    // 先全盘复制
    Stmt callSite = edge.getSource();
    if (callSite instanceof DefinitionStmt<?, ?> defStmt) {
        var lhs = defStmt.getLValue();   // temp$1
        if (lhs instanceof Var var && canHoldInt(var)) {
            result.remove(var);          // 把 temp$1 删掉！
        }
    }
    return result;  // 返回 {a=6}（temp$1 被删了，a 保留）
}
```

**作用**：让数据在"调用还没返回"时就先流到调用后面的语句。**kill 掉等号左边的变量**（因为方法还没返回，这个变量的值还不确定），但**保留其他所有变量**。

**例子**：`[1] temp$1 = addOne(a)` 的 OUT=`{a=6}`，走 CallToReturnEdge 到 `[2]`。kill temp$1（它还不知道是多少），保留 a=6 → 返回 `{a=6}`。

> **这条边的设计动机**：如果没有这条边，`[2]` 只能等 ReturnEdge 的数据到达才开始计算。但 ReturnEdge 要等 addOne 整个方法跑完才会有值。CallToReturnEdge 保证了**不依赖方法的变量的值（如 a=6）能提前到达后面语句**。

#### 边④ ReturnEdge：把结果传回来

```java
// InterConstantPropagation.java 第 129-143 行
protected CPFact transferReturnEdge(ReturnEdge<Stmt> edge, CPFact returnOut) {
    CPFact result = new CPFact();    // 全新的 fact，只装调用点 lhs
    Stmt callSite = edge.getCallSite();
    if (callSite instanceof DefinitionStmt<?, ?> defStmt) {
        var lhs = defStmt.getLValue();   // temp$1
        for (Var retVar : edge.getReturnVars()) {     // 返回值变量 y
            Value val = cp.meetValue(result.get(lhs), returnOut.get(retVar));
            result.update(lhs, val);   // {temp$1 = y的值}
        }
    }
    return result;  // 只返回 lhs 的映射
}
```

**作用**：从被调方法的 exit 的 OUT 中提取返回变量的值，赋给调用点的左侧变量。

**例子**：addOne 的 exit OUT=`{x=6, y=7}` → ReturnEdge 提取 `y=7`，赋给 `temp$1` → 返回 `{temp$1=7}`。注意只传了 temp$1，没有传 x——x 是 addOne 的内部变量，调用者不需要它。

### 4.6 第四步：四种边合在一起——完整数据流追踪

现在把四条边放在一起，看 `[2] b = temp$1` 的 IN 怎么得到 `{a=6, temp$1=7}`：

```
第一步：addOne 内部执行完毕
  addOne.exit.OUT = {x=6, y=7}

第二步：InterSolver 处理节点 [2] b = temp$1
  IN[2] 初始 = {}

  入边② CallToReturnEdge [1]→[2]:
    transferEdge(边②, OUT[1])  →  OUT[1]={a=6}, kill temp$1 → {a=6}
    meetInto({a=6}, IN[2])      →  IN[2] = {a=6}

  入边④ ReturnEdge addOne.exit→[2]:
    transferEdge(边④, OUT[addOne.exit])  →  {x=6, y=7}, 取 y=7 赋给 temp$1 → {temp$1=7}
    meetInto({temp$1=7}, IN[2])          →  IN[2] = {a=6, temp$1=7}  ← 合并完毕！

第三步：transferNode 处理 "b = temp$1"
  IN = {a=6, temp$1=7}
  evaluate(temp$1) = 7 → out.update(b, 7)
  OUT = {a=6, temp$1=7, b=7}  ← ★ b=7！
```

**每条边只运一部分数据**：

| 边 | 运了什么 | 没运什么 | 原因 |
|----|---------|---------|------|
| CallToReturnEdge | `a=6` | `temp$1` | temp$1 的值还没确定（addOne 没返回） |
| ReturnEdge | `temp$1=7` | `x=6` | x 是 addOne 的局部变量，调用者不需要 |

两条边 meet 后：`{a=6}` meet `{temp$1=7}` = `{a=6, temp$1=7}`。完整了。

### 4.7 四种边速查表

| 边 | 什么时候触发 | 做什么 | 输入 → 输出 | 为什么需要它 |
|----|-------------|--------|------------|------------|
| **NormalEdge** | 任何时候（方法内普通跳转） | 恒等 | `{a=6}` → `{a=6}` | 方法内数据流动的基础 |
| **CallEdge** | 调用发生时（进入被调方法） | 实参→形参 | `{a=6}` → `{x=6}` | 被调方法需要知道参数值 |
| **CallToReturnEdge** | 调用发生时（同时走调用后面的路） | kill lhs | `{a=6}` → `{a=6}`（temp$1 被删） | 让不依赖方法调用的变量提前传播 |
| **ReturnEdge** | 方法返回时 | 返回值→lhs | `{x=6, y=7}` → `{temp$1=7}` | 被调方法的结果要传回来 |

### 4.8 InterSolver 实现

InterSolver 与 A2 的 WorkListSolver 只有一处关键区别——**IN 的计算**：

```java
// A2：过程内
for (Node pred : cfg.getPredsOf(node)) {
    analysis.meetInto(result.getOutFact(pred), in);
}

// A4：过程间 —— InterSolver.java 第 88-108 行
for (var edge : icfg.getInEdgesOf(node)) {
    Fact edgeFact = analysis.transferEdge(edge, result.getOutFact(edge.getSource()));
    analysis.meetInto(edgeFact, in);
}
```

A4 在 merge 之前多了一步 `transferEdge`——根据边的类型决定数据如何变形。初始化逻辑也不同（入口方法的 entry 设 boundary fact，其他节点全空），但主循环骨架和 A2 一模一样。

## 第五章：A1→A4 的学习路径总结

### 一条线索串起四个实验

```
A1 活跃变量 ─────────────────────┐
                                  ├──→ A3 死代码检测
A2 常量传播 ─────────────────────┘
      │
      └──→ A4 CHA（构建调用图）
                │
                └──→ A4 过程间常量传播
                          │
                          └──→ A4 InterSolver（过程间 Worklist）
```

### 四层递进，一句话记

| 作业 | 做什么 | 输入 | 输出 | 方向 | 求解器 |
|------|--------|------|------|------|--------|
| A1 | 倒着推，找出"将来还要用的变量" | 源代码 | 每个语句后的活跃变量集 | 后向 | IterativeSolver |
| A2 | 顺着推，找出"值确定的变量" | 源代码 | 每个语句前的常量映射 | 前向 | WorkListSolver |
| A3 | 消费 A1+A2 | A1的活跃集 + A2的常量映射 | 死代码列表 | — | — |
| A4 | 打破方法边界，让常量穿越调用 | CHA调用图 + ICFG + A2逻辑 | 跨方法的精确常量 | 前向 | InterSolver |

### A2 vs A4：过程内 vs 过程间的本质区别

A2 和 A4 共享同一套 `ConstantPropagation` 核心逻辑，区别只在于：

- **A2**：遇到 `invokestatic addOne(a)` → `evaluate` 返回 null → 设为 NAC → 丢失精度
- **A4**：遇到 `invokestatic addOne(a)` → ICFG 的 CallEdge 传实参给形参 → ReturnEdge 传返回值回来 → **保持精度**

A2 就像在没有地图的情况下走路，走到路口（方法调用）就停下来了。A4 有了 CHA 这张地图（调用图），知道路口通向哪里，就敢继续往前走。

### A3 找 "c 没人用" 靠的是 A1 的结果

A3 的 `DeadCodeDetection` 读 A1 的 `liveVars.getOutFact(node)`，检查 `!outLiveVars.contains(var)` ——左边变量不在活跃集里，就是死赋值。

### A4 把 b=7 算出来靠的是两条边 meet

A4 的 `[2] b = temp$1` 有 CallToReturnEdge（传来 `a=6`）和 ReturnEdge（传来 `temp$1=7`），两条边的 edge transfer 结果 meet 后完整。

### 对 SAST 项目的启示

如果你在做漏洞扫描，这套框架可以直接扩展：

1. **污点分析**本质上是常量传播的对偶——常量传播跟踪具体值，污点分析跟踪"污染标签"
2. **CHA 调用图**虽然不够精确（会有虚假边），但对于快速扫描足够了
3. **Edge Transfer 模式**直接适用于跨方法的污点传播——CallEdge 传递标签，ReturnEdge 回收标签
4. **InterSolver**不需要修改，只需替换具体的 transfer 逻辑即可
