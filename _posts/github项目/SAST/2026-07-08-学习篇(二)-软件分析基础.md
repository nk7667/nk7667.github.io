## 第六章：A5 — 上下文不敏感指针

A5 需要你实现一个完整的**上下文不敏感指针分析（Context-Insensitive Pointer Analysis）**：

1. **`Solver`** — 指针分析求解器
2. **`PointerFlowGraph`** — 指针流图（PFG）

### 从数据流到指针——思维转换

A5 是整个学习路径的**转折点**。A1-A4 是数据流分析，A5 开始进入指针分析。两种分析在框架层面的区别：

|                    | A1-A4 数据流路线                                            | A5-A8 指针路线                                      |
| ------------------ | ---------------------------------------------------------- | --------------------------------------------------- |
| 分析框架           | **数据流分析**（CFG/ICFG + transfer function + meet）      | **指针分析**（PFG + WorkList + propagation）         |
| 核心数据结构       | CPFact（Var → Value 映射）                                 | PointsToSet（Var → {Obj} 映射）                     |
| 传播方式           | 逐语句 transfer（IN→OUT），调用时通过四种 ICFG 边变形传递  | 沿 PFG 边传播（source→target），调用时沿实参→形参、返回值→result 边 |
| 格（Lattice）      | UNDEF → 常量 → NAC                                         | 空集 → 对象集合                                     |
| 调用图             | CHA 或 CS CallGraph                                        | 与 PTA 交替推导（On-the-fly）                       |

A4 的四种 ICFG 边做"值变形"（提取参数/kill lhs），A5 的 PFG 边做"原样转发"（对象从 source 直接复制到 target）。这是两套完全不同的计算模型。

代码操作的对应关系：

- `workList.addEntry(varPtr, pts)` = 把新对象加入 varPtr 的 pts
- `propagate(pointer, pts)` = 计算增量 diff，沿 PFG 边传播给后继
- `addPFGEdge(from, to)` = 建立传播边：from 的新对象会传给 to
- `diff` = 只传播新增对象，已存在的不再重复传播

### 核心概念：指针分析在干什么

指针分析回答一个基本问题：**程序运行时，一个指针（变量、字段等）可能指向哪些对象？**

```java
String s = new String();  // s 指向这个新创建的 String 对象
String t = s;             // t 也指向同一个 String 对象
```

如果分析知道 `s` 和 `t` 指向同一个对象，就能判断 `s.length()` 和 `t.length()` 操作的是同一份数据——这对于后续的别名分析、污点分析都至关重要。

### 为什么叫"上下文不敏感"

上下文不敏感（CI）意味着：**不区分同一个方法在不同调用场景下的差异**。

```java
void foo(Object o) {
    bar(o);    // bar 接收的 o 可能是任何对象
}
void bar(Object p) {
    p.toString();  // CI 分析认为 p 可能指向 foo 被调用时传入的所有对象
}

foo(a);  // 第一次调用
foo(b);  // 第二次调用
```

CI 分析把两次调用合并在一起，`p` 的 points-to set 包含 `a` 和 `b` 指向的所有对象。精度不够，但实现简单。

### 指针流图（PFG）

指针流图（Pointer Flow Graph）是指针分析的核心数据结构。**PFG 的边 = 传播规则**：当 source 端获得了新对象，这个对象就沿边传到 target 端。

五种指针节点：

| 节点类型          | 含义   | 示例                        |
| ------------- | ---- | ------------------------- |
| VarPtr        | 局部变量 | `x`, `y`, `%this`         |
| StaticField   | 静态字段 | `A.f`                     |
| InstanceField | 实例字段 | `(obj, A.f)` — 某个对象的 f 字段 |
| ArrayIndex    | 数组索引 | `(arr)` — 某个数组的内容（不区分索引）  |

四种 PFG 边的传播规则（比 A4 的四种 ICFG 边更简单）：

| 边类型            | 代码形式          | PFG 边            | 传播含义                       |
| -------------- | ------------- | ---------------- | -------------------------- |
| **New**        | `x = new T()` | 无边（直接加入 pts）     | 创建对象，直接放进 x 的集合            |
| **Copy**       | `x = y`       | `y → x`          | y 获得新对象 → x 也获得            |
| **StoreField** | `x.f = y`     | `y → (x的obj, f)` | y 获得新对象 → 写进 x 指向的对象的 f 字段 |
| **LoadField**  | `y = x.f`     | `(x的obj, f) → y` | x 指向的对象的 f 字段获得新对象 → y 也获得 |

**关键区别**：A4 的边是 ICFG 边（方法间数据流边），A5 的边是 PFG 边（对象传播边）。ICFG 边做"值变形"（提取参数/kill lhs），PFG 边做"原样转发"（对象从 source 直接复制到 target）。

**为什么实例字段的边是"按需"创建的？**

`x.f = y` 这条语句中，`x` 可能指向多个对象（o1, o2, ...）。在初始化阶段，我们还不知道 `x` 指向谁，所以无法创建 `y → (o1, f)` 这样的边。只有当 propagate 发现 `x` 的 pts 增长了新对象 o1 时，才为 o1 创建 `y → InstanceField(o1, f)` 边。

这就是"按需处理"的含义：**边不是在初始化时一次性建好，而是在传播过程中动态添加**。

### 核心代码实现

#### 初始化：从 main 方法出发

```java
// Solver.java — A5 你的代码
private void initialize() {
    workList = new WorkList();
    pointerFlowGraph = new PointerFlowGraph();
    callGraph = new DefaultCallGraph();
    // 从 main 方法开始
    JMethod main = World.get().getMainMethod();
    callGraph.addReachableMethod(main);
    addReachable(main);
}
```

`addReachable` 遍历方法的所有语句，用访问者模式处理七种语句。

#### 七种语句的处理

```java
// StmtProcessor — A5 你的代码
@Override
public Void visit(New stmt) {
    // x = new T() → 创建对象，直接加入 x 的 pts
    Obj obj = heapModel.getObj(stmt);
    VarPtr varPtr = pointerFlowGraph.getVarPtr(stmt.getLValue());
    PointsToSet pts = PointsToSetFactory.make(obj);
    workList.addEntry(varPtr, pts);
    return null;
}

@Override
public Void visit(Copy stmt) {
    // x = y → 添加 PFG 边 y -> x
    VarPtr from = pointerFlowGraph.getVarPtr(stmt.getRValue());
    VarPtr to = pointerFlowGraph.getVarPtr(stmt.getLValue());
    addPFGEdge(from, to);
    return null;
}

@Override
public Void visit(StoreField stmt) {
    // x.f = y → 静态字段: y -> StaticField(f)
    //           实例字段: 需要等 x 的 pts 算出来再处理
    if (stmt.isStatic()) {
        VarPtr from = pointerFlowGraph.getVarPtr(stmt.getRValue());
        StaticField to = pointerFlowGraph.getStaticField(
                stmt.getFieldRef().resolve());
        addPFGEdge(from, to);
    }
    // 实例字段在 propagate 时按需处理
    return null;
}

@Override
public Void visit(LoadField stmt) {
    // y = x.f → 静态字段: StaticField(f) -> y
    //           实例字段: 需要等 x 的 pts 算出来再处理
    if (stmt.isStatic()) {
        StaticField from = pointerFlowGraph.getStaticField(
                stmt.getFieldRef().resolve());
        VarPtr to = pointerFlowGraph.getVarPtr(stmt.getLValue());
        addPFGEdge(from, to);
    }
    return null;
}

// StoreArray/LoadArray 类似：需要等数组变量的 pts 算出来
```

**关键设计**：实例字段和数组访问不能在 `addReachable` 时处理，因为那时还不知道 base 变量指向哪些对象。它们在 `propagate` 中按需处理——当 base 变量的 pts 增长时，才为新的对象添加对应的 PFG 边。

#### propagate：增量传播的核心

```java
// Solver.java — A5 你的代码
private PointsToSet propagate(Pointer pointer, PointsToSet pts) {
    PointsToSet oldPts = pointer.getPointsToSet();
    // 计算增量 diff = pts - oldPts
    PointsToSet diff = PointsToSetFactory.make();
    for (Obj obj : pts) {
        if (oldPts.addObject(obj)) {  // addObject 返回 true 表示是新对象
            diff.addObject(obj);
        }
    }
    if (diff.isEmpty()) return diff;  // 没有新对象，不需要传播

    // 传播给 PFG 后继
    for (Pointer succ : pointerFlowGraph.getSuccsOf(pointer)) {
        workList.addEntry(succ, diff);
    }

    // ★ 按需处理实例字段/数组/调用
    if (pointer instanceof VarPtr varPtr) {
        Var var = varPtr.getVar();
        for (Obj obj : diff) {
            // x.f = y: 对每个 obj ∈ pts(x), 添加 y -> (obj, f)
            for (StoreField store : var.getStoreFields()) {
                if (!store.isStatic()) {
                    InstanceField iField = pointerFlowGraph.getInstanceField(
                            obj, store.getFieldRef().resolve());
                    VarPtr from = pointerFlowGraph.getVarPtr(store.getRValue());
                    addPFGEdge(from, iField);
                }
            }
            // y = x.f: 对每个 obj ∈ pts(x), 添加 (obj, f) -> y
            for (LoadField load : var.getLoadFields()) {
                if (!load.isStatic()) {
                    InstanceField iField = pointerFlowGraph.getInstanceField(
                            obj, load.getFieldRef().resolve());
                    VarPtr to = pointerFlowGraph.getVarPtr(load.getLValue());
                    addPFGEdge(iField, to);
                }
            }
            // 数组同理
            for (StoreArray store : var.getStoreArrays()) { ... }
            for (LoadArray load : var.getLoadArrays()) { ... }
            // 调用：processCall
            processCall(var, obj);
        }
    }
    return diff;
}
```

**增量传播**是效率的关键：只传播新增加的对象（diff），而不是每次都传播全量。

#### processCall：虚方法的解析与传播

```java
// Solver.java — A5 你的代码
private void processCall(Var var, Obj recv) {
    for (Invoke invoke : var.getInvokes()) {
        if (invoke.isStatic()) continue;  // 静态调用不在这里处理

        // 根据接收者类型解析 callee
        JMethod callee = resolveCallee(recv, invoke);
        if (callee == null) continue;

        // 添加调用图边
        if (callGraph.addEdge(new Edge<>(kind, invoke, callee))) {
            addReachable(callee);
            // 传 receiver → this
            VarPtr thisVar = pointerFlowGraph.getVarPtr(
                    callee.getIR().getThis());
            workList.addEntry(thisVar, PointsToSetFactory.make(recv));
            // 传参数
            for (int i = 0; i < invokeExp.getArgCount(); i++) {
                VarPtr from = pointerFlowGraph.getVarPtr(invokeExp.getArg(i));
                VarPtr to = pointerFlowGraph.getVarPtr(
                        callee.getIR().getParam(i));
                addPFGEdge(from, to);
            }
            // 传返回值
            if (invoke.getResult() != null) {
                for (Var retVar : callee.getIR().getReturnVars()) {
                    VarPtr from = pointerFlowGraph.getVarPtr(retVar);
                    VarPtr to = pointerFlowGraph.getVarPtr(invoke.getResult());
                    addPFGEdge(from, to);
                }
            }
        }
    }
}
```

**核心逻辑**：当变量 `var` 的 pts 增长（新对象 `recv` 加入），检查 `var` 是否是某个实例调用的 receiver。如果是，就用 `recv` 的类型解析具体 callee，然后建立调用图边和 PFG 传播边。

### 完整推算过程 — 简单示例（字段传播）

```java
void main() {
    A a = new A();      // [0] a 指向 o1
    a.f = "hello";      // [1] o1.f 指向 s1
    String s = a.f;     // [2] s 应该指向 s1
}
```

PFG 边构建过程：

```
1. 处理 [0] a = new A():
   创建 o1, workList: { a → {o1} }

2. 处理 workList entry: a → {o1}
   propagate(a, {o1}):
     diff = {o1}  (a 之前是空集)
     - a 有 StoreField: a.f = "hello" → 为 o1 创建 InstanceField(o1, f)
       添加 PFG 边: "hello的变量" → InstanceField(o1, f)
     - a 有 LoadField: s = a.f → 为 o1 创建 InstanceField(o1, f)
       添加 PFG 边: InstanceField(o1, f) → s

3. 处理 "hello" 的传播:
   InstanceField(o1, f) 收到 {s1}
   → 传播给 s: s 的 pts = {s1}  ✓
```

### 完整推算过程 — 方法调用示例

```java
class Id {
    static Object id(Object o) { return o; }
}
void main() {
    Object a = new A();    // [0] a 指向 o1
    Object b = id(a);      // [1] b 应该也指向 o1
    Object c = id(b);      // [2] c 应该也指向 o1
}
```

推算过程（完整 WorkList 循环）：

```
=== 初始化阶段 ===

处理 main 方法，扫描语句：

[0] a = new A():
  → 创建 o1, 加入 workList: { VarPtr(a) → {o1} }

[1] b = id(a):
  → Copy 边? 不是 Copy, 是调用。静态调用 a 是 receiver?
  → 实际上 id 是静态方法，invokeExp.getBase() = a
  → a 是实例调用的 receiver: 注册 a.getInvokes() = [id(a)]
  → 等到 a 的 pts 增长时才处理 processCall

[2] c = id(b):
  → 同理: 注册 b.getInvokes() = [id(b)]
  → 等到 b 的 pts 增长时才处理 processCall

=== WorkList 循环 ===

⏱ 第1轮: 处理 workList entry: VarPtr(a) → {o1}

  propagate(VarPtr(a), {o1}):
    diff = {o1}  (a 之前是空集)
    
    ① a 有 invokes: id(a) → processCall(a, o1)
       resolveCallee(o1, id) → 因为 id 是静态方法，特殊处理
       （注意：id 是 static 方法，所以用 invokestatic 解析）
       → callGraph.addEdge(main[1] → id)
       → addReachable(id): 扫描 id 方法的语句
         id 中只有 return o; → 这是 Copy 边: o → result(id)
         但 result 是 id 方法的返回值变量
      
       传参数: a → id 的形参 o
         addPFGEdge(VarPtr(a), VarPtr(o_in_id))  ← a→o 参数边
         但 a 已经有 {o1}, 所以也 workList: VarPtr(o_in_id) → {o1}
       
       传返回值: id 的 return → main 的 b
         addPFGEdge(VarPtr(return_of_id), VarPtr(b))  ← 返回值边
    
    ② a 有 invokes: id(b)???  不是, b.getInvokes() 才是 id(b)
       a 只有一个 invoke: id(a)

⏱ 第2轮: 处理 workList entry: VarPtr(o_in_id) → {o1}

  propagate(VarPtr(o_in_id), {o1}):
    diff = {o1}
    
    o_in_id 有 PFG 后继:
    - Copy 边 o → return_of_id（id 方法内部: return o;）
      → workList: VarPtr(return_of_id) → {o1}

⏱ 第3轮: 处理 workList entry: VarPtr(return_of_id) → {o1}

  propagate(VarPtr(return_of_id), {o1}):
    diff = {o1}
    
    return_of_id 有 PFG 后继:
    - 返回值边: return_of_id → b（在 processCall 时建立的）
      → workList: VarPtr(b) → {o1}

⏱ 第4轮: 处理 workList entry: VarPtr(b) → {o1}

  propagate(VarPtr(b), {o1}):
    diff = {o1}  (b 之前是空集)
    
    b 有 invokes: id(b) → processCall(b, o1)
       resolveCallee(o1, id) → id 方法
       → callGraph.addEdge(main[2] → id)  ← 第二次调用边
       
       但 id 已经 reachable 了，不会再 addReachable
       传参数: b → id 的形参 o
         addPFGEdge(VarPtr(b), VarPtr(o_in_id))
         但 b 已经有 {o1}, 所以 workList: VarPtr(o_in_id) → {o1}
         ← 注意: o_in_id 已经有 {o1} 了，这次 diff 是空的
       
       传返回值: id 的 return → main 的 c
         addPFGEdge(VarPtr(return_of_id), VarPtr(c))

⏱ 第5轮: 处理 workList entry: VarPtr(o_in_id) → {o1}

  propagate(VarPtr(o_in_id), {o1}):
    o_in_id 已经有 {o1}，diff = 空 → 不传播任何东西 ✓

=== 最终结果 ===

a 的 pts = {o1}    ← [0] 直接 new
b 的 pts = {o1}    ← 从 id(a) 的返回值传回来
c 的 pts = {o1}    ← 从 id(b) 的返回值传回来

这就是指针分析的传播链条:
  a →(参数边)→ o_in_id →(Copy边)→ return_of_id →(返回值边)→ b
  b →(参数边)→ o_in_id →(Copy边)→ return_of_id →(返回值边)→ c
```

**对比 A4 的理解**：

A4 用 ICFG 的四种边传播**常量值**（Value），A5 用 PFG 边传播**对象**（Obj）。A4 的 CallEdge 做"提取参数/kill lhs"，A5 的 processCall 做"建立 PFG 边/传 receiver"。传播的数据不同，但"沿边传递"的骨架是一样的。

### A5 执行流程总结

```
initialize()          → 扫描 main 方法
  addReachable()        → 处理七种语句，注册 PFG 边和 workList entry

主循环: while workList 不空
  poll entry           → 取出 (pointer, pts)
  propagate()          → 计算 diff，传播给 PFG 后继
    如果是 VarPtr:      → 按需处理实例字段/数组/调用
      StoreField       → 为新 obj 创建 y → InstanceField(obj, f) 边
      LoadField        → 为新 obj 创建 InstanceField(obj, f) → y 边
      processCall      → 解析 callee，建立参数/返回值边，传 receiver
```

指针分析是 SAST 的基础设施：

**判断别名**：两个引用是否指向同一对象？→ 影响污点传播路径

- **虚调用解析**：`obj.method()` 到底调的是哪个实现？→ 影响调用图精度
- **字段访问建模**：`x.f = taint; y = z.f` 如果 x 和 z 指向同一对象，y 就是污点

CI-PTA 不区分调用场景，导致同一方法的多次调用混在一起。A6 用上下文敏感性解决这个精度问题。

## 第七章：A6 — 上下文敏感指针分析，区分调用场景

A6 需要你实现**上下文敏感指针分析**和**上下文选择器**：

1. **`Solver`** — 上下文敏感的指针分析求解器
2. **`_2ObjSelector`** — 2-对象敏感性上下文选择器（以及 1-obj、1-call 等其他选择器）

### A5 的精度问题在哪？

考虑这个经典例子：

```java
class Id {
    static Object id(Object o) { return o; }
}
void main() {
    Number n1 = new One();     // n1 → o1
    Number n2 = new Two();     // n2 → o2
    Number x = Id.id(n1);      // 期望 x → o1
    Number y = Id.id(n2);      // 期望 y → o2
}
```

**A5（CI）的结果**：

```
id 方法的 o 指向 {o1, o2}    ← 两次调用的实参混在一起
id 方法的 return 指向 {o1, o2}
x = id(n1) → x 指向 {o1, o2}  ← 精度丢失，实际 x 只能是 o1
y = id(n2) → y 指向 {o1, o2}  ← 精度丢失，实际 y 只能是 o2
```

CI-PTA 对 `id` 方法只有一份分析，变量 `o` 只有一个。两次调用 `id(n1)` 和 `id(n2)` 都往同一个 `o` 传参数，`o1` 和 `o2` 就混进了同一个集合。返回值也只有一个 `return`，它指向 {o1, o2}，传回给 x 和 y 时两个都收到了全部对象。

A6 为 `id(n1)` 和 `id(n2)` 创建两个独立的分析副本（上下文分别是 o1 和 o2），每个副本有自己的 `o` 和 `return`，互不干扰。

### 怎么区分调用场景？

上下文选择器决定"用什么信息来区分同一方法的不同调用"。A6 需要实现多种选择器：

| 选择器       | 方法上下文           | 堆上下文              | 精度     | 代价     |
| --------- | --------------- | ----------------- | ------ | ------ |
| CI        | 总是空             | 总是空               | 最低     | 最低     |
| 1-call    | 最近 1 个调用点       | 空                 | 中      | 中      |
| 2-call    | 最近 2 个调用点       | 方法上下文             | 较高     | 较高     |
| 1-obj     | 最近 1 个接收者对象     | 空                 | 中      | 中      |
| **2-obj** | **最近 2 个接收者对象** | **方法上下文最后 1 个元素** | **较高** | **较高** |
| 1-type    | 最近 1 个接收者类型     | 空                 | 中      | 中      |
| 2-type    | 最近 2 个接收者类型     | 方法上下文最后 1 个元素     | 较高     | 较高     |

#### 2-ObjSelector 实现

```java
// _2ObjSelector.java — A6 你的代码
public class _2ObjSelector implements ContextSelector {

    @Override
    public Context getEmptyContext() {
        return ListContext.make();
    }

    // 静态调用：不区分上下文
    @Override
    public Context selectContext(CSCallSite callSite, JMethod callee) {
        return getEmptyContext();
    }

    // 实例调用：取接收者对象作为上下文
    @Override
    public Context selectContext(CSCallSite callSite, CSObj recv, JMethod callee) {
        Obj cur = recv.getObject();          // 当前接收者对象
        Context recvCtx = recv.getContext(); // 接收者对象的上下文
        int len = recvCtx.getLength();
        if (len >= 1) {
            // 如果接收者上下文的最后一个元素就是 cur，
            // 说明 1-obj 已经能区分，直接返回
            if (recvCtx.getElementAt(len - 1).equals(cur)) {
                return recvCtx;
            }
            // 否则取上下文最后 1 个元素 + cur，组成长度 2 的上下文
            return ListContext.make(recvCtx.getElementAt(len - 1), cur);
        }
        // 上下文为空，返回 [cur]
        return ListContext.make(cur);
    }

    // 堆上下文：取方法上下文的最后 1 个元素
    @Override
    public Context selectHeapContext(CSMethod method, Obj obj) {
        Context ctx = method.getContext();
        int len = ctx.getLength();
        if (len >= 1) {
            return ListContext.make(ctx.getElementAt(len - 1));
        }
        return ListContext.make();
    }
}
```

**2-obj 敏感性**：用最近 2 个接收者对象的分配点作为方法上下文。例如 `_2ObjSelector.selectContext` 返回 `ListContext.make(recvCtx的最后1个元素, 当前cur)`，这样对同一方法的不同接收者对象会产生不同的上下文。

### A6 Solver vs A5 Solver：关键区别

A6 的 Solver 和 A5 结构几乎一样，但所有节点变成了**上下文敏感版本**：

| A5 (CI)   | A6 (CS)                          | 区别      |
| --------- | -------------------------------- | ------- |
| `Var`     | `CSVar = (Context, Var)`         | 变量带上下文  |
| `Obj`     | `CSObj = (Context, Obj)`         | 对象带堆上下文 |
| `JMethod` | `CSMethod = (Context, JMethod)`  | 方法带上下文  |
| `Invoke`  | `CSCallSite = (Context, Invoke)` | 调用点带上下文 |

核心代码变化：

```java
// A5: Var var → VarPtr
VarPtr varPtr = pointerFlowGraph.getVarPtr(var);

// A6: Var var + Context context → CSVar
CSVar csVar = csManager.getCSVar(context, var);

// A5: New → 直接创建 Obj
Obj obj = heapModel.getObj(stmt);

// A6: New → 创建 CSObj（需要选择堆上下文）
Obj obj = heapModel.getObj(stmt);
Context heapContext = contextSelector.selectHeapContext(csMethod, obj);
CSObj csObj = csManager.getCSObj(heapContext, obj);

// A5: processCall → 选择 callee，直接传播
JMethod callee = resolveCallee(recv, invoke);

// A6: processCall → 选择 callee 的上下文，然后在对应上下文下传播
Context calleeContext = contextSelector.selectContext(
        csCallSite, recvObj, callee);
CSMethod csCallee = csManager.getCSMethod(calleeContext, callee);
```

### 上下文选择的过程

```
main() { 
    n1 = new One();        // o1, heapCtx=[]
    n2 = new Two();        // o2, heapCtx=[]
    x = Id.id(n1);         // callSite: main[4], receiver: n1
    y = Id.id(n2);         // callSite: main[6], receiver: n2
}

第一次调用 Id.id(n1):
  recvObj = CSObj(ctx=[], One)
  selectContext(callSite, recvObj, id):
    cur = One对象
    recvCtx = []  (空)
    return ListContext.make(One对象) = [One]
  → id 在上下文 [One] 下被分析
  → id[One].o 指向 {CSObj([], One)} = 只有 o1 ✓

第二次调用 Id.id(n2):
  recvObj = CSObj(ctx=[], Two)
  selectContext(callSite, recvObj, id):
    cur = Two对象
    recvCtx = []  (空)
    return ListContext.make(Two对象) = [Two]
  → id 在上下文 [Two] 下被分析
  → id[Two].o 指向 {CSObj([], Two)} = 只有 o2 ✓

返回值:
  x = id(n1) → return from id[One] → x 指向 {o1} ✓
  y = id(n2) → return from id[Two] → y 指向 {o2} ✓
```

### A5 vs A6 总结

| <br /> | A5 (CI)          | A6 (CS, 2-obj)                          |
| ------ | ---------------- | --------------------------------------- |
| 方法表示   | `JMethod`（唯一）    | `CSMethod = (Context, JMethod)`（按上下文区分） |
| 变量表示   | `Var`            | `CSVar = (Context, Var)`                |
| 对象表示   | `Obj`            | `CSObj = (HeapContext, Obj)`            |
| 虚调用解析  | 所有 receiver 混在一起 | 按上下文分开处理                                |
| 精度     | 低（同一方法只有一份分析结果）  | 高（不同调用场景独立分析）                           |
| 代价     | 低                | 高（上下文数量增加，分析规模膨胀）                       |

上下文敏感性让指针分析精确了，但这只是指针路线的中段。接下来是整个体系最关键的交汇点——A7，它把 A4 的数据流框架和 A6 的指针结果融合在一起。

## 第八章：A7 — 别名感知的过程间常量传播

### 为什么需要 A7：A4 的瓶颈在哪

A4 已经实现了过程间常量传播：数据能穿越方法边界，`addOne(a)` 能算出来返回 7。但它对**堆访问**始终返回 NAC：

```java
x.f = 123;
y = x.f;   // A4: y = NAC
```

为什么不直接在 A4 里改进？因为要知道 `x.f` 的值，首先要回答：**`x`** **指向哪个对象？** 这是一个指针分析问题，A4 的数据流框架无法回答（CPFact：变量→值）。必须借助指针分析（PointsToSet：变量→{对象}）。

这就是 A6→A7 的动机：**A6 解决了"x 指向谁"，A7 利用这个信息算出"x.f 的值是多少"**。

### A7 的框架：复用了什么，新增了什么

A7 沿用了 A4 的过程间框架（ICFG + InterSolver + 四种边），直接复用的部分：

| 复用自 A4                               | 作用                                   |
| ------------------------------------ | ------------------------------------ |
| ICFG                                 | 把多个方法的 CFG 缝成一张大图                    |
| InterSolver                          | WorkList 驱动的过程间求解器，沿 ICFG 边传播 CPFact |
| 四种边（Normal/Call/CallToReturn/Return） | 边转移函数控制数据如何穿越方法边界                    |
| CPFact + Value 三值格                   | 变量→值映射，meetValue 合并                  |

做了修改的部分：

| 修改                                      | 原因                                    |
| --------------------------------------- | ------------------------------------- |
| `ConstantPropagation` 完全重写              | 新增 heapModel，字段/数组访问不再是 NAC           |
| `transferCallEdge` 增加 receiver→%this    | A6 的 PTA 需要处理实例调用                     |
| `transferNonCallNode` 增加 Store→Load 重入队 | heapModel 是全局副状态，InterSolver 感知不到它的变化 |

引入了三个新组件（A4 没有的）：

| 新组件                         | 作用                                   |
| --------------------------- | ------------------------------------ |
| `PointerAnalysisResult pta` | A6 的输出，通过 `getPointsToSet(Var)` 查询别名 |
| `heapModel`                 | 全局堆状态：记录每个对象每个字段的值                   |
| `staticFieldModel`          | 全局静态字段状态                             |

### A6 如何投喂 A7

A7 不运行指针分析，直接获取 A6 已经算好的结果。A6 在 A7 启动前完成，输出一个 `PointerAnalysisResult` 对象，A7 的 ConstantPropagation 持有它的引用：

```java
// StoreField: x.f = val
Set<Obj> pts = pta.getPointsToSet(x);  // ← 调用 A6 的结果
for (Obj obj : pts) {
    heapModel[obj][f] = val;           // 精确写入：知道 x 指向 obj
}
```

没有 A6，`pts` 就是空的，A7 退化为 A4（字段返回 NAC）。**A6 给 A7 提供了"别名信息"——两个变量是否指向同一个对象**。

### A7 需要你实现什么

A7 需要你实现三个文件，它们的分工如下：

| 文件                         | 职责                                                                  | 关键代码                                                 |
| -------------------------- | ------------------------------------------------------------------- | ---------------------------------------------------- |
| `InterSolver`              | 驱动 WorkList 循环；计算 IN（meet 所有入边的 edge fact）；判断 OUT 是否变化来触发后继传播       | `doSolve()` 第87-107行                                 |
| `InterConstantPropagation` | 四种边转移函数（CallEdge/CallToReturnEdge/ReturnEdge）；Store→Load 重入队        | `transferCallEdge` 第168行、`transferNonCallNode` 第129行 |
| `ConstantPropagation`      | 字段/数组读写（handleStoreField/handleLoadField）；通过 pta 查别名；读写全局 heapModel | `transferNode` 第153行、`handleStoreField` 第180行        |

三者的调用关系：`InterSolver` → `InterConstantPropagation.transferNode` → `ConstantPropagation.transferNode` → `handleStoreField`/`handleLoadField`。

### A4 vs A7：具体例子

```java
class Foo {
    int f;
    static void setF(Foo x, int v) { x.f = v; }
    static int getF(Foo x) { return x.f; }
}
void main() {
    Foo a = new Foo();
    setF(a, 123);
    int y = getF(a);  // A4: y = NAC（不知道 x.f 的值）
                       // A7: y = 123（利用 PTA 结果跟踪字段）
}
```

A4 对 `x.f = v` 和 `y = x.f` 的处理是：无法分析其字段语义，直接返回 NAC。A7 利用指针分析的结果，知道 `x` 指向哪个对象，就能精确读写那个对象的字段值。

### 核心设计：堆模型

A7 在 `ConstantPropagation` 中引入了三个全局状态来建模堆：

```java
// ConstantPropagation.java — A7 新增字段
private PointerAnalysisResult pta;                          // PTA 结果
private Map<Obj, Map<Object, Value>> heapModel;             // 实例字段/数组: Obj → (key → Value)
private Map<JField, Value> staticFieldModel;                // 静态字段: JField → Value
private static final Object NAC_INDEX_KEY = new Object();   // 数组弱更新哨兵
```

- **heapModel**：`Obj → (JField/Integer → Value)`，记录每个对象的字段值和数组元素值
- **staticFieldModel**：`JField → Value`，记录静态字段值
- **NAC\_INDEX\_KEY**：当数组索引不确定（NAC）时，写入这个哨兵位置，读取时也要 meet 它

### handleStoreField：写字段

```java
// ConstantPropagation.java — A7 你的代码
private void handleStoreField(StoreField stmt, CPFact in) {
    Var base = stmt.getBase();
    JField field = stmt.getFieldRef().resolve();
    Value val = evaluate(stmt.getRValue(), in);

    if (stmt.isStatic()) {
        // 静态字段：直接更新 staticFieldModel
        Value oldVal = staticFieldModel.getOrDefault(field, Value.getUndef());
        staticFieldModel.put(field, meetValue(oldVal, val));
    } else {
        // 实例字段：遍历 base 的 points-to set
        Set<Obj> pts = getPointsToSet(base);
        if (pts.isEmpty()) {
            // 没有精确对象，使用 instanceFieldSummary
            Value oldVal = instanceFieldSummary.getOrDefault(field, Value.getUndef());
            instanceFieldSummary.put(field, meetValue(oldVal, val));
        } else {
            for (Obj obj : pts) {
                Map<Object, Value> fieldMap = heapModel.computeIfAbsent(
                        obj, k -> new HashMap<>());
                Value oldVal = fieldMap.getOrDefault(field, Value.getUndef());
                fieldMap.put(field, meetValue(oldVal, val));
            }
        }
    }
}
```

**为什么用** **`meetValue`** **而不是直接覆盖？** 因为可能有多个 store 写入同一个位置：

```java
if (c) x.f = 1;
else   x.f = 2;
// x.f 的最终值 = meetValue(1, 2) = NAC
```

### handleLoadField：读字段

```java
// ConstantPropagation.java — A7 你的代码
private Value handleLoadField(LoadField stmt, CPFact in) {
    if (stmt.isStatic()) {
        return staticFieldModel.getOrDefault(
                stmt.getFieldRef().resolve(), Value.getUndef());
    }

    Var base = stmt.getBase();
    JField field = stmt.getFieldRef().resolve();
    Set<Obj> pts = getPointsToSet(base);
    
    Value result = Value.getUndef();
    if (pts.isEmpty()) {
        result = instanceFieldSummary.getOrDefault(field, Value.getUndef());
    } else {
        for (Obj obj : pts) {
            Map<Object, Value> fieldMap = heapModel.get(obj);
            Value val = (fieldMap != null)
                    ? fieldMap.getOrDefault(field, Value.getUndef())
                    : Value.getUndef();
            result = meetValue(result, val);
        }
    }
    return result;
}
```

**别名感知**：如果 `x` 和 `y` 指向同一对象，那么 `x.f = 123; y = z.f` 中 `y` 也能读到 123。因为读和写都是通过 `heapModel.get(obj)` 访问同一个 map。

### handleStoreArray / handleLoadArray：数组访问

数组的难点在于**索引可能不是常量**：

```java
for (int i = 0; i < 10; i++) {
    a[i] = 666;   // i 是循环变量，值不确定 → 索引为 NAC
}
int x = a[3];     // a[3] 应该是 666，但怎么知道？
```

A7 使用**强更新/弱更新**策略：

| 索引值     | 写操作（Store）                                 | 读操作（Load）                                        |
| ------- | ------------------------------------------ | ------------------------------------------------ |
| 常量（如 3） | **强更新**：直接写入 `heapModel[obj][3]`           | 读取 `heapModel[obj][3]`，再 meet `NAC_INDEX_KEY` 的值 |
| NAC     | **弱更新**：写入 `heapModel[obj][NAC_INDEX_KEY]` | 直接返回 NAC                                         |
| UNDEF   | 跳过                                         | 返回 UNDEF                                         |

```java
// handleStoreArray 核心逻辑
Value indexVal = evaluate(stmt.getIndex(), in);
Value val = evaluate(stmt.getRValue(), in);

if (indexVal.isConstant()) {
    // 强更新：写入具体索引
    Integer idx = indexVal.getConstant();
    meetToHeap(obj, idx, val);
} else if (indexVal.isNAC()) {
    // 弱更新：写入哨兵位置
    meetToHeap(obj, NAC_INDEX_KEY, val);
}
// UNDEF: 跳过

// handleLoadArray 核心逻辑
if (indexVal.isConstant()) {
    Integer idx = indexVal.getConstant();
    Value specificVal = getFromHeap(obj, idx);      // 具体索引的值
    Value nacVal = getFromHeap(obj, NAC_INDEX_KEY); // 哨兵值
    result = meetValue(specificVal, nacVal);         // 合并
} else if (indexVal.isNAC()) {
    result = Value.getNAC();
}
```

**为什么读取常量索引时要 meet NAC\_INDEX\_KEY？** 因为之前可能有 NAC 索引的写操作，那个写操作可能修改了任何位置。`a[i] = 666`（i 不确定）等价于"可能修改了 a 的任何元素"。

### 核心修复：Store 触发 Load 重传播

这是 A7 最关键的 bug 修复。问题现象：

```
setF(a, 123);    // StoreField: heapModel[a_obj][f] = 123
int y = getF(a); // LoadField: 从 heapModel 读取 y = 123... 但实际上 y = UNDEF
```

**为什么？** 因为 `StoreField` 只修改了 `heapModel` 这个全局状态，不改当前节点的 `out` fact。`InterSolver` 看到节点的 out 没变，就不会重传播后继的 `LoadField` 节点。

修复方案：**在** **`transferNonCallNode`** **中，当 StoreField/StoreArray 执行后，把 ICFG 上所有 LoadField/LoadArray 节点重新加入 worklist**。

```java
// InterConstantPropagation.java — A7 你的代码
@Override
protected boolean transferNonCallNode(Stmt stmt, CPFact in, CPFact out) {
    boolean changed = cp.transferNode(stmt, in, out);
    // Store 改变了全局堆状态，需要让所有 Load 节点重新计算
    if (stmt instanceof StoreField || stmt instanceof StoreArray) {
        for (Stmt node : icfg) {
            if (node instanceof LoadField || node instanceof LoadArray) {
                solver.addToWorkList(node);
            }
        }
    }
    return changed;
}
```

这个修复方式虽然全面（把所有 Load 节点加入 worklist），但能保证正确性。作业文档也说了"不必过早考虑效率"。

### InterSolver：过程间工作列表求解器

A7 的 `InterSolver` 和 A4 的结构一致，核心区别在于 boundary 节点的处理：

```java
// InterSolver.java — A7 你的代码
private void doSolve() {
    while (!workList.isEmpty()) {
        Node node = workList.poll();
        Fact in;
        if (boundaryNodes.contains(node)) {
            // boundary 节点（如 main 的 entry）: IN fact 固定不变
            in = result.getInFact(node);
        } else {
            // 其他节点: IN = meet 所有入边的 edge fact
            in = analysis.newInitialFact();
            for (var edge : icfg.getInEdgesOf(node)) {
                Fact edgeFact = analysis.transferEdge(
                        edge, result.getOutFact(edge.getSource()));
                analysis.meetInto(edgeFact, in);
            }
            result.setInFact(node, in);
        }
        Fact out = result.getOutFact(node);
        if (analysis.transferNode(node, in, out)) {
            // OUT 变了，把后继加入 worklist
            for (var edge : icfg.getOutEdgesOf(node)) {
                workList.add(edge.getTarget());
            }
        }
    }
}
```

### 边转移函数：和 A4 的区别

A7 的边转移函数和 A4 类似，但增加了 **receiver → %this** 的传播：

```java
// transferCallEdge — A7 你的代码
@Override
protected CPFact transferCallEdge(CallEdge<Stmt> edge, CPFact callSiteOut) {
    CPFact result = new CPFact();
    Invoke invoke = (Invoke) edge.getSource();
    JMethod callee = edge.getCallee();
    
    // 传 receiver → %this
    if (!invoke.isStatic()) {
        Var base = invoke.getInvokeExp().getBase();
        Var thisVar = callee.getIR().getThis();
        result.update(thisVar, callSiteOut.get(base));
    }
    
    // 传参数
    for (int i = 0; i < invokeExp.getArgCount(); i++) {
        Var param = callee.getIR().getParam(i);  // 注意：getParam 不含 this
        Var arg = invokeExp.getArg(i);
        result.update(param, callSiteOut.get(arg));
    }
    return result;
}
```

**注意**：A7 使用 `IR.getThis()` 获取 this 变量，`IR.getParam(i)` 获取第 i 个显式参数（不含 this）。A4 使用的是 `getParam(0)` 代表 this，这是旧版 API。

### A7 完整数据流 — 以 Interprocedural2 为例

```java
class Interprocedural2 {
    int f, g;
    static void setF(Interprocedural2 x, int v) { x.f = v; }
    static int getF(Interprocedural2 x) { return x.f; }
    static void setG(Interprocedural2 x, int v) { x.g = v; }
    static int getG(Interprocedural2 x) { return x.g; }

    public static void main(String[] args) {
        Interprocedural2 x = new Interprocedural2();
        setF(x, 123);
        setG(x, 456);
        int a = getF(x);  // 期望 a = 123
        int b = getG(x);  // 期望 b = 456
    }
}
```

数据流追踪：

```
1. main: x = new Interprocedural2()
   → x 指向 obj_x

2. main: setF(x, 123)
   CallEdge: x → %this, 123 → v
   → setF 内部: %this.f = v → heapModel[obj_x][f] = 123

3. main: setG(x, 456)
   CallEdge: x → %this, 456 → v
   → setG 内部: %this.g = v → heapModel[obj_x][g] = 456

4. main: getF(x)
   CallEdge: x → %this
   → getF 内部: temp$0 = %this.f
     PTA: %this 指向 {obj_x}
     → heapModel[obj_x][f] = 123 → temp$0 = 123
   ReturnEdge: 返回 123 → main 的 a = 123 ✓

5. main: getG(x) 同理 → b = 456 ✓
```

| ------ | -------------- | ----------------------- |
| 字段访问   | NAC（不知道）       | 利用 PTA 精确读写             |
| 数组访问   | NAC            | 强/弱更新 + NAC\_INDEX\_KEY |
| 静态字段   | NAC            | staticFieldModel 直接存取   |
| 多分支写入  | 直接覆盖（错误）       | meetValue 合并            |
| 传播机制   | 只靠 out fact 变化 | Store → 全局 Load 重入队     |
| 依赖     | CHA 调用图        | **2-obj CS-PTA 结果**     |

A7 完成了两条路线的交汇，但从数据流路线回到指针路线——A8 直接基于 A6 的 PTA 做污点分析，不需要 A7 的数据流框架。

## 第九章：A8 — 污点分析

### 关键关系

A8 直接基于 A6 的 PTA，不需要 A7 的数据流框架。原因如下：

- **污点对象就是 PTA 中的特殊 Obj**（TaintObj）。`makeTaint(sourceCall, type)` 和 `new Obj(allocation)` 一样——都是创建一个分配点对象，只是打上了污点类型标签。PTA 的 WorkList 和 propagate 机制会自动把污点对象沿 PFG 边传播，不需要单独的 transfer function。
- **不需要单独的 meet 操作**。PTA 中对象集合只增不减，自然合并；数据流中不同路径的值需要 meetValue 来统一。污点分析只要有一个污点对象流到 sink 就够了，不需要"合并值"。
- **只需在 PTA 传播过程中插入回调**。Source 处创建污点对象并加入结果变量的 pts；Transfer 处检测实参是否含污点对象，含了就传递到返回值；Sink 处遍历参数 pts 检查是否含污点对象。
- **与 A7 的对比**：A7 需要 PTA 结果作为额外输入（`getPointsToSet` 查别名），框架仍是数据流（meet、transfer、ICFG 边）。A8 则是 PTA 的**内置插件**，完全在 PTA 框架内部运行。

### 这是 A8 的内容

A8 需要你实现**基于上下文敏感指针分析的污点分析（Taint Analysis）**：

1. **`Solver`** — 上下文敏感指针分析求解器（A6 基础 + 污点插件集成）
2. **`TaintAnalysiss`** — 污点分析插件

### 什么是污点分析？

污点分析追踪**从"源"（source）流到"汇"（sink）的数据**，是 SAST 最核心的分析之一。

```
source() → 数据被标记为"污点" → 经过各种传播 → 到达 sink() → 形成安全漏洞
```

典型场景：

```java
String password = getSource();      // source: 敏感数据
String query = "SELECT * FROM users WHERE pwd='" + password + "'";
execute(query);                     // sink: SQL 注入风险
```

### A8 的三大角色：Source、Sink、TaintTransfer

**Source（源）**：产生污点数据的方法。调用 source 方法后，返回值被标记为污点。

```yaml
# taint-config.yml
sources:
  - { method: "<SourceSink: java.lang.String source()>", type: "java.lang.String" }
```

**Sink（汇）**：接收污点数据的危险方法。如果 sink 的参数是污点，就报告一个安全漏洞。

```yaml
sinks:
  - { method: "<SourceSink: void sink(java.lang.String)>", index: 0 }
```

`index: 0` 表示检查第一个参数是否为污点。

**TaintTransfer（污点转移）**：某些方法会把输入的污点传递到输出。比如 `String.concat()` 会把参数或 base 的污点转移到返回值。

```yaml
transfers:
  - { method: "<java.lang.String: java.lang.String concat(java.lang.String)>", 
      from: base, to: result, type: "java.lang.String" }
  - { method: "<java.lang.String: java.lang.String concat(java.lang.String)>", 
      from: 0, to: result, type: "java.lang.String" }
```

- `from: base`：污点来自调用者（this）
- `from: 0`：污点来自第 0 个参数
- `to: result`：污点转移到返回值
- `to: base`：污点转移到调用者（this）

### 污点分析的实现架构

A8 把污点分析作为**指针分析的插件**运行——不需要单独的数据流框架，直接在 PTA 的传播过程中处理污点。

```
┌──────────────────────────────────────────┐
│              Solver (PTA)                │
│                                          │
│  propagate() → processCall() → analyze() │
│       │              │          │        │
│       ▼              ▼          ▼        │
│  ┌─────────────────────────────────────┐ │
│  │      TaintAnalysiss (插件)          │ │
│  │                                     │ │
```

### 核心代码实现

#### handleCallSite：处理 Source、Sink、Transfer

```java
// TaintAnalysiss.java — A8 你的代码
public void handleCallSite(CSCallSite csCallSite, JMethod callee, CSMethod csCallee) {
    Invoke invoke = csCallSite.getCallSite();
    Context callContext = csCallSite.getContext();

    // 1. Source: 创建污点对象
    for (Source source : config.getSources()) {
        if (source.method().equals(callee)) {
            Obj taintObj = manager.makeTaint(invoke, source.type());
            CSObj csTaintObj = csManager.getCSObj(emptyContext, taintObj);
            Var resultVar = invoke.getResult();
            if (resultVar != null) {
                CSVar csResultVar = csManager.getCSVar(callContext, resultVar);
                solver.addWorkListEntry(csResultVar,
                        PointsToSetFactory.make(csTaintObj));
            }
        }
    }

    // 2. Sink: 记录这个调用点
    for (Sink sink : config.getSinks()) {
        if (sink.method().equals(callee)) {
            sinkCallSites.computeIfAbsent(invoke, k -> new HashSet<>()).add(sink);
        }
    }

    // 3. TaintTransfer: 注册并立即处理已有的污点
    for (TaintTransfer transfer : config.getTransfers()) {
        if (transfer.method().equals(callee)) {
            callSiteTransfers.computeIfAbsent(invoke, k -> new ArrayList<>())
                    .add(new TransferInfo(callContext, transfer));
            processTransferForExistingTaints(invoke, callContext, transfer);
        }
    }
}
```

**关键**：Source 产生污点对象时，把它加入返回变量的 points-to set。污点对象作为 PTA 中的 Obj，沿 PFG 边传播。

#### handlePointsToSetChange：增量污点转移

```java
// TaintAnalysiss.java — A8 你的代码
public void handlePointsToSetChange(CSVar csVar, PointsToSet delta) {
    Var var = csVar.getVar();
    Context context = csVar.getContext();

    for (Map.Entry<Invoke, List<TransferInfo>> entry : callSiteTransfers.entrySet()) {
        Invoke invoke = entry.getKey();
        for (TransferInfo info : entry.getValue()) {
            if (!info.callContext().equals(context)) continue;
            
            CSVar fromVar = getFromVar(invoke, context, info.transfer());
            if (fromVar == null || !fromVar.equals(csVar)) continue;
            
            // delta 中有新的污点对象 → 执行转移
            for (CSObj csObj : delta) {
                if (manager.isTaint(csObj.getObject())) {
                    doTransfer(invoke, context, info.transfer(), csObj.getObject());
                }
            }
        }
    }
}
```

**为什么需要增量处理？** 因为当 `handleCallSite` 被调用时，fromVar 的 points-to set 可能还是空的。随着 PTA 的传播，新的污点对象会逐渐加入 fromVar 的 pts。每次 pts 增长，`handlePointsToSetChange` 都会被调用，检查是否需要执行新的污点转移。

#### doTransfer：执行污点转移

```java
// TaintAnalysiss.java — A8 你的代码
private void doTransfer(Invoke invoke, Context callContext,
                        TaintTransfer transfer, Obj sourceTaintObj) {
    // 去重：避免对同一个污点对象重复转移
    if (transferred.contains(sourceTaintObj)) return;
    transferred.add(sourceTaintObj);

    // ★ 关键：保持原始 source 调用点，但使用 transfer 后的新类型
    Invoke sourceCall = manager.getSourceCall(sourceTaintObj);
    Obj newTaintObj = manager.makeTaint(sourceCall, transfer.type());
    CSObj csNewTaintObj = csManager.getCSObj(emptyContext, newTaintObj);

    // 根据 to 决定传播目标
    int to = transfer.to();
    if (to == TaintTransfer.RESULT) {
        Var resultVar = invoke.getResult();
        if (resultVar != null) {
            CSVar csResultVar = csManager.getCSVar(callContext, resultVar);
            solver.addWorkListEntry(csResultVar,
                    PointsToSetFactory.make(csNewTaintObj));
        }
    } else if (to == TaintTransfer.BASE) {
        // 转移到 base（this）
        Var base = ((InvokeInstanceExp) invokeExp).getBase();
        CSVar csBaseVar = csManager.getCSVar(callContext, base);
        solver.addWorkListEntry(csBaseVar,
                PointsToSetFactory.make(csNewTaintObj));
    }
}
```

**最关键的 bug 修复**：`makeTaint` 的第一个参数。

- ❌ 错误写法：`manager.makeTaint(invoke, transfer.type())` — 用当前调用点（如 concat）作为 source call
- ✅ 正确写法：`manager.makeTaint(sourceCall, transfer.type())` — 用**原始 source 调用点**

为什么？因为 TaintFlow 的定义是 `(sourceCall, sinkCall, index)`，source call 必须是原始产生污点的地方（如 `SourceSink.source()`），而不是中间的 transfer 调用（如 `concat`）。但 type 可以改变——`StringBuilder.toString()` 会把 StringBuffer 类型的污点转成 String 类型。

#### collectTaintFlows：收集污点流

```java
// TaintAnalysiss.java — A8 你的代码
private Set<TaintFlow> collectTaintFlows() {
    Set<TaintFlow> taintFlows = new TreeSet<>();
    for (Map.Entry<Invoke, Set<Sink>> entry : sinkCallSites.entrySet()) {
        Invoke sinkInvoke = entry.getKey();
        for (Sink sink : entry.getValue()) {
            Var argVar = sinkInvoke.getInvokeExp().getArg(sink.index());
            // 检查所有上下文敏感版本的参数变量
            for (CSVar csVar : csManager.getCSVarsOf(argVar)) {
                PointsToSet pts = csVar.getPointsToSet();
                if (pts == null) continue;
                for (CSObj csObj : pts) {
                    if (manager.isTaint(csObj.getObject())) {
                        Invoke sourceCall = manager.getSourceCall(csObj.getObject());
                        taintFlows.add(new TaintFlow(sourceCall, sinkInvoke, sink.index()));
                    }
                }
            }
        }
    }
    return taintFlows;
}
```

### 完整推算 — 以 SimpleTaint 为例

```java
class SimpleTaint {
    public static void main(String[] args) {
        String s1 = SourceSink.source();     // [0] source
        SourceSink.sink(s1);                 // [1] sink(s1) → 应报污点

        String s2 = SourceSink.source();     // [3] source
        SourceSink.sink(s2);                 // [4] sink(s2) → 应报污点
    }
}
```

追踪过程：

```
1. 处理 main 方法，发现 [0] SourceSink.source() 调用
   → callee 是 source → 创建污点对象 TaintObj(alloc=[0], type=String)
   → 加入 s1 的 pts: s1 → {TaintObj}

2. 处理 [1] SourceSink.sink(s1) 调用
   → callee 是 sink → 记录 sinkCallSites: [1] → {Sink(index=0)}
   → 同时发现 s1 的 pts 包含 TaintObj
   → collectTaintFlows 时检查 s1 → 发现 TaintObj → sourceCall=[0]
   → 报告 TaintFlow(source=[0], sink=[1], index=0) ✓

3. [3], [4] 同理
```

```java
class ArgToResult {
    public static void main(String[] args) {
        String taint = SourceSink.source();  // [0] taint → TaintObj
        String s1 = new String();            // [1] s1 → 普通对象
        String s2 = s1.concat(taint);        // [5] concat: arg0→result transfer
        SourceSink.sink(s2);                 // [7] sink(s2) → 应报污点
    }
}
```

追踪过程：

```
1. [0] taint = SourceSink.source()
   → source: TaintObj(alloc=[0], type=String) 加入 taint 的 pts

2. [5] s2 = s1.concat(taint)
   → callee 是 concat → 匹配两个 transfer:
     - base→result: s1 不是污点 → 不触发
     - arg(0)→result: taint 是污点 → 触发 transfer
       sourceCall = [0] (原始 source)
       创建 TaintObj(alloc=[0], type=String) 加入 s2 的 pts

3. [7] sink(s2)
   → 检查 s2 的 pts → 发现 TaintObj(sourceCall=[0])
   → 报告 TaintFlow(source=[0], sink=[7], index=0) ✓
```

### 污点转移的增量处理 — 为什么不能只在 handleCallSite 里做？

考虑这个场景：

```java
String taint = source();     // taint 产生污点
String s = list.get(0);      // get 是 transfer: base→result
list.add(taint);             // taint 传播到 list 内部
String result = s.concat(""); // s 现在指向污点？
```

如果 transfer 只在 `handleCallSite`（发现调用边时）执行一次，那时 `list` 的 pts 可能还没有污点对象。只有当 `add` 把污点传播到 list 内部后，`get` 的 base 才会包含污点。

**增量处理**保证：每次变量的 pts 增长时，都会重新检查是否需要触发新的污点转移。

### A7 vs A8 总结

| <br /> | A7 (别名感知 CP)                 | A8 (污点分析)                             |
| ------ | ---------------------------- | ------------------------------------- |
| 分析目标   | 变量的**常量值**                   | 变量的**污点状态**                           |
| 传播的数据  | Value (UNDEF/常量/NAC)         | Obj (是否为 TaintObj)                    |
| 基础设施   | PTA 提供别名信息                   | **PTA 直接承载传播**                        |
| 全局状态   | heapModel / staticFieldModel | sinkCallSites / transferredTaints     |
| 触发机制   | Store → Load 重入队             | pts 增长 → 检查 transfer                  |
| 输出     | 每个语句前的常量映射                   | **TaintFlow(source, sink, index) 集合** |

## 第十章：A1→A8 的完整学习路径总结

### 一条 Y 字形的线索串起八个实验

完整的 Y 字形递进结构图见本篇序章。核心要点回顾：

- **数据流路线**（A1→A2→A3→A4）：从过程内到过程间，但堆访问始终返回 NAC
- **指针路线**（A5→A6）：从 CI 到 CS，为后续分析提供别名信息
- **交汇点**（A7）：A4 框架 + A6 结果，精确建模堆访问
- **应用层**（A8）：基于 A6 PTA 的污点分析，直接产出安全漏洞报告

### 八层递进，一句话记

| 作业     | 做什么                    | 核心数据结构                             | 精度        | 输出                    | 所属路线    |
| ------ | ---------------------- | ---------------------------------- | --------- | --------------------- | ------- |
| A1     | 倒着推，找出"将来还要用的变量"       | SetFact（变量集合）                      | —         | 每个语句后的活跃变量集           | 数据流     |
| A2     | 顺着推，找出"值确定的变量"         | CPFact（Var→Value）                  | —         | 每个语句前的常量映射            | 数据流     |
| A3     | 消费 A1+A2               | A1的活跃集 + A2的常量映射                   | —         | 死代码列表                 | 消费      |
| A4     | 打破方法边界，让常量穿越调用         | CHA调用图 + ICFG + CPFact             | 低（字段=NAC） | 跨方法的常量映射              | 数据流     |
| A5     | 算出每个变量指向哪些对象           | PFG + WorkList                     | 低（CI）     | points-to set         | 指针      |
| A6     | 区分不同调用场景下的指向关系         | CS-PFG + ContextSelector           | 高（CS）     | context-sensitive pts | 指针      |
| **A7** | **A4框架 + A6结果，精确常量传播** | **heapModel + ICFG + InterSolver** | **高**     | **常量映射**              | **交汇点** |
| A8     | 利用 PTA 传播污点标签          | TaintObj + PFG + TaintTransfer     | 高         | TaintFlow 集合          | 指针      |

### A5→A6：从"不区分"到"区分"

A5 把所有调用混在一起，A6 用上下文选择器区分不同场景。2-obj 敏感性通过最近 2 个接收者对象的分配点链来区分方法上下文：接收者对象每增加一层嵌套调用，上下文链就增长一位。

### A6→A7/A8：PTA 变成了基础设施

A6 的 PTA 结果在 A7 和 A8 中扮演不同角色：

- **A7**：PTA 提供 `getPointsToSet(Var)` 来回答"这个变量指向哪些对象"，然后 `ConstantPropagation` 用这个信息做别名感知的堆访问
- **A8**：PTA 直接承载污点传播——污点对象就是 PTA 中的特殊对象，沿着 PFG 边传播

### A4→A7 的核心升级

A7 的 `InterSolver` 和 `InterConstantPropagation` 基本沿用了 A4 的框架（四种边、worklist 算法），但 `ConstantPropagation` 完全重写了：不再靠 `evaluate` 返回 null 来保守处理，而是利用 PTA 结果精确建模堆访问。**A7 是 A4 和 A6 的交汇点**。

### A7 的核心突破：Store 触发 Load 重传播

这是 A7 最难调试的 bug。堆状态是**全局副状态**，不在 CPFact 中，InterSolver 感知不到它的变化。修复方法：Store 操作后，把所有 Load 节点重新加入 worklist。

### A8 的核心突破：增量污点转移 + 保持原始 Source

A8 有两个关键设计决策：

1. **增量转移**：不能只在发现调用边时执行一次 transfer，必须在 fromVar 的 pts 增长时重新检查
2. **保持原始 source call**：`makeTaint(sourceCall, type)` 而不是 `makeTaint(invoke, type)`，保证 TaintFlow 的 source 是原始产生污点的地方

### 对 SAST 项目的启示

1. **指针分析是 SAST 的基石**：几乎所有过程间分析都需要 PTA 提供别名信息
2. **上下文敏感性是精度的关键**：CI-PTA 对真实项目精度不够，2-obj 是较好的平衡点
3. **污点分析 = 特殊的指针分析**：污点对象就是 PTA 中的特殊对象，不需要独立框架
4. **插件化架构**：A8 把污点分析作为 PTA 的插件，通过回调接口交互——这是实际 SAST 工具的常见设计
5. **增量传播**：无论是 A5 的 diff-based propagate，还是 A8 的 handlePointsToSetChange，增量思想贯穿始终
6. **数据流 + 指针分析可以结合**：A7 证明了两条路线可以融合，这为更复杂的混合分析打开了思路

### 后续拓展方向

A1-A8 构成了静态分析的基础框架，但实际 SAST 工具还有大量延伸：

- **更复杂的上下文敏感性**：2-call、2-type 等策略在不同场景下的精度-代价权衡
- **On-the-fly 调用图构建**：A5/A6 的 processCall 在传播过程中动态发现调用边，这种方式可以与其他调用图构建算法对比
- **工业级 SAST 工具**：Soot/WALA 等框架提供了更完整的程序分析基础设施，商业工具（如 CodeQL）在此基础上加入了更多规则和优化
- **动态分析与静态分析的结合**：纯静态分析可能产生大量虚假报警，动态验证可以减少误报
- **其他分析方向**：类型状态分析（Typestate Analysis）、信息流控制（IFC）等，都是指针分析和污点分析的延伸应用

这些方向暂不展开，留作后续学习笔记的扩展内容。
