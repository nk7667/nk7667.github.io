## 序章

**CodeQL 扫描结果，本质上就是“静态分析规则命中后的证据展示”。**

如果用我们前面学过的 Tai-e 视角来看：

- **A1-A4** 让我们理解了数据流分析、过程间传播、调用图
- **A5-A7** 让我们理解了指针分析、别名、跨方法对象传播
- **A8** 让我们理解了**污点分析（taint analysis）**

而 **CodeQL** 做的事情可以概括成：

1. 先把程序建模成可查询的数据库
2. 再用 QL 规则描述“我要找什么问题”
3. 对于安全类规则，底层常常会用到：
   - 控制流
   - 数据流
   - 污点传播
   - 调用图
   - 类型层次
   - API 语义建模
4. 最后把“命中的程序点”和“传播证据”展示成结果

所以：

> **CodeQL 结果 = 静态分析结论的可视化输出。**

它不是“程序真的被攻击了”，而是：

> **在静态语义上，程序存在一条满足某种漏洞模式的路径或程序点。**

***

## 第一章：CodeQL 和 Tai-e 里 taint analysis 的关系

### 1.1 最短结论

如果只说一句话：

> **CodeQL 的很多安全查询，本质上就是工业化、可扩展、可建模的污点分析/数据流分析。**

Tai-e A8 做的是教学版污点分析，CodeQL 做的是工程版污点分析。

### 1.2 两者共同点

它们都在回答同一个问题：

> 某个“不可信输入”能不能沿着程序传播，最后到达“危险操作”？

例如 SQL 注入场景：

```java
String id = request.getParameter("id");
String sql = "select * from user where id = " + id;
stmt.executeQuery(sql);
```

这里的分析思路是：

- `request.getParameter` 是 **source**
- `stmt.executeQuery` 是 **sink**
- 字符串拼接、变量赋值、参数传递是 **传播边**
- 如果中间有 `PreparedStatement` 或白名单校验，可能是 **sanitizer / barrier**

这和你在 Tai-e A8 里学的四元组非常接近：

- source
- sink
- transfer
- sanitizer

### 1.3 两者的区别

| 维度   | Tai-e A8      | CodeQL                   |
| ---- | ------------- | ------------------------ |
| 目标   | 教学实验，理解污点传播原理 | 工业级漏洞发现与规则编写             |
| 输入   | IR/程序分析框架     | CodeQL 数据库               |
| 描述方式 | Java 代码里写分析逻辑 | QL 规则声明 source/sink/path |
| 建模范围 | 课程给定 IR、语义有限  | 大量标准库 + 框架 API 建模        |
| 结果展示 | 传播结论/测试输出     | 警告、位置、路径、解释、严重级别         |
| 扩展性  | 适合学习分析原理      | 适合写规则、做安全扫描              |

所以你可以把 CodeQL 理解成：

> **把 Tai-e 中“分析算法 + 漏洞规则 + 结果展示”全部产品化之后的形态。**

## 第二章：CodeQL 扫描结果里，到底在看什么

在 CodeQL 插件或 SARIF 结果里，一条结果通常不是"一个布尔值"，而是一组结构化信息。

### 2.1 最核心的三层含义

可以把一条结果拆成三层：

1. **告警对象是什么** — 告警落在哪个方法/表达式上，它叫什么
2. **为什么命中** — 规则怎么判定的，属于什么类别
3. **证据路径是什么** — 数据从哪来、经过哪些传播点、最终到达哪里

拿我们实际跑过的 HelloWorld.ql 来说，它查的是所有名叫 `getStudent` 的方法。结果会返回每个命中的方法对象、它的全限定名、所在文件和行号。虽然这不是安全规则，但结构是完全一样的——只不过安全规则的"证据路径"更长、更复杂。

下面用两类结果对比说明：

#### 简单查询结果（HelloWorld.ql）

这是我们实际跑过的 HelloWorld.ql，它查所有叫 `getStudent` 的方法。用 `bqrs decode --entities=all` 解码后，真实结果是这样的：

```text
| ID  | 方法名     | URL（文件+行号）                          | 描述信息                                |
+-----+-----------+-----------------------------------------+-----------------------------------------+
| 294 | getStudent | file:///...IndexLogic.java:18:26:18:35 | method: com.l4yn3...logic.IndexLogic.getStudent |
|1680 | getStudent | file:///...IndexDb.java:39:26:39:35    | method: com.l4yn3...db.IndexDb.getStudent       |
```

用 CSV 格式看更简洁：

```csv
"m","col1"
"getStudent","method: com.l4yn3.microserviceseclab.logic.IndexLogic.getStudent"
"getStudent","method: com.l4yn3.microserviceseclab.db.IndexDb.getStudent"
```

可以看到，结果里每个命中项包含了：**方法名、精确到列的源码位置、全限定名的描述字符串**。这就是 CodeQL 最基础的结果形态——命中的实体 + 它的位置 + 你在 `select` 里自定义的输出信息。

这种结果只有一层：**命中的对象 + 它在哪 + 描述信息**。

#### 污点规则结果（比如 SQL 注入）

```
[source]    HttpServletRequest.getParameter()  →  request 对象
[传播]      赋值给 name 变量  →  拼接进 sql 字符串
[sink]      Statement.executeQuery(sql)  →  执行了包含外部输入的 SQL
[消息]      User-controlled data flows into SQL query
[位置]      executeQuery 调用所在行
```

这种结果有三层：**source → 传播链 → sink，外加告警描述和落点位置**。

区别很明显：简单查询只告诉你"找到了什么"，污点规则还告诉你"怎么来的"。

### 2.2 常见结果字段分别对应什么

下面用你学习时最需要理解的方式来解释。

#### 结果标题（message）

这是规则给出的“人类可读结论”。

例如：

- User-controlled data flows into SQL query
- Potential command injection
- This comparison is always true

它对应的是：

> **这条规则认为这里存在什么问题。**

不是传播细节，而是最后的结论标签。

#### 位置（location）

这是 CodeQL 认为**最值得你关注的代码位置**。

通常是：

- sink 所在行
- 或真正触发问题的 API 调用点
- 有时是定义点或结果点

它对应的是：

> **“最终告警落点”**。

也就是插件里你双击后跳过去的那一行。

#### 路径（path / path-problem）

如果这是 path query，结果里通常会有一条路径：

- source
- 中间传播节点
- sink

这对应的是：

> **静态分析找到的证据链。**

这条路径非常像你在 Tai-e 里脑子里手工推导的：

```text
request.getParameter -> id -> sql -> executeQuery
```

只是 CodeQL 把它可视化了。

#### 严重级别（severity）

常见有：

- error
- warning
- recommendation

它表示：

> **规则作者认为这个问题的重要程度。**

这不是分析算法算出来的数学事实，而是规则层面的安全优先级表达。

#### 精度（precision）

常见有：

- very-high
- high
- medium
- low

它表示：

> **这条规则命中后，误报概率大不大。**

比如：

- very-high：通常很可信
- medium：需要人工复核
- low：可能只是可疑模式

这和我们前面学静态分析时经常说的“精度 vs 召回率”是对应的。

#### 查询 ID / 规则 ID

例如：

- `java/sql-injection`
- `java/concatenated-sql-query`

这对应的是：

> **到底是哪条规则命中了。**

以后做规则管理、白名单、基线对比，都靠这个 ID。

#### 标签（tags）

例如：

- security
- external/cwe/cwe-089
- correctness
- maintainability

这表示：

> **这条结果属于什么类别，可能对应什么漏洞标准。**

其中 CWE 标签很重要，因为它把 CodeQL 结果映射到了通用漏洞知识体系。

## 第三章：CodeQL 结果和 taint analysis 是怎么对上的

### 3.1 对应关系总表

| Tai-e / 污点分析概念 | CodeQL 中对应概念                                  |
| -------------- | --------------------------------------------- |
| Source         | source / remote source / untrusted input      |
| Sink           | sink / dangerous API / sensitive call         |
| Transfer       | data flow / taint step / additional flow step |
| Sanitizer      | sanitizer / barrier / guard                   |
| 污点路径           | path query result                             |
| 可达性            | flow exists / path exists                     |
| 调用图传播          | interprocedural data flow                     |
| 别名/对象传播        | points-to + library modeling 的底层支持            |

### 3.2 具体怎么理解一条 path

假设结果里显示：

```text
Source: HttpServletRequest.getParameter
Step: assigned to local variable
Step: concatenated into SQL string
Sink: Statement.executeQuery
```

它的意思不是“运行时一定这么走”。

而是：

> **在静态分析构建的抽象程序图里，存在一条从 source 到 sink 的可行传播链。**

这点非常重要。

静态分析里的 path 是：

- 抽象的
- 保守的
- 近似的

所以它表示的是：

> **程序可能存在这类风险，需要进一步人工判断。**

而不是动态调试意义上的真实执行轨迹。

### 3.3 为什么有些结果没有 path

因为不是所有规则都是污点规则。

CodeQL 规则大致可以分成几类：

1. **模式匹配类**
   - 比如查某个危险 API 是否被调用
2. **局部数据流类**
   - 只关心方法内传播
3. **全局数据流 / 污点类**
   - 会给出 source 到 sink 的路径
4. **结构/类型/继承关系类**
   - 比如某个类实现了什么接口，某种覆盖关系是否存在

所以如果某条结果没有 path，不代表它不重要，只代表：

> **它不是“路径问题（path-problem）”这一类查询。**

## 第四章：CodeQL 的产出内容到底意味着什么

这一章最关键，因为很多初学者会把 CodeQL 结果误解成“已经证实漏洞存在”。

### 4.1 CodeQL 产出不是漏洞利用结果

CodeQL 的结果意味着：

> **程序满足某个静态规则定义的问题模式。**

它不等同于：

- 一定能被攻击成功
- 一定在生产环境可利用
- 一定能被真实输入触发

它给你的是：

- **高价值线索**
- **程序语义证据**
- **人工审计的起点**

### 4.2 结果可能有三种情况

#### 第一种：真实漏洞

路径成立，外部可控，危险 API 真的被触发，没有合理净化。

这就是我们最想找到的情况。

#### 第二种：代码味道 / 风险点

例如字符串拼接 SQL，但实际只拼固定值，或者上游已经保证是白名单。

它未必是可利用漏洞，但确实是：

- 不安全编码习惯
- 脆弱实现方式
- 后续演化风险点

#### 第三种：误报

静态分析的近似导致多报。

比如：

- 某个分支实际上不可达
- 某个 sanitize 函数没被模型识别
- 某个框架封装已经保证安全，但模型不知道

### 4.3 所以 CodeQL 的价值是什么

CodeQL 的价值不是“替代人工判断”，而是：

> **把海量代码中最值得看的位置优先捞出来。**

换句话说：

- 人工审计负责“定罪”
- CodeQL 负责“定位嫌疑人”

这和我们前面学静态分析的目标完全一致：

> **在可接受成本下，尽量覆盖更多潜在风险路径。**

## 第五章：从 Tai-e 学习路径过渡到 CodeQL，应该怎么理解

如果你前面已经学过 Tai-e，那么可以按下面的映射来建立心智模型。

### 5.1 A1-A4 对 CodeQL 的帮助

你已经知道：

- 什么是 CFG / ICFG
- 什么是前向/后向分析
- 什么是 meet 和 transfer
- 什么是过程间传播

这会帮助你理解：

- CodeQL 为什么能跨方法找路径
- 为什么结果里会出现多跳调用链
- 为什么有些值传播要经过参数、返回值、字段

### 5.2 A5-A7 对 CodeQL 的帮助

你已经知道：

- points-to 是怎么传播的
- receiver 变了，callee 也会跟着变
- 别名会让字段传播更复杂

这会帮助你理解：

- CodeQL 为什么能跟踪对象字段
- 为什么框架对象上的数据也能往下传
- 为什么一些 query 的精度依赖库模型和对象建模

### 5.3 A8 对 CodeQL 的帮助最大

A8 让你最容易理解下面这些 CodeQL 术语：

- source
- sink
- sanitizer
- path
- barrier
- taint step

所以从学习顺序来说：

```text
Tai-e A1-A8 学“分析原理”
        ↓
CodeQL 学“规则表达 + 工程化结果输出”
```

这也是为什么学过 Tai-e 后再看 CodeQL，会感觉很多概念其实并不陌生。

## 第六章：实际看结果时，应该怎么读

我建议用下面这个顺序读一条 CodeQL 结果。

### 第一步：先看这是什么规则

先看：

- 规则名
- query id
- tag
- severity
- precision

判断这是：

- 安全问题
- 正确性问题
- 代码味道
- 还是可维护性建议

### 第二步：看最终落点是不是危险操作

也就是 location 那一行。

问自己：

- 这里是不是危险 API？
- 这里为什么会被标红？
- 这里的参数是否来自外部输入？

### 第三步：如果有 path，就顺着 path 看

重点看三件事：

1. source 是谁
2. 中间经过了哪些传播点
3. sink 到底是不是危险调用

### 第四步：判断是否有人工可识别的净化逻辑

例如：

- prepared statement
- encode
- whitelist
- canonicalization
- 参数类型约束

如果有，而 CodeQL 没识别到，这可能是误报或可建模点。

### 第五步：给结果定性

最后把结果分成：

- 真漏洞
- 风险点
- 误报
- 待补模型

这一步就是安全分析师真正创造价值的地方。

## 第七章：一个最小例子，帮你把结果和污点路径对起来

代码：

```java
String name = request.getParameter("name");
String sql = "select * from user where name = '" + name + "'";
stmt.executeQuery(sql);
```

如果 CodeQL 报 SQL 注入，通常你可以这样理解：

### 7.1 在污点分析视角里

- source：`request.getParameter("name")`
- transfer：赋值给 `name`
- transfer：拼接进入 `sql`
- sink：`stmt.executeQuery(sql)`

### 7.2 在 CodeQL 结果视角里

- **message**：用户可控数据流入 SQL 执行点
- **location**：`executeQuery(sql)` 那一行
- **path**：从 `getParameter` 到 `executeQuery`
- **severity**：高
- **precision**：高或中高
- **tag**：security / cwe-089

### 7.3 它真正意味着什么

它的意思不是：

> “攻击已经发生。”

而是：

> “静态分析证明，这段代码具有 SQL 注入模式，建议人工确认为漏洞。”

***

## 第八章：学习 CodeQL 时最值得抓住的三件事

### 8.1 第一件事：不要只看“有没有报”

更重要的是看：

- 为什么报
- 报在哪
- 路径怎么走
- 规则在找什么模式

### 8.2 第二件事：把每条结果翻译成分析语言

例如看到结果后，脑子里自动翻译成：

- source 是什么
- sink 是什么
- 中间有哪些 transfer
- sanitizer 是否缺失

这样你就不是在“点插件”，而是在真正理解静态分析。

### 8.3 第三件事：把 CodeQL 当成“可编程的静态分析”

CodeQL 最强的地方不只是现成规则，而是：

> **你可以自己定义什么是 source、什么是 sink、什么路径算风险。**

这也是它和普通“黑盒扫描器”最大的不同。

## 第九章：实测对比——CodeQL 到底能扫出多少漏洞

前面都是理论，这章拿真实靶场项目跑一遍。

### 9.1 靶场项目

用我自己的 owasp-lab 项目，一个包含 7 类 30 个已知漏洞的 Java Spring Boot 靶场。

实际漏洞分布：

| 类型 | 数量 | 典型场景 |
|---|---|---|
| SQLi | 7 | MyBatis XML `${}` 拼接 |
| RCE | 4+5(WEAK) | `ProcessBuilder` + 弱过滤绕过 |
| XSS | 8+9(WEAK) | 反射型/存储型/DOM型/CSP绕过 |
| SSRF | 3+4(WEAK) | `URL.openConnection` + 弱过滤绕过 |
| XXE | 1 | `DocumentBuilder.parse` 默认配置 |
| CSRF | 2 | GET改密 + XSS链绕token |
| JSONP | 2 | callback未校验 |==---

### 9.2 扫描方式

用 CodeQL CLI 跑自定义的 FindDangerousCalls.ql，覆盖了常见的危险 API 调用：

```ql
from MethodCall call
where
  call.getCallee().hasName("executeQuery")
  or call.getCallee().hasName("executeUpdate")
  or call.getCallee().getQualifiedName() = "java.lang.Runtime.exec"
  or call.getCallee().getQualifiedName() = "java.lang.ProcessBuilder.start"
  or call.getCallee().getQualifiedName() = "java.net.URL.openConnection"
  or call.getCallee().hasName("parse")
  or call.getCallee().hasName("createNativeQuery")
  // ... 更多模式
select call, "Dangerous call: " + call.getCallee().getQualifiedName()
```

实际执行的命令和结果解码流程：

```powershell
# 第一步：建数据库（Maven 编译 + CodeQL 提取）
codeql database create owasp-lab-db `
  --language=java `
  --source-root=owasp-lab `
  --overwrite --ram 2048

# 第二步：跑自定义查询
codeql query run FindDangerousCalls.ql `
  --database=owasp-lab-db `
  --search-path=java-queries `
  --output=owasp-result.bqrs

# 第三步：解码结果看实际命中
codeql bqrs decode owasp-result.bqrs --format=csv
```

解码后的原始命中数据（共 16 处调用，按文件+行号+API 列出）：


### 9.3 对比结果

| 漏洞类型 | 实际数量 | CodeQL命中 | 漏报原因 |
|---|---|---|---|
| RCE | 9 | 4 | WEAK模式5处漏报（需理解过滤逻辑能否绕过） |
| SSRF | 7 | 2 | WEAK模式4处漏报 |
| XXE | 1 | 2 | 基本命中（2次parse调用对1个XXE漏洞） |
| XSS | 17 | 4 | 严重漏报：反射型/存储型/DOM型/CSP绕过全是模板拼接，不是API调用 |
| SQLi | 7 | 2 | 严重漏报：7个全在MyBatis XML `${}`，不在Java源码层 |
| CSRF | 2 | 0 | 全漏：是"缺少防护"而非source→sink |
| JSONP | 2 | 0 | 全漏：callback未校验不是数据流问题 |

**漏报率约 67%-73%。**

### 9.4 漏报的根本原因

归纳下来就四类：

**1. 看不见非源码层**

7个SQLi全在 `SqliMapper.xml` 的 `${username}` 拼接里，CodeQL 只建了 Java 源码的数据库，XML 层它根本不解析。

**2. 不是所有漏洞都是 source→sink**

CSRF 的问题是"没有 token 校验"，JSONP 的问题是"callback 参数未做白名单"。这些不是数据流从外部流入危险API，而是"缺少防护措施"，污点分析天然无法检测。

**3. XSS 的 sink 不在 Java API 层**

XSS 的危险点在 `innerHTML = raw`、模板渲染、`document.write()` 这些前端操作。Java 后端代码只是把字符串返回给前端，CodeQL 的 Java 模型看不到前端渲染上下文。

**4. 弱过滤绕过需要语义推理**

WEAK 模式的漏洞不是"没过滤"，而是"过滤了但能绕过"。判断过滤是否有效需要语义推理（比如 `replace("\n", "")` 能不能防住 `;command`），简单的模式匹配做不到。

### 9.5 学术语料验证

这不是我们一个人的发现。威斯康星大学 2026 年的纵向研究分析了 3993 个 CVE，CodeQL 只识别出 171 个（4.3%）。IEEE 2025 年的基准测试中 CodeQL F1=0.546，远低于 LLM 的 0.797。

> **CodeQL 报出来的大概率是真漏洞，但它漏掉的远比报出来的多。**

## 第十章：SAST 的缺陷怎么解决

SAST 的缺陷不是某个工具的问题，而是这条技术路线本身的天花板。所有主流 SAST 都有同样的困境，只是各有各的"瞎法"。

### 10.1 四个根本缺陷

**缺陷一：只看源码，看不见非源码层**

MyBatis XML mapper、HTML 模板、YAML 配置、SQL 脚本、前端 JS——这些东西里有大量注入点，但 SAST 只分析编译型源码。

**缺陷二：只能追数据流，追不了"缺少防护"**

污点分析能回答"数据能不能从 A 流到 B"，但回答不了"这里该不该有 token 校验"。CSRF、CORS、JSONP、不安全的直接对象引用——这些是"缺东西"的问题，不是"多东西"的问题。

**缺陷三：静态分析追不到动态行为**

反射调用、动态代理、运行时类加载、依赖注入框架——代码写出来的时候，调用关系还不确定，静态分析只能做保守估计。

**缺陷四：通用规则覆盖不了项目特异逻辑**

官方规则库只能覆盖已知的 CWE 模式。业务逻辑漏洞（比如越权、价格篡改、条件竞争）必须针对项目写定制规则，而写规则本身就需要安全专家。

### 10.2 解决方案：不是换工具，是组合工具

没有单一工具能解决所有问题。业界实际在做的是**分层互补**：

```
┌─────────────────────────────────────────┐
│            安全审计全景覆盖              │
├─────────────────────────────────────────┤
│                                         │
│  SAST（CodeQL/Semgrep）                 │
│  ├─ 职责：源码层深度数据流/污点追踪     │
│  ├─ 强项：精度高、误报低                │
│  └─ 盲区：非源码层、缺少防护类、动态行为 │
│                                         │
│  SCA（依赖扫描）                        │
│  ├─ 职责：第三方依赖已知漏洞            │
│  ├─ 强项：CVE匹配、许可证合规           │
│  └─ 盲区：不知道漏洞在项目中是否可利用   │
│                                         │
│  DAST（动态扫描）                       │
│  ├─ 职责：运行时黑盒渗透测试            │
│  ├─ 强项：真实漏洞验证、覆盖CSRF等      │
│  └─ 盲区：需要部署环境、覆盖率受限于爬虫 │
│                                         │
│  IAST（交互式测试）                     │
│  ├─ 职责：运行时插桩污点追踪            │
│  ├─ 强项：精度极高、误报极低            │
│  └─ 盲区：需要运行、需要测试覆盖        │
│                                         │
│  人工审计                               │
│  ├─ 职责：SAST看不见的地方              │
│  ├─ 强项：理解业务逻辑、非代码层        │
│  └─ 盲区：慢、贵、难以规模化            │
│                                         │
└─────────────────────────────────────────┘
```

用你的靶场项目举例：

| 漏洞 | 最佳检测工具 | 原因 |
|---|---|---|
| MyBatis XML SQLi | **Semgrep** 或人工审计 | Semgrep 能扫 XML 文件，CodeQL 不行 |
| XSS（模板拼接） | **DAST** 或人工审计 | 运行时检测比静态更准 |
| CSRF | **DAST** 或人工审计 | 需要"缺少防护"的检测能力 |
| RCE（ProcessBuilder） | **CodeQL** | 这正是 CodeQL 污点追踪的强项 |
| XXE | **CodeQL** | source→sink 数据流清晰 |
| SSRF | **CodeQL + DAST** | CodeQL 找可疑点，DAST 验证可利用性 |

### 10.3 LLM + SAST：当前最热的方向

前面提到的学术研究表明，LLM 和 SAST 组合效果显著提升：

| 方案 | 检出率 | 来源 |
|---|---|---|
| CodeQL 官方规则 | 24/62 (38.7%) | QLPro 论文 2025 |
| CodeQL + Claude (QLPro) | 41/62 (66.1%) | QLPro 论文 2025 |
| CodeQL 官方规则 | 171/3993 CVE (4.3%) | 威斯康星纵向研究 2026 |
| QLCoder 自动生成规则 | 53.4% CVE 检出 | QLCoder 论文 2026 |

LLM 解决的是 SAST 的第四个缺陷——"写规则难"。它不能替代 SAST 引擎本身，但可以：

1. **自动生成 CodeQL 规则**：给一个 CVE 描述，自动写出检测查询
2. **解释结果**：把技术术语翻译成开发者能理解的语言
3. **补充模型**：为框架特有 API 补充 source/sink/barrier 定义
4. **分类匹配**：从项目中提取污点规格，匹配到对应 CWE

### 10.4 实际落地的最小方案

如果你是一个安全团队，不需要等 LLM 成熟，现在就能做的：

**第一步：CodeQL + Semgrep 并行**

```text
CodeQL  → 深度数据流（精度高，速度慢）
Semgrep → 快速模式匹配（速度快，覆盖广，能扫 XML/YAML/JS）
```

两者互补：CodeQL 保证深度，Semgrep 保证广度。

**第二步：关键漏洞用 DAST/IAST 验证**

```text
CodeQL/Semgrep 发现可疑点 → DAST 跑一遍确认可利用性
```

这样误报问题也解决了——静态分析找嫌疑人，动态分析定罪。

**第三步：非代码层靠人工审计**

```text
MyBatis XML → 搜 ${}
HTML 模板   → 搜 v-html / innerHTML / dangerouslySetInnerHTML
配置文件     → 搜 hardcoded credentials / debug=true
```

这些搜索成本很低，但覆盖的是 SAST 完全看不见的盲区。

**第四步：把经验沉淀成规则**

每次人工审计发现的新模式，写成 CodeQL 或 Semgrep 规则。规则库会越滚越大，检出率会越来越高。

### 10.5

> **SAST 的缺陷不是靠换工具解决的，而是靠"工具组合 + 人工补位 + 经验沉淀"来弥补的。CodeQL 做深度，Semgrep 做广度，DAST 做验证，人工做兜底。**
