> 这篇规划不是继续讲某一个具体实验，而是把前面已经学过的 Tai-e、CodeQL、靶场验证，以及接下来准备看的 SAST-demo 串起来，形成一条更清晰的白盒学习路线。目标不是一上来重写一个完整 SAST 引擎，而是先把“原理 → 最小实现 → 规则扩展 → 工程落地”这四层打通。

***

## 第一章：先回答一个问题——为什么还要看 SAST-demo

前面我们已经学过：

- Tai-e 中的数据流分析、调用图、过程间分析、污点分析
- CodeQL 的查询写法、数据库、规则、结果解释
- 用真实靶场去验证 CodeQL 的能力边界和漏报原因

按理说，已经不是“完全没接触过 SAST”的状态了。

但现在还差一层关键东西：

> **如何把这些分析概念真正组织成一个“能运行、能展示、能产出漏洞结果”的最小系统。**

Tai-e 更偏“分析原理与框架训练”，CodeQL 更偏“成熟引擎与规则使用”，而 SAST-demo 更像一个中间层：

- 它没有 CodeQL 那么重
- 也没有 Tai-e 那么偏课程实验
- 它会把 AST、IR、CFG、污点路径这些东西直接展示出来

所以它适合拿来做“从理论走向实现”的桥梁。

我的判断是：

> **SAST-demo 不是最终教材，但足够作为“最小实现教材”。**

它不一定能覆盖工业级 SAST 的所有难点，但非常适合用来补上这样一个空档：

- 我们知道 source / sink / path 是什么
- 但还不够直观地理解这些东西是怎么被程序组织起来的

***

## 第二章：把当前学习状态整理清楚

### 2.1 已经掌握的部分

结合前面的学习，当前已经有这些基础：

1. **程序分析主线已经建立**
   - 知道什么是 CFG、数据流分析、污点分析、调用图、过程间传播
2. **CodeQL 基础已经建立**
   - 能写简单查询
   - 能理解 source / sink / path query
   - 能用真实数据库跑查询
3. **已经开始接触真实工程问题**
   - 知道 MyBatis XML 会断链
   - 知道框架建模会影响命中率
   - 知道 SAST 会漏报/误报
4. **已经知道“规则”并不是全部**
   - 漏报很多时候不是规则字符串没写对，而是前端解析、框架映射、数据流边没有补上

### 2.2 Tai-e 已经明显涉及到的部分

前面的 Tai-e 学习里，其实已经明显接触过这些内容：

- 方法 / 类 / 字段解析
- 方法签名
- interface / implementation
- 调用表达式
- 继承关系
- 符号解析相关能力
- AST / IR / CFG / 数据流这些上层依赖

也就是说，我们并不是完全没有 Java 前端和静态分析基础，只是这些知识目前更多停留在“知道它重要”和“能看懂部分效果”的层面。

### 2.3 Tai-e 里通常只是间接接触的部分

这些内容前面不是完全没碰到，但通常只是间接接触，还没有系统化：

- 词法分析
- 语法分析
- import / package 解析
- 泛型细节
- 注解处理

这些点平时不一定直接冒出来，但一旦开始看分析器实现、看调用解析、看框架建模，就会频繁碰到。

### 2.4 Tai-e 里通常还不够的部分

真正缺口更明显的地方，主要在工程化和框架层：

- 常见 Java 安全框架习惯
- Spring / MyBatis / Hibernate / JPA 等框架建模
- 真实项目中的 source / sink / sanitizer 工程化识别

所以当前的短板主要不在“概念没听过”，而在下面几件事还不够扎实：

1. **前端知识知道作用，但没有真正落到实现**
   - 词法分析
   - 语法分析
   - AST 构建
   - 符号解析
2. **知道污点分析原理，但没有自己做过一个最小版本**
3. **知道框架建模重要，但还没有系统整理“哪些前端信息支撑哪些漏洞规则”**
4. **对一个 SAST 工具内部模块如何协同，仍然缺少整体感**

所以接下来不该再继续横向发散，而是要进入一个更聚焦的阶段：

> **围绕 SAST-demo 做“结构化拆解学习”。**

***

## 第三章：这些基础要不要系统补

结论是：**要补，但不是按“从零造编译器”的方式补。**

目标不是成为编译器工程师，而是补到足以支撑下面这些事情的程度：

- 看懂现成 parser 或轻量分析器
- 理解 AST / 符号 / 调用解析如何影响污点分析
- 为漏洞规则补 source / sink / path 所需的前端信息
- 支撑后面的 CodeQL、规则扩展、自研轻量分析器学习

如果这些基础不补，后面就会一直卡在类似问题上：

- 为什么 call graph 会有误差
- 为什么 source / sink 会断链
- 为什么 MyBatis XML 很难连起来
- 为什么反射、泛型、接口、多态会让静态分析变难
- 为什么 CodeQL / Fortify / 自研引擎都要做前端和建模

但也不用一开始就学到“编译器实现级别”。

不需要一上来掌握：

- LR(1) 构造
- 复杂 parser generator
- Java 语法规范全文
- 完整 type checker

更合适的目标是：

- 知道这些东西在 SAST 里起什么作用
- 能看懂现成分析器的实现思路
- 能为规则开发和最小引擎开发补足必要前端认知

***

## 第四章：这轮规划的核心目标

这轮学习不追求“立刻造一个完整 SAST”，而是完成下面四件事：

### 目标一：把 SAST-demo 当成“最小实现样本”看懂

重点不是记住仓库结构，而是回答清楚：

- 源码是怎么被读取的
- 程序表示是怎么落下来的
- CFG 是怎么生成的
- 污点分析是怎么组织的
- 最终漏洞结果是怎么产出的

### 目标二：把 Tai-e 学过的概念对上实现模块

要做到：

- 看到 CFG 代码时，能联想到 Tai-e 里的控制流分析
- 看到污点传播时，能联想到 source / sink / transfer / reachability
- 看到 IR 时，知道它为什么是 AST 和数据流分析之间的桥梁

### 目标三：补上“面向 SAST 的前端认知”

这里不是去硬啃完整编译原理，而是聚焦下面这些与 SAST 直接相关的内容：

- 词法分析在做什么
- 语法分析在做什么
- AST 为何重要
- 符号解析为何影响调用图与污点传播
- 方法 / 类 / 字段解析如何支撑规则命中
- Java 中的 package / import / 方法签名 / 继承 / 多态 / 注解 为什么会影响安全分析

### 目标四：形成后续自己的实现路线图

最终要回答：

- 我们以后如果自己做一个最小 SAST，要先做哪几层
- 哪些层可以先简化
- 哪些能力必须先具备
- CodeQL / Tai-e / SAST-demo 各自能提供什么养分

***

## 第五章：这些基础怎么补最合适

这里不按“编译原理大全”路线走，而按“静态分析需要什么就补什么”的路线走。

### 5.1 第一层：概念层

先做到能把这条主线讲清楚：

> **源码 → AST → IR / CFG → Data Flow → 漏洞结果**

这一层重点不是自己实现，而是明确每一层在做什么、解决什么问题。

### 5.2 第二层：静态分析映射层

这一层要把前端知识和 SAST 直接对起来。

比如：

- 为什么方法签名会影响调用图精度
- 为什么继承 / interface 会影响 source / sink 命中
- 为什么注解会影响 Spring 入口识别
- 为什么 package / import 会影响跨文件调用解析
- 为什么 XML namespace 会影响 MyBatis 映射

目标不是“会编译原理考试”，而是：

> **能把 Java 前端知识转化成安全分析里的 source / sink / path 建模能力。**

### 5.3 第三层：最小实现层

不能只停在概念层，后面还要做最小练习。

但这里不追求完整 Java 解析器，而是做足够小的练习：

- 支持变量赋值
- 支持方法调用
- 支持 if
- 支持 return
- 能输出简化 AST / CFG / Use-Def

这样学的价值在于：

- 会真正理解 parser 为什么难
- 会明白轻量分析器为什么大量依赖启发式
- 会更容易理解 CodeQL 为什么需要 extractor / database / model

## 第六章：学习顺序怎么安排

### 第一阶段：先把 SAST-demo 当“白盒解剖对象”看

这一阶段不要着急改代码，也不要急着评价它好不好。

先只做四件事：

1. 看它的入口
2. 看它的数据结构
3. 看它如何生成 IR / CFG
4. 看它如何做污点分析

这一步的目标是：

> **先把系统的大框架看清楚。**

优先关注的问题：

- 输入是什么
- 中间表示是什么
- 分析在哪一层做
- 结果怎么回传到前端

### 第二阶段：把每个模块和 Tai-e 对齐

这一步开始建立“概念映射”：

| SAST-demo 模块 | 在 Tai-e / 静态分析里对应什么 |
| --- | --- |
| AST / Token 流 | 前端表示 |
| IR | 中间表示 |
| BasicBlock / CFG | 控制流分析基础 |
| Use-Def | 数据依赖 |
| Source/Sink 规则 | 污点分析配置 |
| Reachability 校验 | 路径可达性验证 |

这一阶段的目标不是记模块名，而是做到：

> **看到实现，就知道它对应哪类分析思想。**

### 第三阶段：把缺失的前端基础按需补上

这一步不从“编译原理大全”开始，而是按 SAST-demo 里真实用到的东西补。

比如：

- 为什么它要先切 token / 扫行
- 为什么它要识别方法调用与赋值
- 为什么 if / while 会影响 CFG
- 为什么变量定义和使用关系会影响污点传播

这里的原则是：

> **用实现倒逼基础，而不是先把所有基础学完再回头看实现。**

### 第四阶段：开始总结“如果我们自己做，要怎么做”

这一阶段再回头总结：

1. 最小可行版需要哪些模块
2. 哪些地方可以先启发式实现
3. 哪些地方以后必须升级成更正规前端
4. 哪些漏洞规则最适合先做

建议先选：

- SQL 注入
- SSRF
- RCE

因为这几类规则的 source / sink 比较清晰，利于建立第一版引擎直觉。

## 第七章：SAST-demo 是否够用

结论是：

> **够用，但只够做“第一阶段教材”，不够当最终教材。**

### 7.1 为什么说它够用

因为它能帮我们建立以下直觉：

1. 一个 SAST 系统最少有哪些层
2. AST / IR / CFG / 污点分析是怎么串起来的
3. 分析结果如何映射回代码位置
4. 漏洞结果不只是一个布尔值，而是一条带证据的路径

这对当前阶段非常重要。

### 7.2 为什么说它又不够用

因为它毕竟不是工业级引擎，很多地方会做简化：

- Java 前端是轻量解析，不是完整编译前端
- 规则和数据流处理更偏教学 / 演示
- 框架建模能力有限
- 面向真实大型项目的精度、覆盖和性能不是重点

所以它更适合回答：

> “一个最小 SAST 是怎么长出来的？”

而不适合直接回答：

> “一个工业级 SAST 为什么能覆盖 Spring / MyBatis / Hibernate / 反射 / 大型项目？”

这部分还是要靠后面的 CodeQL、框架建模、自定义规则扩展来补。

***

## 第八章：接下来具体看什么

后面我会按这个顺序带着继续看 SAST-demo：

### 8.1 第一步：看入口与整体链路

先看：

- 服务入口
- 扫描入口
- 一次扫描从哪里开始
- 结果怎么返回

重点回答：

- 系统输入是什么
- 扫描在哪触发
- 输出结果长什么样

### 8.2 第二步：看核心数据结构

重点看：

- IR 指令结构
- BasicBlock
- FunctionIR / ProgramIR
- 漏洞结果结构

重点回答：

- 系统如何表示程序
- 为什么需要这些表示

### 8.3 第三步：看语言前端

优先看 Java 部分。

重点看：

- 它怎么识别 if / while / call / assignment
- 怎么把代码变成 IR
- 怎么划 BasicBlock

重点回答：

- 前端到底做了哪些事
- 哪些是简化实现
- 哪些是以后必须升级的地方

### 8.4 第四步：看污点分析引擎

重点看：

- source / sink 如何定义
- 污点如何传播
- reachability 怎么验证
- 为什么会有误报 / 漏报

重点回答：

- 它是“模式匹配”还是“数据流分析”
- 它的路径证据是怎么形成的

### 8.5 第五步：回头总结“我们能抄什么、不能抄什么”

最终会整理成三类：

1. **可以直接借鉴的**
   - 模块拆分
   - 最小 IR / CFG 设计
   - 规则与结果展示方式
2. **可以先简化借鉴的**
   - 轻量 Java 前端
   - 简化污点传播
3. **不能照抄的**
   - 工业级框架建模
   - 大项目精度与性能问题
   - 大规模语义解析能力

## 第九章：这一轮学习的产出形式

这一轮不只看代码，还要沉淀成文档。

建议后续形成这样几篇内容：

1. **学习篇(四)**：当前这篇规划
2. **学习篇(五)**：SAST-demo 整体架构拆解
3. **学习篇(六)**：SAST-demo 中的 IR / CFG / 污点分析实现
4. **学习篇(七)**：从 SAST-demo 回看 Tai-e 与 CodeQL
5. **学习篇(八)**：如果自己做一个最小 SAST，该先实现什么

这样目录会更清楚，也能把“原理、工具、实现、规划”分开。

***

## 第十章：一句话收尾

当前最合适的路线，不是继续无边界发散，也不是一头扎进工业级自研引擎，而是：

> **先把 SAST-demo 当成“最小实现样本”看透，把 Tai-e 里的分析思想和一个真实可运行的小系统对齐，再决定后面是做规则增强，还是走自研轻量引擎。**

这一步做扎实了，后面无论继续学 CodeQL、补框架建模，还是自己做最小 SAST，都不会再悬空。
