> fastjson-1.2.47的类缓存绕过为何能打破AutoType限制？本质是 `java.lang.Class` 反序列化触发 `Class.forName(...)`，走类加载流程。

## 一、前置基础：Java类加载机制

### 1.1 类的生命周期与加载时机

Java类从加载到内存到卸载出内存，经历七个阶段：**加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载**。其中验证、准备、解析合称为“连接”。

JVM 规范规定，只有发生以下**主动引用**时，会触发类加载：

- 使用 `new` 关键字实例化对象
- 读取或设置类的静态变量`getstatic`、`putstatic`
-调用类的静态方法`invokestatic` 
- 使用 java.lang.reflect 包对类进行反射调用（`Class.forName()` 默认会初始化）
- 初始化子类时父类未初始化，则先触发父类初始化

**被动引用**（不触发类加载）：

- 通过子类引用父类静态变量
- 通过数组定义来引用类（如 `Demo[] demos = new Demo[10]`，不会触发 Demo 的初始化）
- 引用`static final` 常量在编译期存入常量池


### 1.2关键api： Class.forName() + ClassLoader.loadClass()

| 方法                                 | 是否执行静态初始化块 | 典型用途                         |
| ---------------------------------- | ---------- | ---------------------------- |
| `Class.forName(className)`         | **是**      | 执行`static{}`中的恶意代码 |
| `ClassLoader.loadClass(className)` | 否          | 仅加载，不初始化，配合字节码注入场景           |
很多反序列化利用链（如 Fastjson 的 Class.forName 调用）就是通过控制传入的类名，触发恶意静态代码块来完成攻击。

## 二、 类加载器层次结构与双亲委派模型

###2.1 标准三层结构
```
Bootstrap ClassLoader (JVM层，C++实现，加载rt.jar)
        ↑
Extension ClassLoader (加载lib/ext目录)
        ↑
Application ClassLoader (加载ClassPath)
        ↑
Custom ClassLoader (用户自定义)
```

- `Bootstrap ClassLoader`加载的类（如`java.io.File`），`getClassLoader()`返回`null`。
- 自定义类加载器通常继承 `java.lang.ClassLoader`，重写 `findClass()` 方法。

### 2.2 双亲委派模型的工作流程（源码级）

双亲委派模型的核心逻辑定义在 `java.lang.ClassLoader.loadClass()` 方法中：

```java
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    synchronized (getClassLoadingLock(name)) {
        // 第一步：检查该类是否已被当前类加载器加载过
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                // 第二步：如果父加载器存在，则委派给父加载器
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    // 父加载器为 null，表示 Bootstrap ClassLoader，去引导路径查找
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // 父加载器加载失败，忽略异常，继续往下
            }

            if (c == null) {
                // 第三步：父加载器找不到，才由自己加载（调用 findClass）
                c = findClass(name);
            }
        }
        if (resolve) {
            // 第四步：如果需要，执行解析（链接阶段的一部分）
            resolveClass(c);
        }
        return c;
    }
}
```

**流程总结**：自底向上检查是否已加载 → 自顶向下尝试委派加载 → 最终由自己加载。

这种机制保证了核心类库（如 `java.lang.Object`）的**唯一性**和**安全性**，防止用户自定义的类冒充核心类。

### 2.3 【关键补充】线程上下文类加载器（TCCL）：双亲委派的“例外”

线程上下文类加载器（Thread Context ClassLoader，TCCL）是理解 Java 很多高级特性（如 JNDI、JDBC）和安全漏洞（如 JNDI 注入）的关键。

- **是什么**：每个线程都有一个 `contextClassLoader` 属性，可通过 `Thread.currentThread().getContextClassLoader()` 获取，也可通过 `setContextClassLoader()` 设置。
- **它解决了什么问题**：Java 的核心库（由 Bootstrap/Extension ClassLoader 加载）经常需要**加载应用提供的 SPI 实现类**（如数据库驱动、JNDI 服务提供者）。根据双亲委派模型，核心类无法访问应用类（AppClassLoader）的路径。TCCL 提供了一种“反向”机制——允许**父类加载器（核心类）请求子类加载器（应用类）** 去加载类。
- **安全意义**：在很多漏洞利用（尤其是 JNDI 注入）中，攻击者会控制 JNDI 的 `InitialContextFactory` 等 SPI 类名，而 JNDI 服务会使用 TCCL 来加载这个工厂类。如果 TCCL 指向的是一个恶意的或可被攻击者控制的类加载器，就会导致任意类加载，进而实现远程代码执行（RCE）。

## 三、核心阶段拆解：连接与初始化

### 3.1 验证（Verification）

确保被加载类的字节码符合 JVM 规范，防止恶意或损坏的 Class 文件危害 JVM。分为四个子阶段：

| 验证阶段 | 检查内容 | 失败示例 |
| :--- | :--- | :--- |
| 文件格式验证 | 魔数 `0xCAFEBABE`、主次版本号、常量池类型 | 篡改 Class 文件头 |
| 元数据验证 | 父类继承关系、`final` 类是否有子类 | 类实现了 `final` 接口 |
| 字节码验证 | 操作数栈类型、数据流、跳转指令合法性 | `int` 值赋值给 `Object` 变量 |
| 符号引用验证 | 被引用的类/字段/方法是否存在、权限是否正确 | 调用不存在的方法 |

**安全意义**：某些攻击会尝试向 JVM 提交畸形字节码以绕过安全检查，但现代 JVM 的验证器非常严格。

### 3.2 准备（Preparation）

为 **类变量（static）** 分配内存，并设置默认零值。

```java
public static int value = 123;
// 准备阶段后，value = 0，而不是 123
// 真正的赋值 123 在初始化阶段执行
```

**特殊情况**：
- `static final` 常量：在准备阶段就会赋值为代码中指定的值（编译期常量）。
- 实例变量：不在此阶段分配，而是在对象实例化时在堆中分配。

### 3.3 解析（Resolution）

将常量池中的 **符号引用** 替换为 **直接引用**（内存地址）。

| 引用类型 | 符号引用示例 | 直接引用示例 |
| :--- | :--- | :--- |
| 类/接口 | `"java/lang/String"` | 方法区中该类的内存地址 |
| 字段 | `"value"` | 字段在类实例中的偏移量 |
| 方法 | `"println"` | 方法入口的指针（或 vtable 索引） |

```java
// 编译期：符号引用 "java/lang/System.out:println"
// 运行时解析后：直接指向 JVM 内存中的实际方法地址
System.out.println("Hello");
```

### 3.4 初始化（Initialization）

执行 **类构造器 `<clinit>()`**，完成静态变量赋值和静态代码块的执行。

```java
public class Demo {
    static int a = 10;           // 赋值在此阶段执行
    static {                     // 静态代码块也在此阶段执行
        System.out.println("static block");
    }
}
```

`<clinit>()` 方法由编译器自动收集类中所有静态变量赋值和静态代码块合并而成，并且**线程安全**。

## 四、类加载机制的特例

### 4.1 【补充】数组类的加载

数组类型（如 `String[]`、`int[][]`）在 Java 中是一种特殊的类型，它们：

- **没有对应的 `.class` 文件**，不是通过类加载器“加载”的，而是由 JVM 在运行时**直接创建**。
- 数组类的类名以 `[` 开头，如 `[Ljava.lang.String;` 表示 `String[]`，`[I` 表示 `int[]`。
- 其 `getClassLoader()` 返回值与数组元素类型的类加载器一致；如果元素类型是基本类型，则返回 `null`。

**对 RASP/IAST 的意义**：我们无法像 Hook 普通类那样对数组类进行字节码插桩，因为它们根本就没有可供修改的字节码。

## 五、安全攻防视角

### 5.1 漏洞利用中的常见手法

1.  **`Class.forName()` 触发恶意静态块**：通过控制传入的类名，利用其静态初始化代码完成攻击（如 Fastjson 反序列化漏洞中的 `com.sun.rowset.JdbcRowSetImpl` 利用链）。
2.  **自定义 ClassLoader + `defineClass()`**：从网络、数据库等非标准来源加载恶意字节码，然后实例化执行。
3.  **利用 TCCL 加载攻击类**：在 JNDI 注入等场景中，通过 SPI 机制，让 JVM 使用当前线程的上下文类加载器去加载攻击者指定的工厂类。

### 5.2 RASP 的对抗与依赖

1.  **RASP 自身的类加载需求**：
    - 当 RASP 需要 Hook `java.lang.ProcessBuilder` 这类由 Bootstrap ClassLoader 加载的核心类时，RASP 自己的 `Protector` 检测类默认由 AppClassLoader 加载，会导致核心类找不到检测类（`NoClassDefFoundError`）。
    - **解决方式**：RASP Agent 通过 `Instrumentation.appendToBootstrapClassLoaderSearch()` 方法，将自己的 Jar 包添加到 Bootstrap ClassLoader 的搜索路径中，从而“提升”检测类的可见性，这本质上是**对双亲委派模型的主动打破**。

2.  **RASP 对敏感类加载方法的 Hook**：
    - 为了检测恶意类加载行为，RASP 会 Hook `java.lang.Class.forName()`、`java.lang.ClassLoader.defineClass()` 等关键方法，在字节码层面对其进行插桩监控。

3.  **类加载机制对 RASP 热更新的影响**：
    - 通过 `agentmain` + `retransformClasses` 机制，RASP 可以**在 JVM 运行时替换已有类的字节码**（热更新）。
    - **关键点**：`retransformClasses` 的字节码替换**不会重新执行类的初始化和静态代码块**。它只是在“使用”阶段，用新的字节码逻辑替换了旧的实现。因此，如果 RASP 更新需要重新初始化一些静态状态，需要额外设计（例如通过管理接口来触发热加载逻辑）。



## 六、总结

Java 类加载机制是 JVM 的核心。从漏洞利用（`Class.forName` 扳机、JNDI 注入）到 RASP 防御（Bootstrap 类加载器注入、敏感方法 Hook），处处都有类加载的身影。

需要掌握：
- 类生命周期的七个阶段，特别是 **初始化** 和 **解析** 对漏洞触发的影响。
- 双亲委派模型及其源码级实现，以及 **TCCL** 这个关键的例外。
- 数组类加载、`static final` 常量在准备阶段的特殊处理等细节。
- 类加载机制在 RASP 场景下的应用与限制（如热更新对静态状态的影响）。
